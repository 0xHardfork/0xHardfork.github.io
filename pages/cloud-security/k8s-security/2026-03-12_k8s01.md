## 🧠 Concept Overview

Before you can secure Kubernetes, you must understand its architecture. A Kubernetes cluster consists of two main types of nodes: **Control Plane (Master) nodes** and **Worker nodes**. 

The **Control Plane** makes global decisions about the cluster (like scheduling) and detects/responds to cluster events. Its primary components are:
* **kube-apiserver**: The front-end of the control plane. All communication (internal and external) goes through here. It is the primary target for attackers.
* **etcd**: Consistent and highly-available key-value store used as Kubernetes' backing store for all cluster data. If an attacker gains access to etcd, they own the cluster.
* **kube-scheduler**: Watches for newly created Pods with no assigned node, and selects a node for them to run on.
* **kube-controller-manager**: Runs controller processes (like node controllers, replica controllers).

The **Worker Nodes** host the workloads (Pods). Their primary components are:
* **kubelet**: An agent that runs on each node in the cluster. It makes sure that containers are running in a Pod. If an attacker compromises a kubelet, they can execute commands in any pod on that node.
* **kube-proxy**: Maintains network rules on nodes. These rules allow network communication to your Pods from network sessions inside or outside of your cluster.
* **Container Runtime**: The software that is responsible for running containers (e.g., containerd, CRI-O).

From a security perspective, understanding this architecture is crucial because it defines your **attack surface**. Securing the cluster means securing the API Server communication, encrypting etcd, restricting kubelet access, and isolating the nodes.

---

## 🔑 Key Points

- The **API Server** is the heart of the cluster; it handles all REST requests and acts as the gatekeeper.
- **etcd** stores all cluster state, including sensitive `Secrets`. Direct access to etcd bypasses all API Server authorization mechanisms.
- The **kubelet** running on worker nodes has high privileges and talks back to the API Server.
- All control plane components usually communicate over TLS, but older or misconfigured clusters might expose components over insecure HTTP ports.
- Worker nodes should never be exposed directly to the public internet; only the API Server (and specifically configured workloads via LoadBalancers/Ingresses) should be reachable.

## 💻 Code & Configuration Examples

### Example 1 — Inspecting Control Plane Components

```bash
# View the status of core control plane pods
kubectl get pods -n kube-system -l tier=control-plane

# Example output you might see:
# NAME                                   READY   STATUS    RESTARTS   AGE
# etcd-controlplane-1                    1/1     Running   0          10d
# kube-apiserver-controlplane-1          1/1     Running   0          10d
# kube-controller-manager-controlplane-1 1/1     Running   0          10d
# kube-scheduler-controlplane-1          1/1     Running   0          10d
```

**What this demonstrates**: How to identify the control plane components running as static pods in the `kube-system` namespace. In managed environments (like EKS/GKE), you won't see these as they are abstracted away by the cloud provider.

### Example 2 — Checking API Server Endpoints

```bash
# Check where your kubectl is pointing to connect to the API server
kubectl cluster-info

# Example output:
# Kubernetes control plane is running at https://192.168.49.2:8443
# CoreDNS is running at https://192.168.49.2:8443/api/v1/namespaces/kube-system/services/kube-dns:dns/proxy
```

**What this demonstrates**: Identifying the API server endpoint. Attackers will often scan networks for port 6443 or 8443 to find exposed API servers. Your API Server should ALWAYS serve over HTTPS.

## ⚠️ Common Misconfigurations & Risks

| Misconfiguration | Risk Assessment | Secure Approach |
|------------------|----------------|----------------|
| Exposed API Server | Anyone on the internet can attempt to authenticate to your cluster or exploit zero-day vulnerabilities in the API server. | Use private API endpoints, IP allowlists, or VPNs to restrict access to the `kube-apiserver`. |
| Unencrypted etcd communication | Man-in-the-middle attacks could intercept cluster data. | Ensure `--peer-client-cert-auth` and `--client-cert-auth` are set on etcd to enforce mTLS. |
| Insecure Kubelet port (10250) | Attackers who can reach the worker node IP can execute commands in pods via the unauthenticated kubelet API. | Set `--anonymous-auth=false` on the kubelet. |

### Exercise 2 — Exploring the impact (Security context)

**Goal**: Understand what happens when you try to hit the API server directly without credentials.

**Instructions**:
1. Get your API server URL from `kubectl cluster-info`.
2. Use `curl -k <API-SERVER-URL>/api` (the `-k` ignores certificate validation).
3. Note the API server error response or runtime blockage.

**Hint**: You should receive a `403 Forbidden` error with a message about `system:anonymous` not having access, proving the API server is defending itself from unauthenticated traffic.
