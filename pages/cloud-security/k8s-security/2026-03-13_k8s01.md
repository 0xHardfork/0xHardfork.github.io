## 🧠 Concept Overview

The `kubectl` command-line interface is the primary mechanism for administrators and developers to interact with the Kubernetes API Server. From a security perspective, `kubectl` is essentially an HTTP client that authenticates to the API Server and submits RESTful JSON requests (often abstracting away the raw JSON into user-friendly YAML).

Why is this critical for security? 
1. **The Kubeconfig File**: `kubectl` relies on a configuration file (usually located at `~/.kube/config`). This file contains the API Server's address, the cluster's CA certificate (to prevent MITM attacks), and the **client credentials** (certificates, bearer tokens, or auth exec plugins) required to authenticate you. If an attacker compromises your `kubeconfig` file, they compromise the cluster with *your* level of access.
2. **Contexts**: A `kubeconfig` can store credentials for multiple clusters and namespaces. `kubectl` uses "Contexts" to determine which cluster and credentials to use for a given command. Mismanaging contexts often leads to executing destructive or permissive commands on Production clusters instead of Dev.

Understanding how `kubectl` structures auth is the first step to understanding how Kubernetes authenticates users globally.

---

## 🔑 Key Points

- **Authentication is Pluggable**: `kubectl` doesn't handle users inherently. It passes tokens or client certificates to the API Server, which then asks configured identity providers (like X.509, OIDC, or cloud IAM integrators) "Who is this?".
- **Kubeconfig Security**: The default file permissions for `~/.kube/config` should always strictly be `600` (read/write by owner only).
- **Namespaces as Security Boundaries**: By default, `kubectl` queries the `default` namespace. In multi-tenant clusters, namespaces form the primary boundary for Role-Based Access Control (RBAC) and Network Policies.
- **Dry Runs**: `kubectl` supports `--dry-run=client` (validates syntax locally) and `--dry-run=server` (sends auth to the API server to validate authorization and admission controllers without actually persisting the resource).

## 💻 Code & Configuration Examples

### Example 1 — Exploring the Kubeconfig Structure

```yaml
# file: ~/.kube/config (conceptual example)
apiVersion: v1
kind: Config
clusters:
- cluster:
    certificate-authority-data: LS0tLS1CR... # Validates the API Server's identity
    server: https://192.168.1.100:6443
  name: prod-cluster
users:
- name: admin-user
  user:
    client-certificate-data: LS0tLS1CR... # Sent to API Server to prove who you are
    client-key-data: LS0tLS1CR...
contexts:
- context:
    cluster: prod-cluster
    user: admin-user
    namespace: kube-system # The default namespace for this context
  name: prod-admin
current-context: prod-admin
```

**What this demonstrates**: The tripartite structure of `kubeconfig`: **Clusters** (Where is it?), **Users** (Who am I?), and **Contexts** (Mapping a User to a Cluster + default namespace). 

### Example 2 — Securely Managing Contexts via CLI

```bash
# View all contexts to ensure you aren't accidentally pointing at PROD
kubectl config get-contexts

# View exactly which user credentials and cluster are currently active
kubectl config view --minify

# Switch to a safer context (e.g., a dev namespace)
kubectl config use-context dev-context
```

**What this demonstrates**: Essential commands to safely navigate between clusters and minimize the risk of executing commands in the wrong environment.

## ⚠️ Common Misconfigurations & Risks

| Misconfiguration | Risk Assessment | Secure Approach |
|------------------|----------------|----------------|
| Checking in `kubeconfig` to Git | Extremely High. Attackers immediately gain cluster access with those credentials. | Never commit `kubeconfig`. Use external secret managers or CI/CD specific OIDC integrations with short-lived tokens. |
| Broad `~/.kube/config` file permissions (e.g., `644`) | Any local process or malicous script running as your user (or another user on a shared jumpbox) can read the file and steal tokens. | Run `chmod 600 ~/.kube/config`. |
| Hardcoding contexts in scripts | A script might execute destructive operations (like `kubectl delete`) on whatever happens to be the `current-context`. | Always enforce context explicitly in scripts: `kubectl --context=prod-cluster ...` |

## 🏋️ Practice Exercises

### Exercise 1 — Verifying Your Authorization (Practical application)

**Goal**: Use `kubectl`'s built-in `auth can-i` command to check what your current kubeconfig allows you to do. This is the most crucial command for debugging RBAC permissions.

**Requirements**:
- Check if you can create deployments in the `default` namespace.
- Check if you can read secrets in the `kube-system` namespace.

**Starter YAML/Command**:

```bash
# TODO: Check if you can create deployments
kubectl auth can-i _______ ________

# TODO: Check if you can read secrets in a specific namespace
kubectl auth can-i _______ ________ -n ________
```

**Hint**: The syntax is `kubectl auth can-i [VERB] [RESOURCE]`. The verbs are REST-like (create, get, list, watch, delete).

---

### Exercise 2 — Exploring the impact (Information Leakage)

**Goal**: Observe how the API server responds to unauthorized requests.

**Instructions**:
1. Try to view the raw `kubeconfig` on your current system: `cat ~/.kube/config`. Identify the `client-certificate-data` or `token`. 
2. (Optional, if you have a cluster) Try to get secrets in `kube-system` without explicit permissions. You should receive a standard `403 Forbidden` response outlining explicitly which RBAC rule failed.

**Hint**: Look for the `User "system:anonymous" cannot get resource "secrets" in API group ""` message.

```bash

# Basic resource viewing (Security context: ensuring you only see what you should)
kubectl get pods --all-namespaces
kubectl get secrets -n kube-system
```
