## 🧠 Concept Overview

In Kubernetes, you rarely deploy individual containers directly. Instead, the fundamental unit of scheduling is the **Pod**. A Pod is a logical wrapper that encapsulates one or more co-located containers, sharing the same network namespace (IP address), storage volumes, and IPC namespace. 

However, Pods are intrinsically **ephemeral**. If a node dies, the Pods on it die permanently. To build highly available, self-healing applications, Kubernetes uses controllers. 

A **ReplicaSet** ensures a specified number of identical Pod replicas are running at all times. If a Pod crashes, the ReplicaSet creates a new one. A **Deployment** is a higher-level abstraction that manages ReplicaSets, enabling declarative updates, rollbacks, and scaling for your Pods. 

From a **security perspective**, understanding this hierarchy is critical. Security controls (like Security Contexts, Resource Limits, and Service Accounts) are fundamentally applied at the **Pod template level** within a Deployment. If an attacker compromises a container inside a Pod, they gain access to the shared network namespace and any mounted volumes—meaning a vulnerable sidecar container can expose your primary application. Furthermore, if you allow developers to create raw Pods bypassing Deployments, it can evade organizational audit controls and deployment gates.

---

## 🔑 Key Points

- **Pods are mortal**: Never rely on a single Pod for state or availability. Always use a Deployment or StatefulSet.
- **Shared Context**: All containers in a single Pod share `localhost` and the same network identity. A vulnerability in a helper container puts the main app at risk.
- **Declarative Security**: Ensure security postures (like `runAsNonRoot` or dropped capabilities) are codified in the Deployment YAML, not applied manually.
- **Immutable Infrastructure**: Deployments enable rolling out new container image versions rather than patching running containers via SSH or `kubectl exec`.
- **Labels parameterize control**: Deployments match Pods using labels. An attacker who can modify a Pod's labels can remove it from a Deployment's supervision or hijack traffic.

## 💻 Code & Configuration Examples

### Example 1 — A Secure Deployment Manifest

```yaml
# File: secure-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-frontend
  namespace: prod-frontend
  labels:
    app: web
spec:
  replicas: 3
  selector:
    matchLabels: # The Deployment tracks Pods with these labels
      app: web
  template: # This is the "Pod Template"
    metadata:
      labels:
        app: web
    spec:
      serviceAccountName: frontend-sa # Least privilege identity
      containers:
      - name: nginx
        image: nginx:1.25.4-alpine # Pin exact image digests or tags
        ports:
        - containerPort: 8080 # Unprivileged port
        resources:
          limits:
            cpu: "200m"      # Prevents resource exhaustion (DoS)
            memory: "128Mi"
```

**What this demonstrates**: A foundational Deployment YAML highlighting that identity (`serviceAccountName`) and resource constraints (`resources.limits`) are defined inside the `spec.template.spec`, ensuring every replicated Pod inherits the exact same security posture.

### Example 2 — Creating and Editing Deployments via CLI

```bash
# Imperatively create a deployment (useful for fast testing, but YAML is better for GitOps)
kubectl create deployment sec-test --image=busybox -- sleep 3600

# Scale the deployment up to 5 replicas
kubectl scale deployment sec-test --replicas=5

# Watch the ReplicaSet orchestrate the creation of the Pods
kubectl get rs -w

# Check the rollout history (crucial for incident response/rollback)
kubectl rollout history deployment web-frontend
```

**What this demonstrates**: The imperative commands used to manage the lifecycle of a Deployment and observe the underlying ReplicaSet reacting to scaling changes.

## ⚠️ Common Misconfigurations & Risks

| Misconfiguration | Risk Assessment | Secure Approach |
|------------------|----------------|----------------|
| Deploying raw, naked Pods | If the node fails or the Pod is evicted, it will not be recreated. This causes availability outages. | Always wrap workloads in a `Deployment`, `DaemonSet`, or `StatefulSet`. |
| Omitting Resource Requests/Limits | A compromised or buggy container can consume 100% of the Node's CPU or memory, starving other mission-critical workloads (Noisy Neighbor / DoS). | Always configure `resources.requests` and `resources.limits` in the Pod template. |
| Using the `latest` image tag | It becomes impossible to audit exactly what code is running in production, and rollbacks may pull a new, unexpected image. | Always pin images to a specific version tag or immutable SHA256 digest (e.g., `nginx:1.25.4` or `nginx@sha256:...`). |

## 🏋️ Practice Exercises

### Exercise 1 — Implementing Resource Limits (Practical application)

**Goal**: Deploy a simple application but enforce strict resource limits to prevent denial-of-service.

**Requirements**:
- Create a Deployment named `resource-tester`.
- Use the `nginx:alpine` image.
- Enforce a memory limit of `64Mi`.

**Starter YAML/Command**:

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: resource-tester
spec:
  replicas: 1
  selector:
    matchLabels:
      app: tester
  template:
    metadata:
      labels:
        app: tester
    spec:
      containers:
      - name: nginx
        image: nginx:alpine
        # TODO: Add the resources block here to limit memory
```

**Hint**: Look under `spec.containers[]` for the `resources:` block. You need `limits.memory`.

### Exercise 2 — Observing the ReplicaSet (Security context)

**Goal**: Observe how Kubernetes enforces the desired state of a deployment, which provides innate resilience against application crashes.

**Instructions**:
1. Apply a deployment with 2 replicas: `kubectl create deployment resilience-test --image=nginx --replicas=2`
2. Get the specific names of the running pods: `kubectl get pods`
3. Simulate an attacker or crash killing one of the pods: `kubectl delete pod <pod-name-from-step-2>`
4. Immediately run `kubectl get pods` again.

**Hint**: You should see a new pod entering the `ContainerCreating` status immediately, demonstrating the control loop.
