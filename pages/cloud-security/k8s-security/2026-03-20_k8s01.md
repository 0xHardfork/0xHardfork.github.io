## 🧠 Concept Overview

Before attempting to secure a fortress, you must intimately understand its blueprints, the guards at the gates, and the secret tunnels underneath it. Kubernetes (K8s) is no different. It is not just a container runner—it is a massive, distributed state-machine designed to reliably orchestrate workloads across fleets of servers. This flexibility, unfortunately, introduces an immense attack surface.

At its core, a Kubernetes cluster is divided into two distinct logical planes: the **Control Plane** (often called the Master nodes) and the **Data Plane** (the Worker nodes).

### The Control Plane: The Brain of the Operation

The Control Plane makes global decisions about the cluster (like scheduling), detects and responds to cluster events, and serves as the single entry point for all commands. If an attacker gains control of the Control Plane, it is game over. Your cluster is entirely compromised.

1.  **kube-apiserver**: This is the heart of Kubernetes. It exposes the Kubernetes API. *Every* communication in the cluster—whether it's from a developer typing `kubectl`, a worker node reporting its health, or a pod requesting a secret—goes through the API server.
    *   **Security Context**: Because it's the front door, the API server handles Authentication (AuthN), Authorization (AuthZ), and Admission Control. If its TLS certificates are weak or it's exposed directly to the public internet without restrictions, you are inviting attackers right to your front door.
2.  **etcd**: This is the cluster's backing store. It is a highly available, strongly consistent key-value store. It holds the *entire state* of the cluster. This includes all configuration data, the state of all running pods, and crucially, all **Secrets** (passwords, TLS certs, API tokens).
    *   **Security Context**: `etcd` is the crown jewels. By default, historically, data in `etcd` was stored in plaintext. If an attacker gains read access to the `etcd` filesystem or a network port to query it directly, they can instantly dump every single secret in your cluster, bypassing all K8s API server RBAC controls.
3.  **kube-scheduler**: Watches for newly created Pods with no assigned node, and selects a node for them to run on based on resource requirements, affinity rules, and constraints.
4.  **kube-controller-manager**: Runs controller processes. These are background loops that watch the state of the cluster through the API server and make changes to move the *current* state towards the *desired* state (e.g., if a Node goes down, the Node Controller notices and triggers pod rescheduling).
5.  **cloud-controller-manager**: Links your cluster into your cloud provider's API (e.g., to automatically provision an AWS Load Balancer when you request a Service of type `LoadBalancer`).

### The Data Plane: The Worker Nodes

The worker nodes run your application workloads (Pods). While compromising a worker node might not instantly give an attacker the whole cluster, it provides a massive beachhead to launch lateral movement (privilege escalation) attacks against the API server.

1.  **kubelet**: An agent that runs on *every* node. It communicates with the API server, takes instructions, and ensures that the containers described in PodSpecs are actually running and healthy on its machine.
    *   **Security Context**: The kubelet operates with high privileges on the host node to manage container runtimes (like containerd or cri-o). If an attacker can reach an unsecured kubelet API, they can execute arbitrary commands in any pod on that node or siphon tokens to attack the Master.
2.  **kube-proxy**: A network proxy running on every node, maintaining network rules that allow network communication to your Pods from network sessions inside or outside of your cluster.
3.  **Container Runtime**: The software responsible for actually pulling the images and running the containers (e.g., `containerd`, `CRI-O`).

### The Security Implications of Distributed Trust

The most critical takeaway from K8s architecture is the **distributed nature of trust**.
The `kube-apiserver` trusts `etcd`. The `kubelet` trusts the `kube-apiserver`. The Pods trust the `kubelet`. If an attacker compromises a single container on a worker node, their goal is to abuse that trust hierarchy: break out of the container to compromise the `kubelet`, use the `kubelet`'s credentials to lie to the `kube-apiserver`, or use the API server to dump `etcd`.

## 🔑 Key Points

- Kubernetes is split into the **Control Plane** (managing cluster state) and **Data Plane** (running workloads).
- The **kube-apiserver** is the central hub. All components—both internal and external—communicate strictly through it.
- **etcd** stores the entire cluster configuration and state, including Secrets. Direct access to `etcd` is a total cluster compromise.
- The **kubelet** runs on worker nodes and manages the container lifecycle. Unauthenticated kubelet APIs are a severe vulnerability.
- Security in Kubernetes is about breaking the attack chain: preventing container escapes, restricting lateral API access, and encrypting state at rest.

## 💻 Code & Configuration Examples

### Example 1 — Inspecting Control Plane Components from a Pod

If you are an attacker (or a curious engineer) who has gained a shell inside a standard Pod, you might wonder what the cluster architecture looks like. Kubernetes implicitly injects connection information into every Pod.

```bash
# From inside a pod, print the environment variables
env | grep KUBERNETES

# Sample Output:
# KUBERNETES_SERVICE_PORT_HTTPS=443
# KUBERNETES_SERVICE_PORT=443
# KUBERNETES_PORT_443_TCP=tcp://10.96.0.1:443
# KUBERNETES_PORT_443_TCP_PROTO=tcp
# KUBERNETES_PORT_443_TCP_ADDR=10.96.0.1
# KUBERNETES_SERVICE_HOST=10.96.0.1
# KUBERNETES_PORT=tcp://10.96.0.1:443
# KUBERNETES_PORT_443_TCP_PORT=443
```

**What this demonstrates**: Kubernetes inherently relies on its architecture to be discoverable by its components. `10.96.0.1` is almost universally the internal default ClusterIP for the `kube-apiserver`. An attacker immediately knows where the "brain" of the operation lives.

### Example 2 — Querying the API Server (The Attacker's Perspective)

Since we know the API server is at `10.96.0.1`, we can try to talk to it. But wait, it uses HTTPS.

```bash
# Inside a pod shell
# Let's find the auto-mounted service account credentials
cd /var/run/secrets/kubernetes.io/serviceaccount/

# Set up variables for authentication
TOKEN=$(cat token)
CACERT=ca.crt
APISERVER=https://${KUBERNETES_SERVICE_HOST}:${KUBERNETES_PORT_443_TCP_PORT}

# Query the API server directly using the pod's identity
curl -s --cacert $CACERT --header "Authorization: Bearer $TOKEN" $APISERVER/api/v1/namespaces
```

**What this demonstrates**: This shows the architecture in action. The worker node's pod uses a token (injected by the kubelet) to make a TLS-secured API call (using the cluster's CA cert) to the Control Plane `kube-apiserver`. If RBAC is misconfigured, this API call could return sensitive namespace data or even secrets.

## ⚠️ Common Misconfigurations & Risks

| Misconfiguration | Risk Assessment | Secure Approach |
|------------------|----------------|----------------|
| Exposing API Server to `0.0.0.0/0` | Severe. Anyone on the internet can attempt to authenticate or exploit zero-days in the API server. | Restrict API server access via cloud security groups or internal private networks. |
| Anonymous Auth enabled on `kubelet` | Critical. The kubelet API (port 10250) can be queried by anyone on the network, allowing remote command execution (`exec`) into pods without hitting the central API server. | Start kubelet with `--anonymous-auth=false` and use webhook authorization. |
| Insecure API Port enabled (`--insecure-port=8080`) | Critical. Deprecated, but if running, it completely bypasses AuthN/AuthZ. Any request locally on the master node is granted full admin rights. | Ensure `--insecure-port=0` (or absent in newer K8s versions) in API server manifests. |

## 🏋️ Practice Exercises

### Exercise 1 — Mapping the Environment (Practical application)

**Goal**: Understand the default credentials injected into pods for API communication.

**Requirements**:
- Deploy a basic `nginx` pod.
- Exec into the pod (`kubectl exec -it <pod-name> -- /bin/sh`).
- Locate the ServiceAccount token directory.

**Starter YAML/Command**:

```yaml
# deploy-nginx.yaml
apiVersion: v1
kind: Pod
metadata:
  name: exploration-pod
spec:
  containers:
  - name: nginx
    image: nginx:alpine
```

**Hint**: The directory path fundamentally starts with `/var/run/secrets/`. Explore what files are mounted there.

### Exercise 2 — Testing the API Gateway (Basic Exploration)

**Goal**: Observe how the security control behaves when violated.

**Instructions**:
1. From inside the pod created in Exercise 1, attempt to `curl` the API server *without* using the bearer token.
2. Note the API server error response or runtime blockage.

**Hint**: Use `curl -k https://$KUBERNETES_SERVICE_HOST`. You should see a "Forbidden" or "Unauthorized" 403/401 JSON response because the API server rejects unauthenticated requests by default.

### Exercise 3 — The Node's Perspective (Medium Analysis)

**Goal**: Analyze existing configurations for security weaknesses.

**Instructions**:
1. Run `kubectl get nodes -o yaml`.
2. Look through the output to find details about the `kubelet` version and the `kube-proxy` running on the nodes. 
3. Identify why leaking this configuration data might help an attacker.

**Hint**: Attackers look for specific versions of containerization software (`containerd://1.6.4`) to cross-reference with known CVE databases.

### Exercise 4 — Component Health Check (Medium Implementation)

**Goal**: Implement a specific security control or feature by checking the health of the control plane.

**Requirements**:
- Use `kubectl` to query the status of the core control plane components.

**Starter YAML/Command**:

```bash
# TODO: Complete the implementation
kubectl get componentstatuses
```
*(Note: `componentstatuses` is deprecated in newer k8s variants but still conceptually useful, alternatively check kube-system pods).*

**Hint**: Look at the pods in the `kube-system` namespace. You will clearly see `etcd`, `kube-apiserver`, and `kube-controller-manager`.

### Exercise 5 — Evaluating Kubelet Configurations (Medium Hardening)

**Goal**: Modify an existing resource to improve its security posture.

**Instructions**:
1. Check the kubelet configuration file typically found on a node at `/var/lib/kubelet/config.yaml` (you can simulate this by looking up the default documentation).
2. Identify the `authentication.anonymous.enabled` flag. Ensure it is set to `false`.

**Hint**: In managed clusters (EKS/GKE), this is hardened for you. In kubeadm or bare-metal clusters, you must actively verify this.

### Exercise 6 — The etcd Dump Simulation (Hard Escalation)

**Goal**: Simulate an attack to gain elevated privileges or access.

**Instructions**:
1. Assume an attacker has achieved root access on the physical Master node.
2. Document the exact `etcdctl` command the attacker would theoretically run to dump all secrets, bypassing the API server entirely.

**Hint**: They would need the etcd CA certificate, the client certificate, and the client key (usually stored in `/etc/kubernetes/pki/etcd/`). The command uses `etcdctl get "" --prefix --keys-only`.

### Exercise 7 — Isolating the API Server (Hard Mitigation)

**Goal**: Design and implement a complex security mitigation strategy.

**Requirements**:
- Propose a network architecture design that completely shields the `kube-apiserver` from the public internet.
- Explain how administrators will manage the cluster if the API is internal-only (e.g., VPNs, Bastion hosts).

**Hint**: Look up "Private GKE Clusters" or "EKS Private Endpoints" to understand how cloud providers implement this mitigation.

### Exercise 8 — Threat Modeling the Architecture (Challenge Simulation!)

**Goal**: Solve a multi-step security challenge involving multiple concepts.

**Instructions**:
1. Draw (mentally or on paper) the path a request takes: Developer -> `kubectl` -> `API Server` -> `kubelet` -> `ContainerRuntime` -> `Pod`.
2. For each "arrow" (network hop) in that path, write down one potential vulnerability (e.g., Man-in-the-Middle) and its mitigation (e.g., mTLS).

**Hint**: Think about encryption in transit, authentication of the developer, authorization of their action, and the integrity of the container image.
