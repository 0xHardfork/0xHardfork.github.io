# Kubernetes Security Daily Learning

**Date**: 2026-03-22
**Topic**: 0x01: Cluster Architecture & Control Plane
**Phase**: Phase 1: Kubernetes Fundamentals

---

## 🧠 Concept Overview

Kubernetes is a distributed system that orchestrates containerized workloads across a fleet of machines. Before you can secure it, you need to understand its anatomy — every component, every port, every trust boundary. Think of a Kubernetes cluster like a traditional datacenter: there's a management plane that makes decisions, and a set of workers that execute them. The attack surface lives in the gaps between these components.

### The Two-Tier Architecture: Control Plane vs Data Plane

A Kubernetes cluster is split into two halves:

1. **Control Plane (Master) nodes** — the "brain" of the cluster. They decide *what* runs, *where*, and *when*. In a production setup you typically run 3 or 5 control plane nodes for high availability (odd numbers to maintain quorum in etcd).

2. **Worker (Data Plane) nodes** — the "muscle." They actually run your containerized workloads. A production cluster might run tens, hundreds, or thousands of worker nodes.

Every request — from `kubectl apply` to an autoscaler spinning up pods — flows through the **API Server** on the control plane. This makes the API Server the single most critical component from a security perspective. Compromise the API Server, and you own the cluster.

### Control Plane Components — Deep Dive

Let's walk through each control plane component, what it does, how it communicates, and what an attacker might target.

---

#### `kube-apiserver` — The Front Door

The API Server is the **only** component that exposes a REST API to the outside world. Every interaction with Kubernetes goes through it — `kubectl`, the dashboard, CI/CD pipelines, even other control plane components. It's a gatekeeper: every request passes through a pipeline of **authentication → authorization → admission control** before it touches cluster state.

**Key facts:**
- Listens on port **6443** (HTTPS by default). If you ever find a cluster exposing 6443 without TLS, run.
- It's the **sole point of contact** with etcd. No other component talks to etcd directly — they all go through the API Server.
- It validates and transforms (mutates) all resource definitions before persisting them.
- It serves the **watch** API, which lets controllers efficiently react to state changes via long-lived HTTP streams.

**Security relevance:**
The API Server is the #1 attack target. Common attack vectors:
- Anonymous access left enabled (`--anonymous-auth=true` is the default — you need explicit RBAC to restrict it)
- Exposed on the public internet without network-level restrictions
- Weak or no audit logging, making post-compromise forensics near-impossible
- Misconfigured admission webhooks that fail-open instead of fail-closed

```
          ┌─────────────────────────────────────────────┐
          │              kube-apiserver                  │
          │                                             │
          │  1. AuthN (who are you?)                    │
          │  2. AuthZ (are you allowed?)                │
          │  3. Admission Control (should we allow it?) │
          │  4. Validation (is it well-formed?)         │
          │  5. Persist to etcd                         │
          └─────────────────────────────────────────────┘
```

**Ports & protocols:**
| Port | Protocol | Purpose |
|------|----------|---------|
| 6443 | HTTPS | Secure API endpoint (production default) |
| 8080 | HTTP | Insecure port (legacy, should NEVER be enabled) |

The insecure port (`--insecure-port`) was deprecated in Kubernetes 1.20 and removed in 1.24. If you see a cluster still using it, that's a critical vulnerability — it bypasses all AuthN and AuthZ.

---

#### `etcd` — The Source of Truth

etcd is a distributed key-value store that holds the **entire cluster state** — every pod definition, every secret, every ConfigMap, every RBAC rule. It's the single source of truth.

**Key facts:**
- It's a **consistent** (CP in CAP theorem) datastore using the Raft consensus protocol.
- It typically runs on ports **2379** (client communication) and **2380** (peer communication between etcd instances).
- All data is stored as key-value pairs under paths like `/registry/pods/default/my-pod`.
- Kubernetes Secrets are stored in etcd **in plaintext by default** (base64-encoded, which is NOT encryption).

**Security relevance:**
etcd is arguably the highest-value target in a Kubernetes cluster. If an attacker gets read access to etcd, they can:
- Extract all Secrets (database passwords, API keys, TLS certificates)
- Read all ConfigMaps and RBAC configurations
- Understand the entire cluster topology

If they get **write** access, they can:
- Create cluster-admin ServiceAccount tokens
- Modify existing RBAC to grant themselves full access
- Inject malicious workloads directly

**Critical hardening steps:**
1. **Encrypt etcd at rest** — use `EncryptionConfiguration` with `aescbc` or `secretbox` providers (covered in Day 22)
2. **Restrict network access** — etcd should ONLY be reachable from the API Server
3. **Use mutual TLS (mTLS)** for all etcd communication — client-to-server and peer-to-peer
4. **Never expose etcd ports** to the pod network or the internet

```bash
# Check if etcd is running with TLS
ps aux | grep etcd | grep -E 'cert-file|key-file|trusted-ca-file'

# If you see --client-cert-auth=false, that's a red flag
```

---

#### `kube-scheduler` — The Matchmaker

The scheduler watches for newly created Pods that have no assigned node and selects a node for them to run on.

**Key facts:**
- It uses a two-phase process: **filtering** (which nodes CAN run this pod — CPU? Memory? Taints? Affinity?) and **scoring** (which node is the BEST fit).
- Communicates with the API Server only — it never talks to nodes directly.
- Listens on port **10259** (HTTPS) for health and metrics.

**Security relevance:**
The scheduler itself is a lower-priority target, but its behavior matters for security:
- **Node affinity/anti-affinity** can be abused to force pods onto specific nodes (e.g., a node with a GPU or a node with access to a sensitive network segment)
- A compromised scheduler could route workloads to attacker-controlled nodes in a multi-tenant cluster
- The scheduler's leader election mechanism uses the API Server — if the API Server is compromised, the scheduler is too

---

#### `kube-controller-manager` — The Reconciliation Loop

The controller manager runs a set of **controllers** — each one is a control loop that watches a specific resource type and drives the cluster's actual state toward the desired state.

**Important controllers:**
| Controller | What it does |
|-----------|-------------|
| ReplicaSet controller | Ensures the right number of pod replicas are running |
| Deployment controller | Manages rollouts and rollbacks |
| Node controller | Monitors node health, marks unhealthy nodes |
| ServiceAccount controller | Creates default ServiceAccounts in new namespaces |
| EndpointSlice controller | Updates endpoints when pods come and go |

**Key facts:**
- Listens on port **10257** (HTTPS).
- Runs as a single active instance (leader election via API Server lease objects).
- Uses a ServiceAccount (or kubeconfig) to authenticate against the API Server.

**Security relevance:**
- The controller manager's credentials give it **broad access** to the cluster (it needs to create pods, update statuses, etc.). If an attacker steals these credentials, they can impersonate the controller manager.
- The ServiceAccount controller automatically creates a `default` ServiceAccount in every namespace — this is a well-known attack surface (covered in Day 7).
- `--use-service-account-credentials=true` flag makes each built-in controller use its own ServiceAccount token (better least-privilege).

---

#### `cloud-controller-manager` — The Cloud Glue

If your cluster runs in AWS, GCP, or Azure, this component bridges Kubernetes with the cloud provider's APIs.

**What it handles:**
- Provisioning LoadBalancers (when you create a `Service` of type `LoadBalancer`)
- Managing cloud routes for pod-to-pod networking
- Handling node lifecycle (detecting when a VM is terminated)

**Security relevance:**
- It often runs with **highly privileged cloud IAM credentials** (e.g., an AWS IAM role that can create ELBs, modify security groups, or manage VMs)
- If compromised, an attacker could pivot from Kubernetes into the cloud account
- Isolate the cloud controller manager's credentials using mechanisms like IRSA (IAM Roles for Service Accounts) or Workload Identity

### Worker Node Components — Deep Dive

#### `kubelet` — The Node Agent

The kubelet is an agent that runs on **every** node (control plane and worker). It's responsible for ensuring that the containers described by PodSpecs are running and healthy.

**Key facts:**
- Receives pod assignments from the API Server
- Interfaces with the **Container Runtime Interface (CRI)** to start/stop containers (containerd, CRI-O)
- Reports node status and pod status back to the API Server
- Exposes a local API on port **10250** (HTTPS, authenticated) and optionally **10255** (HTTP, read-only — **dangerous!**)

**Security relevance:**
The kubelet is the most critical component on a worker node:

1. **Port 10250** — if an attacker can reach this port and the kubelet's authentication is misconfigured (`--anonymous-auth=true`), they can:
   - List all pods on the node
   - Execute commands inside any container (`/exec` endpoint)
   - Read logs from any container
   - This is literally a backdoor into every workload on the node

2. **Port 10255** — the read-only port that exposes pod specs, metrics, and environment variables without any authentication. It should **always** be disabled (`--read-only-port=0`).

3. **Static Pods** — the kubelet can run pods defined as local manifest files (usually in `/etc/kubernetes/manifests/`). The control plane components themselves are often run as static pods. An attacker with write access to these manifests can inject malicious containers that start automatically.

```bash
# Check kubelet configuration
# Look for anonymous auth and read-only port settings
systemctl cat kubelet | grep -E 'anonymous|read-only-port'

# Or check the kubelet config file directly
cat /var/lib/kubelet/config.yaml | grep -A3 authentication
```

**Critical kubelet settings:**
```yaml
# /var/lib/kubelet/config.yaml
authentication:
  anonymous:
    enabled: false        # MUST be false in production
  webhook:
    enabled: true         # Delegate auth to the API Server
authorization:
  mode: Webhook           # API Server decides if requests are allowed
readOnlyPort: 0           # Disable the insecure read-only port
```

---

#### `kube-proxy` — The Network Plumber

kube-proxy runs on every node and maintains network rules (iptables or IPVS) that implement Kubernetes `Services`.

**Key facts:**
- It doesn't actually proxy traffic in most modes — it manages iptables/IPVS rules
- Three modes: `iptables` (default), `ipvs` (high-performance), `userspace` (legacy, slow)
- Listens on port **10256** for health checks
- Gets its configuration from a ConfigMap in `kube-system` namespace

**Security relevance:**
- kube-proxy itself is not a high-value direct target, but its rules define how network traffic flows within the cluster
- A compromised kube-proxy could redirect traffic to attacker-controlled endpoints
- The iptables rules it creates are visible to anyone with access to the node (`iptables -t nat -L -n`)
- In IPVS mode, you can inspect rules with `ipvsadm -Ln`

---

#### Container Runtime (containerd, CRI-O)

The container runtime is the software that actually runs containers. Kubernetes delegates to it via the **Container Runtime Interface (CRI)**.

**Key facts:**
- `containerd` is the default runtime in most modern Kubernetes distributions
- Docker was deprecated as a runtime in Kubernetes 1.20 (removed in 1.24), though Docker-built images still work
- The runtime interacts with Linux kernel features: **namespaces** (isolation), **cgroups** (resource limits), **seccomp** (syscall filtering), and **AppArmor/SELinux** (mandatory access control)

**Security relevance:**
- Container escapes (breaking out of the container's namespace/cgroup isolation) are one of the most severe Kubernetes vulnerabilities
- The runtime socket (e.g., `/run/containerd/containerd.sock`) grants full control over all containers on the node — NEVER mount it into a pod
- Running containers as `--privileged` disables almost all isolation — this is the #1 container security anti-pattern

### Communication & Trust Boundaries

Understanding how components talk to each other reveals the trust boundaries:

```
                          ┌───────────────────────────────────┐
                          │         Control Plane              │
                          │                                   │
  kubectl ──HTTPS:6443──▶ │  API Server ◀──mTLS──▶ etcd      │
                          │    ▲    ▲                          │
                          │    │    │                          │
                          │    │    ├──▶ Scheduler             │
                          │    │    └──▶ Controller Manager    │
                          └────│──────────────────────────────┘
                               │ HTTPS (port 10250)
                          ┌────▼──────────────────────────────┐
                          │         Worker Node                │
                          │                                   │
                          │  kubelet ──CRI──▶ containerd      │
                          │                                   │
                          │  kube-proxy (iptables/IPVS rules) │
                          │                                   │
                          │  [Pod A]  [Pod B]  [Pod C]         │
                          └───────────────────────────────────┘
```

**Key trust boundaries:**
1. **External → API Server**: Must be authenticated & authorized (TLS + AuthN + AuthZ)
2. **API Server → etcd**: Mutual TLS with client certificate verification
3. **API Server → kubelet**: TLS (kubelet serves a TLS cert; API Server verifies)
4. **kubelet → API Server**: kubelet uses a client certificate or bootstrap token
5. **Pod → API Server**: ServiceAccount token (projected volume, JWT)
6. **Pod → Pod**: Flat network by default — no isolation unless NetworkPolicies are applied

The flat pod network is one of the most dangerous defaults in Kubernetes. By default, **any pod can talk to any other pod** in any namespace. This means a compromised pod in a dev namespace can reach production databases. NetworkPolicies (Day 17-18) fix this.

### How a Pod Actually Gets Created

Understanding this end-to-end flow is essential for identifying where security controls kick in:

1. User runs `kubectl apply -f pod.yaml`
2. `kubectl` sends an HTTPS POST to the API Server on port 6443
3. **Authentication**: API Server verifies the user's identity (client cert, bearer token, or OIDC)
4. **Authorization**: API Server checks RBAC — does this user have permission to create pods in this namespace?
5. **Admission Control**: A chain of admission webhooks and built-in plugins validate and potentially mutate the request. For example:
   - `PodSecurity` admission checks if the pod meets the namespace's security standard
   - A mutating webhook might inject a sidecar container
   - `LimitRanger` might set default resource limits
6. **Validation**: API Server validates the pod spec against the schema
7. **Persistence**: The pod object is written to etcd (status: `Pending`)
8. **Scheduling**: The scheduler notices the unscheduled pod, selects a node, and writes the binding to the API Server
9. **kubelet picks it up**: The kubelet on the chosen node sees the new pod (via watch), pulls container images, and starts the containers via the container runtime
10. **Status updates**: kubelet reports back to the API Server — pod is `Running`

Security controls touch steps 3, 4, 5, 6, and 9. Missing any of these creates gaps.

### Ports Summary & Attack Surface

Here's a comprehensive table of ports you should know for every CKS exam and real-world audit:

| Component | Port | Protocol | Purpose | Risk if Exposed |
|-----------|------|----------|---------|----------------|
| API Server | 6443 | HTTPS | Primary API endpoint | AuthN bypass, cluster compromise |
| API Server | 8080 | HTTP | Insecure port (removed in 1.24) | CRITICAL: no auth at all |
| etcd | 2379 | HTTPS | Client communication | Secret theft, state manipulation |
| etcd | 2380 | HTTPS | Peer communication | Cluster state corruption |
| Scheduler | 10259 | HTTPS | Health/metrics | Info leakage |
| Controller Manager | 10257 | HTTPS | Health/metrics | Info leakage |
| kubelet | 10250 | HTTPS | kubelet API | Container exec, log access |
| kubelet | 10255 | HTTP | Read-only API | Pod spec/env leakage |
| kube-proxy | 10256 | HTTP | Health check | Minimal direct risk |

## 🔑 Key Points

- **The API Server is the single gateway** to the cluster — every request flows through its AuthN → AuthZ → Admission pipeline. Securing it is job #1.
- **etcd stores everything in plaintext by default** — including Secrets. Enable encryption at rest and restrict network access to API Server–only.
- **The kubelet on every node is a high-value target** — disable anonymous auth, disable the read-only port (10255), and enforce webhook authorization.
- **The pod network is flat by default** — any pod can reach any other pod. NetworkPolicies are essential but not applied by default.
- **Control plane components authenticate to each other via mTLS** — certificate rotation and verification are critical security hygiene tasks.

## 💻 Code & Configuration Examples

### Example 1 — Inspecting Cluster Components with `kubectl`

```bash
# View the component statuses (deprecated in newer versions, but still useful)
kubectl get componentstatuses

# View control plane pods (if running as static pods)
kubectl get pods -n kube-system

# Check API Server configuration flags
kubectl -n kube-system describe pod kube-apiserver-<node-name> | grep -A 50 Command

# View the kubelet's configuration on the current node
kubectl proxy &
curl -s http://localhost:8001/api/v1/nodes/<node-name>/proxy/configz | jq .

# View etcd member list (from the control plane node)
ETCDCTL_API=3 etcdctl member list \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key
```

**What this demonstrates**: How to inspect the running configuration of every major cluster component. This is the first thing you do in any security audit — understand what's actually running and how it's configured.

### Example 2 — Verifying kubelet Security Settings

```bash
# SSH into a worker node, then:

# 1. Check if anonymous auth is enabled
cat /var/lib/kubelet/config.yaml | grep -A2 "anonymous"
# Expected: enabled: false

# 2. Check if read-only port is open
cat /var/lib/kubelet/config.yaml | grep "readOnlyPort"
# Expected: readOnlyPort: 0

# 3. Check if webhook authorization is enabled
cat /var/lib/kubelet/config.yaml | grep -A2 "authorization"
# Expected: mode: Webhook

# 4. Test if port 10255 is actually closed
curl -s http://localhost:10255/pods
# Expected: connection refused

# 5. Test if port 10250 requires authentication
curl -sk https://localhost:10250/pods
# Expected: 401 Unauthorized (if anonymous is disabled)
# Danger: If this returns pod data, anonymous auth is ON
```

**What this demonstrates**: A practical checklist for verifying kubelet hardening on any node. This is a core CKS exam task and an essential real-world audit step.

### Example 3 — Exploring etcd Data (Security Audit)

```bash
# From a control plane node with etcd access:

# List all keys in etcd (there could be thousands)
ETCDCTL_API=3 etcdctl get / --prefix --keys-only \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key \
  | head -20

# Read a specific secret directly from etcd (to prove it's plaintext!)
ETCDCTL_API=3 etcdctl get /registry/secrets/default/my-secret \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key

# If encryption at rest is NOT enabled, you'll see the secret data
# in base64-encoded plaintext. Anyone with etcd access can decode it.
```

**What this demonstrates**: Why encrypting etcd at rest is non-negotiable. If an attacker obtains the etcd certificates or gets direct network access to port 2379, all Secrets are exposed.

### Example 4 — Minimal kubeadm Cluster Bootstrap (Reference)

```yaml
# kubeadm-config.yaml — a basic cluster config showing security-relevant settings
apiVersion: kubeadm.k8s.io/v1beta3
kind: ClusterConfiguration
kubernetesVersion: "v1.29.0"
networking:
  podSubnet: "10.244.0.0/16"
  serviceSubnet: "10.96.0.0/12"
apiServer:
  extraArgs:
    anonymous-auth: "false"           # Disable anonymous access
    audit-log-path: "/var/log/k8s-audit.log"   # Enable audit logging
    audit-log-maxage: "30"
    audit-log-maxbackup: "10"
    audit-log-maxsize: "100"
    enable-admission-plugins: "PodSecurity,NodeRestriction"
    # encryption-provider-config: "/etc/kubernetes/enc/encryption-config.yaml"
  certSANs:
    - "k8s.mycompany.com"
    - "10.0.0.10"
etcd:
  local:
    extraArgs:
      client-cert-auth: "true"        # Require client certs for etcd
controllerManager:
  extraArgs:
    use-service-account-credentials: "true"  # Per-controller SA tokens
---
apiVersion: kubeadm.k8s.io/v1beta3
kind: InitConfiguration
nodeRegistration:
  kubeletExtraArgs:
    read-only-port: "0"               # Disable read-only kubelet port
    protect-kernel-defaults: "true"    # Refuse to start if sysctl doesn't match
```

**What this demonstrates**: How a security-conscious cluster is bootstrapped from the start. Each `extraArgs` entry maps directly to a security control we discussed in the Concept Overview.

## ⚠️ Common Misconfigurations & Risks

| Misconfiguration | Risk Assessment | Secure Approach |
|------------------|----------------|-----------------|
| API Server `--anonymous-auth=true` (default) | Allows unauthenticated requests — combined with permissive RBAC, allows cluster enumeration or worse | Set `--anonymous-auth=false` or ensure RBAC denies anonymous requests |
| kubelet read-only port 10255 enabled | Exposes pod specs, environment variables, and metrics without any authentication | Set `readOnlyPort: 0` in kubelet config |
| etcd accessible from the pod network | Any compromised pod can read/write the entire cluster state | Firewall etcd ports (2379/2380) to control plane nodes only |
| Secrets stored unencrypted in etcd | Anyone with etcd access (backup tapes, disk snapshots) can read all secrets | Configure `EncryptionConfiguration` with aescbc or KMS provider |
| `--insecure-port` enabled on API Server | Bypasses ALL authentication and authorization — essentially root access | Never enable this flag (removed in K8s 1.24+) |
| No audit logging configured | Post-breach investigation is nearly impossible without API audit trail | Enable `--audit-log-path` with a comprehensive audit policy |
| kubelet `--anonymous-auth=true` | Any network-adjacent attacker can exec into containers, read logs, steal data | Set `authentication.anonymous.enabled: false` + webhook authorization |

### Exercise 1 — Component Identification (Easy)

**Goal**: Given a running Kubernetes cluster (or documentation), identify every control plane and worker node component, the port it listens on, and its primary function.

**Requirements**:
- List all 6 control plane components (API Server, etcd, scheduler, controller-manager, cloud-controller-manager, CoreDNS)
- List all 3 worker node components (kubelet, kube-proxy, container runtime)
- For each, note the default port and whether it uses TLS

**Starter Command**:

```bash
# On a control plane node:
kubectl get pods -n kube-system -o wide
# Map each pod to its component and find its ports
kubectl describe pod <component-pod-name> -n kube-system | grep -E 'Port|Args'
```

**Hint**: Not all components run as pods — some might be systemd services. Check `systemctl list-units | grep kube` as well.

### Exercise 3 — kubelet Security Check (Medium)

**Goal**: Audit the kubelet configuration on a node and determine if anonymous access is possible.

**Instructions**:

**Hint**: If anonymous auth is enabled and you can reach port 10250, try `curl -sk https://localhost:10250/runningpods/` — it will return JSON with all running pod info.

### Exercise 4 — etcd Health & Certificate Verification (Medium)

**Goal**: Verify that etcd is properly configured with TLS and client certificate authentication.

**Requirements**:
- Connect to etcd using `etcdctl` with the correct certificates
- Verify the etcd cluster is healthy
- Attempt to connect **without** certificates and confirm it fails
- List the certificates etcd is using and check their expiration dates

**Starter Command**:

```bash
# Healthy check with proper certs
ETCDCTL_API=3 etcdctl endpoint health \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key

# Now try WITHOUT certificates:
ETCDCTL_API=3 etcdctl endpoint health \
  --endpoints=https://127.0.0.1:2379
# This should fail if client-cert-auth is enabled

# Check cert expiration
openssl x509 -in /etc/kubernetes/pki/etcd/server.crt -noout -dates
```

**Hint**: If the connection without certs succeeds, `--client-cert-auth=false` is set in etcd — a critical misconfiguration.

### Exercise 7 — Simulate an etcd Data Breach (Hard)

**Goal**: Demonstrate the risk of unencrypted etcd by reading Kubernetes Secrets directly from the etcd datastore.

**Requirements**:
- Create a Secret in Kubernetes: `kubectl create secret generic test-secret --from-literal=password=SuperS3cret!`
- Use `etcdctl` to read the raw Secret data from etcd
- Decode the base64-encoded data to reveal the plaintext password
- Document the precise command sequence that proves Secrets are not encrypted at rest
- Research what `EncryptionConfiguration` you would need to prevent this

**Starter Command**:

```bash
# Create the secret
kubectl create secret generic test-secret --from-literal=password=SuperS3cret!

# Read from etcd
ETCDCTL_API=3 etcdctl get /registry/secrets/default/test-secret \
  --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key | hexdump -C | head -20

# Look for the password in the output
```

**Hint**: The raw etcd data is a protobuf-encoded Kubernetes object. You'll see the password visible as ASCII in the hexdump output. A proper `EncryptionConfiguration` would encrypt this data before it reaches etcd.

**Goal**: Perform a comprehensive security audit of a Kubernetes cluster's architecture components and produce a structured report.

**Instructions**:
1. For each control plane component, check:
   - Are TLS certificates valid and not expired?
   - Is anonymous access disabled?
   - Is audit logging enabled?
   - Are admission controllers properly configured?
2. For each worker node, check:
   - kubelet anonymous auth status
   - Read-only port status
   - Whether the container runtime socket is exposed to pods
3. For network trust boundaries:
   - Can pods reach the API Server? (usually yes — but is RBAC protecting it?)
   - Can pods reach etcd directly? (should be NO)
   - Can pods reach kubelet ports on other nodes?
4. Compile your findings into a report with:
   - Component name
   - Finding (secure / misconfiguration)
   - Severity (Critical / High / Medium / Low)
   - Recommendation

**Hint**: Use `kubectl auth can-i --list --as system:anonymous` to check what anonymous users can do. Use `nmap -p 2379,10250,10255 <node-ip>` from a pod to test network reachability to sensitive ports. Compare your findings against the CIS Kubernetes Benchmark.

## 🗒️ Quick CLI Cheat Sheet

```bash
# View cluster components
kubectl get componentstatuses              # (deprecated but may still work)
kubectl get pods -n kube-system            # control plane pods

# Inspect kubelet config from outside the node
kubectl get --raw /api/v1/nodes/<node>/proxy/configz | jq

# Check etcd health
ETCDCTL_API=3 etcdctl endpoint health --endpoints=https://127.0.0.1:2379 \
  --cacert=/etc/kubernetes/pki/etcd/ca.crt \
  --cert=/etc/kubernetes/pki/etcd/server.crt \
  --key=/etc/kubernetes/pki/etcd/server.key

# Check what anonymous users can do
kubectl auth can-i --list --as system:anonymous

# Scan for open kubelet ports from a pod
curl -sk https://<node-ip>:10250/pods     # should return 401
curl -s http://<node-ip>:10255/pods       # should refuse connection
```
