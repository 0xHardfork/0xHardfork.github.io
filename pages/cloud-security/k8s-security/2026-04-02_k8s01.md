## 🧠 Concept Overview

`kubectl` is the single most important tool you'll ever use to interact with a Kubernetes cluster. It's a command-line binary that talks directly to the Kubernetes API Server over HTTPS. Every time you type `kubectl get pods`, behind the scenes it constructs a REST request, authenticates you using credentials from your local config file, sends that request to the API Server, and hands the response back to your terminal in a human-readable format. Understanding how this works — and how it can go wrong from a security standpoint — is absolutely fundamental.

### How kubectl Talks to the API Server

When you install `kubectl`, it doesn't know where your cluster is. It needs three critical pieces of information:

1. **The API Server endpoint** — a URL like `https://10.0.1.50:6443` or `https://my-cluster.example.com`
2. **Authentication credentials** — who are you? This can be a client certificate, a bearer token, an OIDC token, or even a basic username/password (deprecated but still lurking in old clusters)
3. **A CA certificate** — so `kubectl` can verify it's actually talking to the legitimate API Server and not to some man-in-the-middle attacker

All three of these are stored in a single file called **kubeconfig**.

### The Kubeconfig File — Your Cluster Keyring

By default, `kubectl` looks for your kubeconfig at `~/.kube/config`. This is a YAML file that contains:

- **clusters** — each entry defines a cluster endpoint (server URL) and a certificate authority (CA) to trust
- **users** — each entry defines authentication credentials (client certs, tokens, etc.)
- **contexts** — each entry *binds* a cluster to a user plus an optional default namespace

Think of it like a physical keyring. Each "key" (user credential) is tagged with the "door" (cluster) it opens, and the "room" (namespace) you land in. The currently active context tells `kubectl` which key-and-door combination to use right now.

Here's what a typical kubeconfig looks like:

```yaml
apiVersion: v1
kind: Config
current-context: dev-cluster-admin

clusters:
  - name: dev-cluster
    cluster:
      server: https://10.0.1.50:6443
      certificate-authority-data: LS0tLS1CRUdJ...  # base64 CA cert

  - name: prod-cluster
    cluster:
      server: https://prod-k8s.example.com:6443
      certificate-authority-data: LS0tLS1CRUdJ...

users:
  - name: admin-dev
    user:
      client-certificate-data: LS0tLS1CRUdJ...  # base64 client cert
      client-key-data: LS0tLS1CRUdJ...           # base64 client key

  - name: readonly-prod
    user:
      token: eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVC...  # bearer token

contexts:
  - name: dev-cluster-admin
    context:
      cluster: dev-cluster
      user: admin-dev
      namespace: default

  - name: prod-cluster-readonly
    context:
      cluster: prod-cluster
      user: readonly-prod
      namespace: monitoring
```

Notice those `certificate-authority-data`, `client-certificate-data`, and `client-key-data` fields? They are all base64-encoded PEM certificates. This means your kubeconfig file is **extremely sensitive** — it literally contains the keys to your kingdom. Anyone who gets a copy of this file can authenticate as you to your clusters. Let that sink in.

### Security Implications of Kubeconfig

Here's where it gets real:

- **File permissions matter hugely.** If your `~/.kube/config` is world-readable (`chmod 644`), any process or user on that machine can read your cluster credentials. Always ensure it's `chmod 600` — readable only by you.
- **Never commit kubeconfig to Git.** This happens more often than people admit. A leaked kubeconfig in a public repo gives an attacker full access to your cluster with whatever privileges your user has.
- **Token-based kubeconfigs are dangerous if tokens don't expire.** Long-lived bearer tokens (like static ServiceAccount tokens) give permanent access until explicitly revoked.
- **Multiple clusters in one file = blast radius.** If your single kubeconfig contains admin credentials for both dev and prod, compromising that file compromises everything. Consider using separate kubeconfig files per cluster and merging them with the `KUBECONFIG` environment variable.

### The KUBECONFIG Environment Variable

You can tell `kubectl` to load kubeconfig from a different location — or even merge multiple files:

```bash
# Use a specific kubeconfig file
export KUBECONFIG=/path/to/my-cluster-config.yaml

# Merge multiple kubeconfig files (colon-separated on Linux/Mac)
export KUBECONFIG=~/.kube/dev-config:~/.kube/prod-config:~/.kube/staging-config

# kubectl will merge them in memory and make all contexts available
kubectl config get-contexts
```

This is a better pattern for production environments: keep each cluster's credentials in its own file, with tight file permissions, and merge only what you need for a given session.

### Contexts: Switching Between Clusters and Namespaces

Contexts are how you juggle multiple cluster identities without re-specifying credentials every time:

```bash
# List all contexts
kubectl config get-contexts

# Show the current active context
kubectl config current-context

# Switch to a different context
kubectl config use-context prod-cluster-readonly

# Set the default namespace for the current context
kubectl config set-context --current --namespace=kube-system
```

From a security angle, there's a real cautionary tale here. Imagine you're debugging something in your dev cluster and you copy-paste a `kubectl delete` command. But you forgot that your *current context* is pointed at production. You just deleted resources in prod. This is not hypothetical — it happens regularly. Tools like `kubectx` and `kubens` (or shell prompt integrations that display the current context) can help you stay aware of where your commands are going.

### Imperative Commands vs Declarative Manifests

`kubectl` supports two styles of working:

**Imperative** — you tell Kubernetes exactly what to do, step by step:

```bash
# Create a deployment
kubectl create deployment nginx --image=nginx:1.25

# Expose it as a service
kubectl expose deployment nginx --port=80 --type=ClusterIP

# Scale it
kubectl scale deployment nginx --replicas=3

# Delete it
kubectl delete deployment nginx
```

**Declarative** — you describe the desired state in a YAML file and let Kubernetes figure out how to get there:

```bash
# Apply a manifest (create or update)
kubectl apply -f deployment.yaml

# Apply all manifests in a directory
kubectl apply -f ./manifests/

# Apply from a URL
kubectl apply -f https://raw.githubusercontent.com/example/manifests/main/deployment.yaml
```

The declarative approach is strongly preferred for production. Why? Because it gives you version control. You can review changes in pull requests, audit who changed what, and roll back by reverting a Git commit. This is the foundation of **GitOps**, which is considered a security best practice because it creates an auditable trail of every change to your cluster.

The imperative approach is great for quick experimentation and debugging, but it leaves no paper trail. There's no YAML file in version control that says "this is what should be running." That's a security risk in any real environment.

### Viewing and Inspecting Resources

This is where you'll spend a *lot* of your time. The `kubectl get` and `kubectl describe` commands are your bread and butter:

```bash
# List all pods in the current namespace
kubectl get pods

# List pods in ALL namespaces (security audit goldmine)
kubectl get pods --all-namespaces
kubectl get pods -A  # shorthand

# Get detailed output in wide format
kubectl get pods -o wide

# Get the full YAML definition of a specific pod
kubectl get pod my-pod -o yaml

# Get JSON output (useful for scripting with jq)
kubectl get pods -o json | jq '.items[].metadata.name'

# Describe gives human-readable details including events
kubectl describe pod my-pod

# List all resource types available in the cluster
kubectl api-resources

# List specific resources
kubectl get deployments
kubectl get services
kubectl get namespaces
kubectl get nodes
kubectl get secrets         # ⚠️ these are base64-encoded, NOT encrypted!
kubectl get configmaps
kubectl get serviceaccounts
```

That `kubectl get secrets` line deserves special attention. By default, Kubernetes Secrets are **only base64-encoded**, which is NOT encryption. Anyone with permission to `get` secrets can decode them trivially:

```bash
# Retrieve a secret
kubectl get secret my-secret -o yaml

# The 'data' field values are base64 — decode them easily
kubectl get secret my-secret -o jsonpath='{.data.password}' | base64 -d
```

This is why RBAC (which we'll cover in Days 08–10) is absolutely critical. You must tightly control who can `get`, `list`, and `watch` Secrets.

### Labels, Selectors, and Filtering

In large clusters, you'll have hundreds or thousands of resources. Labels and selectors let you slice and dice:

```bash
# List pods with a specific label
kubectl get pods -l app=nginx

# Multiple label selectors
kubectl get pods -l app=nginx,tier=frontend

# Label inequality
kubectl get pods -l 'environment!=production'

# Set-based selectors
kubectl get pods -l 'environment in (staging, qa)'

# Add a label to a pod
kubectl label pod my-pod security-scanned=true

# Remove a label (trailing dash)
kubectl label pod my-pod security-scanned-
```

From a security perspective, labels are used by NetworkPolicies (Days 17–18) and Pod Security Standards (Days 13–14) to target specific workloads. Understanding how selectors work is essential for building network segmentation and access control policies.

### Namespaces: Logical Isolation

Namespaces partition a cluster into virtual segments. They're not hard security boundaries (an attacker with cluster-level access can see everything), but they provide:

- **RBAC scoping** — you can grant permissions within a single namespace
- **NetworkPolicy scoping** — restrict network traffic per namespace
- **Resource quota enforcement** — limit CPU/memory per namespace
- **General organization** — separate teams, environments, or applications

```bash
# List all namespaces
kubectl get namespaces

# Create a namespace
kubectl create namespace secure-apps

# Run a command in a specific namespace
kubectl get pods -n kube-system

# Set a default namespace for your context (so you don't need -n every time)
kubectl config set-context --current --namespace=secure-apps
```

Default namespaces to know about:

| Namespace | Purpose | Security Note |
|-----------|---------|---------------|
| `default` | Where resources go if no namespace specified | Avoid using for real workloads — apply strict policies |
| `kube-system` | Core Kubernetes components (API Server, CoreDNS, etc.) | **Extremely sensitive** — lock down access tightly |
| `kube-public` | Publicly readable data | Contains cluster-info ConfigMap, rarely used |
| `kube-node-lease` | Node heartbeat tracking | Low sensitivity, but don't mess with it |

### Executing Commands Inside Containers

`kubectl exec` lets you open a shell inside a running container. It's incredibly useful for debugging, but it's also a massive security concern:

```bash
# Run a single command inside a pod
kubectl exec my-pod -- ls /etc

# Open an interactive shell (the classic "get shell" move)
kubectl exec -it my-pod -- /bin/bash
# or if bash isn't available (distroless images):
kubectl exec -it my-pod -- /bin/sh

# Exec into a specific container in a multi-container pod
kubectl exec -it my-pod -c sidecar-container -- /bin/sh
```

Why is this a security concern?

- An attacker who compromises a ServiceAccount with `exec` permissions can get a shell inside any pod that account can access
- From inside a container, they can read mounted secrets, environment variables, and possibly reach the metadata API
- `kubectl exec` generates audit log events (`pods/exec` resource), so you can detect it — **if** you have audit logging enabled (Day 25)

In hardened environments, `exec` permissions are restricted or entirely removed via RBAC. Distroless base images (which don't include a shell) provide defense-in-depth by making `exec` useless even if an attacker gets the permission.

### Viewing Logs

Logs are essential for debugging and security monitoring:

```bash
# View logs for a pod
kubectl logs my-pod

# Stream logs in real-time (like tail -f)
kubectl logs -f my-pod

# View logs from a previous crashed container
kubectl logs my-pod --previous

# View logs from a specific container in a multi-container pod
kubectl logs my-pod -c sidecar

# View logs with timestamps
kubectl logs my-pod --timestamps

# View last 100 lines
kubectl logs my-pod --tail=100

# View logs from all pods matching a label
kubectl logs -l app=nginx --all-containers
```

From a security standpoint, logs can contain sensitive information (tokens, passwords, PII) that applications accidentally print. Centralizing logs into a secure logging pipeline (EFK stack, Loki, etc.) with access controls is a best practice.

### Port Forwarding

`kubectl port-forward` creates a secure tunnel from your local machine to a pod or service inside the cluster:

```bash
# Forward local port 8080 to pod port 80
kubectl port-forward pod/my-pod 8080:80

# Forward to a service
kubectl port-forward svc/my-service 8080:80

# Forward to a deployment (picks a random pod)
kubectl port-forward deployment/my-deployment 8080:80

# Listen on all interfaces (not just localhost)
kubectl port-forward --address 0.0.0.0 pod/my-pod 8080:80
```

This is handy for accessing internal services without creating an external Ingress. But be careful — if you `port-forward --address 0.0.0.0`, you're exposing that internal service to your entire local network.

### Dry Run and Diff — The Safety Net

Before applying changes to a live cluster, you can preview them:

```bash
# Generate YAML without actually creating the resource
kubectl create deployment nginx --image=nginx:1.25 --dry-run=client -o yaml

# Apply with dry-run to see what would happen on the server
kubectl apply -f deployment.yaml --dry-run=server

# Diff your local manifest against what's currently running
kubectl diff -f deployment.yaml
```

`--dry-run=client` never talks to the API Server; it generates YAML locally. `--dry-run=server` sends the request to the API Server for validation (including admission webhooks) but doesn't persist it. The `diff` command is brilliant for pre-deploy reviews — it shows you exactly what fields will change.

### API Resources and Verbs — Understanding the Permission Model

Every `kubectl` command maps to an API resource and a verb. Understanding this mapping is essential for setting up RBAC later:

```bash
# List all API resources the cluster supports
kubectl api-resources

# Show details including the API group and supported verbs
kubectl api-resources -o wide

# List only namespaced resources
kubectl api-resources --namespaced=true

# List only cluster-scoped resources
kubectl api-resources --namespaced=false

# Explain a specific resource (see fields and their types)
kubectl explain pod
kubectl explain pod.spec.containers
kubectl explain pod.spec.securityContext
```

The key verbs you'll encounter in RBAC:

| Verb | kubectl command | Meaning |
|------|----------------|---------|
| `get` | `kubectl get` | Read a specific resource |
| `list` | `kubectl get` (multiple) | List resources of a type |
| `watch` | `kubectl get --watch` | Stream real-time changes |
| `create` | `kubectl create / apply` | Create a new resource |
| `update` | `kubectl apply / edit` | Modify an existing resource |
| `patch` | `kubectl patch` | Partially modify a resource |
| `delete` | `kubectl delete` | Delete a resource |
| `exec` | `kubectl exec` | Execute commands in a container |
| `logs` | `kubectl logs` | Read container logs |

Every time you issue a `kubectl` command, the API Server checks: "Does this user have permission to perform this **verb** on this **resource** in this **namespace**?" If not, you get a `Forbidden` error. That permission check is RBAC, which you'll master in Days 08–10.

### Checking Your Permissions — `kubectl auth can-i`

This is a security must-know:

```bash
# Can I create pods in the current namespace?
kubectl auth can-i create pods

# Can I delete deployments in the kube-system namespace?
kubectl auth can-i delete deployments -n kube-system

# Can I get secrets cluster-wide?
kubectl auth can-i get secrets --all-namespaces

# List ALL permissions I have in the current namespace
kubectl auth can-i --list

# Check what a specific ServiceAccount can do (requires admin privileges)
kubectl auth can-i --list --as=system:serviceaccount:default:my-sa

# Impersonate a user to test their permissions
kubectl auth can-i get pods --as=jane@example.com
```

`kubectl auth can-i` is one of the most valuable tools for security auditing. When you inherit a cluster and want to understand who can do what, this is where you start. The `--as` flag lets you impersonate another user or ServiceAccount (if you have impersonation permissions), which is incredibly useful for testing RBAC policies without switching contexts.

### Applying Resources from stdin and Using Plugins

Some advanced patterns:

```bash
# Pipe YAML into kubectl (useful in CI/CD pipelines)
cat <<EOF | kubectl apply -f -
apiVersion: v1
kind: Pod
metadata:
  name: security-test
  namespace: default
spec:
  containers:
    - name: busybox
      image: busybox:1.36
      command: ["sleep", "3600"]
      securityContext:
        runAsNonRoot: true
        runAsUser: 1000
        readOnlyRootFilesystem: true
EOF

# kubectl plugins via krew
# krew is the kubectl plugin manager
kubectl krew install ctx     # kubectx as a plugin
kubectl krew install ns      # kubens as a plugin
kubectl krew install neat    # clean up kubectl get output
kubectl krew install who-can # show who has RBAC access to a resource
```

The `who-can` plugin is especially valuable for security auditing:

```bash
# Who can delete pods in the default namespace?
kubectl who-can delete pods -n default

# Who can create clusterroles? (escalation risk)
kubectl who-can create clusterroles
```

### Proxy Mode and Raw API Access

Sometimes you need to talk to the Kubernetes API directly:

```bash
# Start a local proxy to the API Server (handles auth for you)
kubectl proxy --port=8001

# Then access the API via curl:
curl http://localhost:8001/api/v1/namespaces/default/pods

# Direct API call without the proxy (manual auth)
APISERVER=$(kubectl config view --minify -o jsonpath='{.clusters[0].cluster.server}')
TOKEN=$(kubectl get secret my-sa-token -o jsonpath='{.data.token}' | base64 -d)
curl -s $APISERVER/api/v1/namespaces/default/pods \
  --header "Authorization: Bearer $TOKEN" \
  --cacert /path/to/ca.crt
```

`kubectl proxy` is safe for local debugging because it only listens on localhost by default. But be careful: if someone runs `kubectl proxy --address=0.0.0.0 --accept-hosts='.*'`, they've just exposed an unauthenticated proxy to the Kubernetes API on the network. This is a well-known attack vector.

## 🔑 Key Points

- **Kubeconfig is your cluster keyring** — it contains cluster endpoints, credentials, and contexts. Protect it like a private SSH key (`chmod 600`).
- **Contexts determine where commands go** — always verify your current context before running destructive commands, especially in multi-cluster setups.
- **Declarative (`kubectl apply`) over imperative** — use version-controlled YAML manifests for audit trails and GitOps workflows.
- **`kubectl auth can-i` is your RBAC flashlight** — use it to audit permissions for yourself, ServiceAccounts, and other users.
- **Every kubectl command = API verb + resource** — understanding this mapping is the foundation for writing RBAC policies.

## 💻 Code & Configuration Examples

### Example 1 — Multi-Cluster Kubeconfig with Separate Files

```bash
# Create separate kubeconfig files per environment
# ~/kube/dev.yaml — contains dev cluster credentials
# ~/kube/prod.yaml — contains prod cluster credentials

# Set restrictive permissions
chmod 600 ~/kube/dev.yaml
chmod 600 ~/kube/prod.yaml

# Merge for a session
export KUBECONFIG=~/kube/dev.yaml:~/kube/prod.yaml

# Verify both clusters are available
kubectl config get-contexts
# CURRENT   NAME              CLUSTER      AUTHINFO       NAMESPACE
# *         dev-admin         dev-cluster  dev-admin      default
#           prod-readonly     prod-cluster prod-viewer    monitoring

# Switch to prod context
kubectl config use-context prod-readonly

# Verify
kubectl config current-context
# prod-readonly
```

**What this demonstrates**: Keeping kubeconfig files per-cluster reduces blast radius. If one file is compromised, only that cluster is affected. The `KUBECONFIG` env var merges them without combining credentials into a single file.

### Example 2 — Security Audit with kubectl

```bash
#!/bin/bash
# Quick security audit script using kubectl

echo "=== Current Context ==="
kubectl config current-context

echo ""
echo "=== My Permissions (current namespace) ==="
kubectl auth can-i --list

echo ""
echo "=== Pods Running as Root ==="
kubectl get pods -A -o json | jq -r '
  .items[] |
  select(.spec.containers[].securityContext.runAsNonRoot != true) |
  "\(.metadata.namespace)/\(.metadata.name)"
'

echo ""
echo "=== Secrets in Default Namespace ==="
kubectl get secrets -n default

echo ""
echo "=== ServiceAccounts with Auto-mounted Tokens ==="
kubectl get serviceaccounts -A -o json | jq -r '
  .items[] |
  select(.automountServiceAccountToken != false) |
  "\(.metadata.namespace)/\(.metadata.name)"
'

echo ""
echo "=== Privileged Pods ==="
kubectl get pods -A -o json | jq -r '
  .items[] |
  select(.spec.containers[].securityContext.privileged == true) |
  "\(.metadata.namespace)/\(.metadata.name)"
'

echo ""
echo "=== Nodes ==="
kubectl get nodes -o wide
```

**What this demonstrates**: A practical security-oriented audit script using `kubectl` and `jq` to quickly surface risky configurations: pods running as root, privileged containers, auto-mounted service account tokens, and exposed secrets.

### Example 3 — Safe Apply Pattern with Dry Run and Diff

```bash
# 1. Generate YAML from an imperative command (without creating anything)
kubectl create deployment secure-nginx \
  --image=nginx:1.25 \
  --dry-run=client \
  -o yaml > secure-nginx.yaml

# 2. Edit the YAML to add security context
cat >> secure-nginx.yaml << 'PATCH'
# Add this under spec.template.spec.containers[0]:
#   securityContext:
#     runAsNonRoot: true
#     readOnlyRootFilesystem: true
#     allowPrivilegeEscalation: false
PATCH

# 3. Validate server-side (checks admission webhooks too)
kubectl apply -f secure-nginx.yaml --dry-run=server

# 4. Diff against what's currently in the cluster
kubectl diff -f secure-nginx.yaml

# 5. Only apply if the diff looks safe
kubectl apply -f secure-nginx.yaml
```

**What this demonstrates**: A safe deployment workflow — generate YAML, review it, dry-run against the server for validation, diff to see exactly what will change, and only then apply. This is the pattern you should follow in production environments.

## ⚠️ Common Misconfigurations & Risks

| Misconfiguration | Risk Assessment | Secure Approach |
|------------------|----------------|-----------------|
| Kubeconfig file with `644` permissions | Any local user/process can steal cluster credentials | `chmod 600 ~/.kube/config` — owner-only read/write |
| Committing kubeconfig to Git | Full cluster access to anyone who finds the repo | Add `*kubeconfig*` and `.kube/` to `.gitignore`; rotate credentials if leaked |
| Using `kubectl proxy --address=0.0.0.0` | Unauthenticated API access to anyone on the network | Only use `kubectl proxy` on localhost (default); never bind to `0.0.0.0` |
| Running all workloads in the `default` namespace | No RBAC isolation, no NetworkPolicy boundaries | Create dedicated namespaces per team/app; apply policies per namespace |
| Granting `exec` permission broadly via RBAC | Attackers can shell into containers, read secrets, pivot | Restrict `pods/exec` to emergency break-glass roles only |
| Using `kubectl apply -f` from URLs without review | Applying arbitrary resources from the internet | Always download, review, and version-control manifests before applying |

## 🏋️ Practice Exercises

### Exercise 1 — Explore Your Kubeconfig (Practical Application)

**Goal**: Understand the structure of your kubeconfig and identify its security-sensitive fields.

**Requirements**:
- View your current kubeconfig and identify all clusters, users, and contexts
- Check the file permissions on your kubeconfig
- Identify which authentication method each user entry uses (cert, token, OIDC, etc.)

**Starter Command**:

```bash
# View the full kubeconfig (with secrets redacted)
kubectl config view

# View without redaction (CAREFUL — shows real credentials!)
kubectl config view --raw

# Check file permissions
ls -la ~/.kube/config
```

**Hint**: Use `kubectl config view -o jsonpath='{.users[*].name}'` to quickly list all user names, then inspect each one for the authentication fields.

### Exercise 2 — Context Switching Safety (Basic Exploration)

**Goal**: Practice switching contexts and observe how it affects which cluster receives your commands.

**Instructions**:
1. List all available contexts with `kubectl config get-contexts`
2. Note which context is currently active (marked with `*`)
3. Switch to a different context and verify with `kubectl config current-context`
4. Set a default namespace for that context and verify with `kubectl get pods`

**Hint**: If you only have one context, create a second one pointing to the same cluster but a different namespace using `kubectl config set-context`.

### Exercise 3 — Resource Discovery Audit (Medium Analysis)

**Goal**: Map out all resource types in a cluster and identify security-sensitive ones.

**Instructions**:
1. Run `kubectl api-resources -o wide` and save the output
2. Identify all resources that support the `create` verb
3. Find which resources are cluster-scoped vs namespace-scoped
4. List the resources in the `rbac.authorization.k8s.io` API group — these are the RBAC resources you'll work with later

**Hint**: Use `kubectl api-resources --api-group=rbac.authorization.k8s.io` to filter by API group.

### Exercise 4 — Generate a Hardened Pod Manifest (Medium Implementation)

**Goal**: Use `kubectl`'s dry-run mode to generate a pod manifest and then manually add security settings.

**Requirements**:

- Add a container-level `securityContext` with `readOnlyRootFilesystem: true` and `allowPrivilegeEscalation: false`
- Validate the manifest with `--dry-run=server`

**Starter Command**:

```bash
kubectl run secure-nginx --image=nginx:1.25 \
  --dry-run=client -o yaml > secure-pod.yaml

# TODO: Edit secure-pod.yaml to add securityContext fields
# TODO: Validate with --dry-run=server
```

**Hint**: The pod-level `securityContext` goes under `spec.securityContext`, while the container-level one goes under `spec.containers[].securityContext`.

### Exercise 5 — Permission Audit for a ServiceAccount (Medium Hardening)

**Goal**: Audit the permissions of the `default` ServiceAccount in a namespace and understand why it's a risk.

**Instructions**:
1. Check what the `default` ServiceAccount in the `default` namespace can do using `kubectl auth can-i --list --as=system:serviceaccount:default:default`
2. Check if `automountServiceAccountToken` is set on the ServiceAccount
3. Create a new, more restrictive ServiceAccount and compare its permissions
4. Document the security implications of auto-mounted tokens

**Hint**: New ServiceAccounts (in clusters v1.24+) no longer get auto-created long-lived tokens, but the `automountServiceAccountToken` flag still controls whether a projected token is mounted into pods.

### Exercise 6 — Decode Secrets and Identify the Risk (Hard Escalation)

**Goal**: Demonstrate how easily Kubernetes Secrets can be read by someone with the right RBAC permissions.

**Instructions**:
1. Create a Secret containing a simulated database password
2. Retrieve it via `kubectl get secret -o yaml` and decode the base64 value
3. Try accessing the Secret as a different user/ServiceAccount to confirm RBAC limits access
4. Document the attack path: what RBAC verbs are needed to exfiltrate all secrets from a namespace?

**Hint**: The verbs `get`, `list`, and `watch` on the `secrets` resource are all you need to enumerate and read every secret in a namespace — or cluster-wide with ClusterRole access.

### Exercise 7 — Build a Namespace Isolation Strategy (Hard Mitigation)

**Goal**: Design a namespace structure that supports RBAC-based team isolation.

**Requirements**:

### Exercise 8 — Multi-Step Security Simulation (Challenge Simulation!)

**Goal**: Simulate a scenario where a developer accidentally applies a malicious manifest and you need to detect and remediate it.

## 🗒️ Quick CLI Cheat Sheet

```bash
# === Context & Config ===
kubectl config get-contexts                    # List all contexts
kubectl config current-context                 # Show active context
kubectl config use-context <name>              # Switch context
kubectl config set-context --current --namespace=<ns>  # Set default namespace

# === Resource Inspection ===
kubectl get <resource> [-n <ns>] [-A]          # List resources
kubectl get <resource> -o yaml                 # Full YAML output
kubectl get <resource> -o json                 # JSON output
kubectl describe <resource> <name>              # Detailed info + events
kubectl explain <resource>[.field]              # Documentation for fields

# === Creating & Applying ===
kubectl apply -f <file.yaml>                   # Declarative apply
kubectl create <resource> --dry-run=client -o yaml  # Generate YAML

# === Debugging ===
kubectl logs <pod> [-f] [--previous] [-c <container>]

kubectl port-forward <pod|svc> <local>:<remote>

# === Security ===
kubectl auth can-i <verb> <resource> [-n <ns>]
```
