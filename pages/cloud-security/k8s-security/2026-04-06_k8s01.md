## 🧠 Concept Overview

A **Pod** is the smallest deployable unit in Kubernetes — not a container, but a thin wrapper around one or more containers that share a network namespace, an IPC namespace, and optionally a set of volumes. Think of it like this: in the pre-container world you'd SSH into a box, and all processes on that box share the same `localhost`, the same filesystem mount-points, and the same kernel. A Pod recreates that mental model at a much smaller scale. Every container inside a single Pod can reach another container at `127.0.0.1` on a different port, and they can share an `emptyDir` volume like a shared `/tmp`. The moment you truly internalise that a Pod is an *isolation boundary* (not just a "container"), a huge number of security decisions — like why you should almost never run two application containers in the same Pod — start to make sense.

### Why Pods Are Ephemeral (and Why That Matters for Security)

Pods are designed to be **disposable**. The kubelet creates them, the scheduler places them on a node, and if the node goes down or the container crashes beyond its restart policy, the Pod is gone forever — same PodIP, same filesystem state, everything wiped. This ephemerality is actually a security feature you get for free: a compromised Pod that gets rescheduled is rebuilt from the clean image, effectively "rotating" the workload. But this only works if you treat Pods as cattle, not pets. If you're `kubectl exec`-ing into a Pod to install packages or debug with `tcpdump`, you're undermining that ephemeral contract.

Kubernetes doesn't restart *Pods* — it restarts *containers within* a Pod (controlled by `restartPolicy`). The three values are:

| `restartPolicy` | Behaviour | Typical Use |
|-----------------|-----------|-------------|
| `Always` | Kubelet restarts the container if it exits for any reason | Long-running services (default for Deployments) |
| `OnFailure` | Kubelet restarts only on non-zero exit codes | Batch Jobs |
| `Never` | Container runs once; if it exits, it stays dead | One-shot tasks |

From a security angle, `Always` means an attacker who crashes your process gets a fresh container quickly — but it also means a persistent backdoor process can survive a simple `kill`. Understanding `restartPolicy` is key to designing resilient *and* secure workloads.

### Pod Spec Anatomy — The Fields That Actually Matter

Here's the skeleton of a Pod manifest. Every field listed below will come back in later security sessions (SecurityContext, ServiceAccount tokens, volumes), but let's plant the seeds now:

```yaml
apiVersion: v1
kind: Pod
metadata:
  name: my-app
  namespace: default          # Always be explicit — "default" namespace is an anti-pattern
  labels:
    app: my-app
    version: v1
spec:
  serviceAccountName: my-app-sa   # Which identity this Pod assumes (Day 07 deep dive)
  automountServiceAccountToken: false  # Security best practice — disable unless needed
  containers:
    - name: app
      image: myregistry.io/my-app:v1.2.3   # Always use a digest or specific tag, never :latest
      ports:
        - containerPort: 8080
      resources:
        requests:
          cpu: "100m"
          memory: "128Mi"
        limits:
          cpu: "500m"
          memory: "256Mi"
      securityContext:       # Container-level security settings (Day 0x0B deep dive)
        runAsNonRoot: true
        readOnlyRootFilesystem: true
        allowPrivilegeEscalation: false
  restartPolicy: Always
```

A few things to flag right away:

1. **`namespace: default`** — The `default` namespace has no security policies applied out of the box. Production workloads should always target a dedicated namespace with Pod Security Standards enforced.
2. **`automountServiceAccountToken: false`** — By default, Kubernetes mounts a JWT token at `/var/run/secrets/kubernetes.io/serviceaccount/token` in every Pod. If your app doesn't need to talk to the API server, turn this off. An attacker who compromises the container gets a free API token otherwise.
3. **`image: ...:v1.2.3`** — Using `:latest` is a security risk because the tag is mutable. Someone could push a compromised image under the same `:latest` tag and your next Pod restart pulls the backdoored version. Prefer **image digests** (`myregistry.io/my-app@sha256:abc123...`) for absolute immutability.
4. **Resource limits** — Without CPU/memory limits, a single compromised Pod can starve the entire node (a denial-of-service attack from inside the cluster). Always set them.

### Multi-Container Pod Patterns

Sometimes you legitimately need more than one container in a Pod. Kubernetes formalises three patterns:

**Sidecar**: A helper container that augments the main app — log shippers (Fluentd), service-mesh proxies (Envoy/Istio sidecar), or TLS terminators. The sidecar shares the Pod's network and filesystem.

**Init Container**: A container that runs *before* the main containers start. Used for pre-flight checks (waiting for a database to be ready), populating shared volumes, or running database migrations. Init containers run sequentially and must exit successfully (exit code 0) before the Pod's primary containers start.

```yaml
spec:
  initContainers:
    - name: wait-for-db
      image: busybox:1.36
      command: ['sh', '-c', 'until nc -z postgres-svc 5432; do sleep 2; done']
  containers:
    - name: app
      image: myregistry.io/my-app:v1.2.3
```

**Ephemeral Containers** (debugging only): Introduced as a stable feature in K8s 1.25, ephemeral containers let you `kubectl debug` a running Pod without rebuilding it. They can't have ports, readiness probes, or resource limits. Primarily used for troubleshooting, but be aware — if RBAC lets a user create ephemeral containers, they can inject a shell into a running Pod.

### Security Perspective on Pods

From a security angle, the Pod boundary is where the rubber meets the road:

- **Attack surface**: Every container in a Pod shares the same network namespace. If one container is compromised, the attacker can sniff traffic to other containers at `localhost`. This is why you should minimise the number of containers per Pod.
- **Volume sharing**: Containers in the same Pod can share `emptyDir` volumes. A compromised sidecar with write access to a shared volume can tamper with files the main app reads.
- **Process namespace sharing**: If `shareProcessNamespace: true` is set on the Pod, all containers share PID 1. An attacker in one container can `kill` processes in another or read `/proc/<pid>/environ` to steal environment variables (including secrets passed as env vars).

## Deployments & ReplicaSets — Declarative Workload Management

Now, you'd almost *never* create a bare Pod in production. Instead, you declare a **Deployment**, which manages a **ReplicaSet**, which in turn manages Pods.

```
Deployment  ──manages──▸  ReplicaSet  ──manages──▸  Pod(s)
```

A Deployment is the object you interact with. You tell it "I want 3 replicas of my app, running image v1.2.3, with these resource limits." The Deployment controller creates a ReplicaSet, and the ReplicaSet controller ensures exactly 3 Pods are running at all times. If a node dies and takes a Pod with it, the ReplicaSet notices and schedules a replacement.

### Deployment Manifest — Full Example

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-frontend
  namespace: production
  labels:
    app: web-frontend
    team: platform
spec:
  replicas: 3
  selector:
    matchLabels:
      app: web-frontend        # Must match .spec.template.metadata.labels
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1              # At most 1 extra Pod during update
      maxUnavailable: 0        # Zero downtime — never take a Pod down before the new one is Ready
  template:
    metadata:
      labels:
        app: web-frontend
    spec:
      serviceAccountName: web-frontend-sa
      automountServiceAccountToken: false
      securityContext:          # Pod-level security context
        runAsNonRoot: true
        runAsUser: 1000
        runAsGroup: 1000
        fsGroup: 1000
        seccompProfile:
          type: RuntimeDefault
      containers:
        - name: nginx
          image: nginx:1.25.4-alpine
          ports:
            - containerPort: 8080
          resources:
            requests:
              cpu: "50m"
              memory: "64Mi"
            limits:
              cpu: "200m"
              memory: "128Mi"
          securityContext:
            allowPrivilegeEscalation: false
            readOnlyRootFilesystem: true
            capabilities:
              drop:
                - ALL
          volumeMounts:
            - name: tmp
              mountPath: /tmp
            - name: cache
              mountPath: /var/cache/nginx
            - name: run
              mountPath: /var/run
          livenessProbe:
            httpGet:
              path: /healthz
              port: 8080
            initialDelaySeconds: 5
            periodSeconds: 10
          readinessProbe:
            httpGet:
              path: /ready
              port: 8080
            initialDelaySeconds: 3
            periodSeconds: 5
      volumes:
        - name: tmp
          emptyDir: {}
        - name: cache
          emptyDir: {}
        - name: run
          emptyDir: {}
```

Let's break down the security-relevant decisions in this manifest:

1. **`runAsNonRoot: true` + `runAsUser: 1000`**: The container process will never run as UID 0 (root). If the image's `Dockerfile` has `USER root`, the Pod will fail to start — which is exactly what you want. Running as root inside a container significantly increases the blast radius of a compromise.
2. **`readOnlyRootFilesystem: true`**: The container's root filesystem is mounted read-only. An attacker can't drop a binary into `/tmp` or modify `/etc/passwd`. We mount writable `emptyDir` volumes only where the app actually needs to write (nginx needs `/tmp`, `/var/cache/nginx`, `/var/run`).
3. **`capabilities: drop: [ALL]`**: Linux capabilities are fine-grained root powers. Dropping ALL and adding back only what's needed is the gold standard. Nginx listening on port 8080 (above 1024) doesn't even need `NET_BIND_SERVICE`.
4. **`seccompProfile: RuntimeDefault`**: Applies the container runtime's default Seccomp profile, which blocks dangerous syscalls like `reboot`, `mount`, and `ptrace`. We'll dive deep into Seccomp in Day 0x0F.
5. **`automountServiceAccountToken: false`**: No API token mounted. This Pod doesn't need to talk to the Kubernetes API.
6. **`strategy: RollingUpdate` with `maxUnavailable: 0`**: Zero-downtime deploys — new Pods must pass readiness probes before old Pods are terminated. This also has a security angle: it prevents an attacker from triggering a deploy that takes all replicas offline (denial-of-service via forced rollout).

### ReplicaSets Under the Hood

You rarely interact with ReplicaSets directly, but understanding them matters:

- A Deployment creates a **new** ReplicaSet every time you change the Pod template (e.g., update the image tag). The old ReplicaSet is scaled down, the new one is scaled up — this is the rolling update.
- Old ReplicaSets are kept around (controlled by `.spec.revisionHistoryLimit`, default 10) so you can `kubectl rollout undo` to a previous version.
- Each ReplicaSet has an `ownerReference` pointing to its parent Deployment. If you delete the Deployment, all its ReplicaSets and Pods are garbage-collected.

```bash
# See all ReplicaSets managed by a Deployment
kubectl get rs -n production -l app=web-frontend

# Check rollout history
kubectl rollout history deployment/web-frontend -n production

# Roll back to the previous version
kubectl rollout undo deployment/web-frontend -n production

# Roll back to a specific revision
kubectl rollout undo deployment/web-frontend -n production --to-revision=3
```

### Scaling — Horizontal vs. Desired State

```bash
# Manual scaling
kubectl scale deployment web-frontend -n production --replicas=5

# Or edit the manifest and apply
kubectl apply -f deployment.yaml
```

The ReplicaSet controller continuously reconciles: if current Pod count < desired, it creates Pods; if current > desired, it terminates Pods. This reconciliation loop is one of Kubernetes' core strengths, but it also means that an attacker with `patch` or `update` permissions on a Deployment can scale it to zero (instant denial-of-service) or scale it to a hundred (resource exhaustion on the cluster).

### Labels, Selectors, and the Security Implications

Labels are key-value pairs on Kubernetes objects. Selectors are queries that match those labels. The relationship between a Deployment, its ReplicaSet, and its Pods is entirely label-based:

```yaml
spec:
  selector:
    matchLabels:
      app: web-frontend    # "Find Pods with this label"
  template:
    metadata:
      labels:
        app: web-frontend  # "Give new Pods this label"
```

If an attacker can modify labels on a Pod (e.g., remove `app: web-frontend`), the ReplicaSet loses track of it, thinks it's short one Pod, and creates a replacement. The "orphaned" Pod keeps running — outside the management of the Deployment. This is a known technique for stealthily keeping a compromised Pod alive while the Deployment moves on.

## 🔑 Key Points

- A **Pod** is the smallest deployable unit — it wraps one or more containers that share network, IPC, and optionally PID namespaces plus volumes.
- Pods are **ephemeral by design**. Never store persistent state in a Pod's local filesystem; it will vanish on rescheduling.
- Always set **`automountServiceAccountToken: false`** unless the Pod genuinely needs to call the Kubernetes API.
- Use **Deployments** (not bare Pods) for all production workloads — they give you declarative updates, rollbacks, and self-healing via ReplicaSets.
- The label-selector relationship between Deployments, ReplicaSets, and Pods is critical: misconfigured or attacker-manipulated labels can orphan or hijack Pods.

## 💻 Code & Configuration Examples

### Example 1 — Bare Pod vs. Deployment-Managed Pod

```yaml
# BAD: A bare Pod — if the node dies, this Pod is gone forever
apiVersion: v1
kind: Pod
metadata:
  name: lonely-app
spec:
  containers:
    - name: app
      image: myregistry.io/app:v1
---
# GOOD: Same app wrapped in a Deployment — self-healing, rollbacks, scaling
apiVersion: apps/v1
kind: Deployment
metadata:
  name: resilient-app
  namespace: production
spec:
  replicas: 2
  selector:
    matchLabels:
      app: resilient-app
  template:
    metadata:
      labels:
        app: resilient-app
    spec:
      containers:
        - name: app
          image: myregistry.io/app:v1
          resources:
            requests:
              cpu: "100m"
              memory: "128Mi"
            limits:
              cpu: "500m"
              memory: "256Mi"
```

**What this demonstrates**: Bare Pods have no controller watching them — they cannot self-heal after node failure. Deployments provide declarative desired-state management, automatic rescheduling, and rolling updates.

### Example 2 — Hardened Deployment with Init Container and Security Context

```yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: api-server
  namespace: backend
spec:
  replicas: 3
  selector:
    matchLabels:
      app: api-server
  template:
    metadata:
      labels:
        app: api-server
    spec:
      serviceAccountName: api-server-sa
      automountServiceAccountToken: false
      securityContext:
        runAsNonRoot: true
        runAsUser: 65534        # 'nobody' user
        runAsGroup: 65534
        fsGroup: 65534
        seccompProfile:
          type: RuntimeDefault
      initContainers:
        - name: db-migration
          image: myregistry.io/migrate:v2.1.0
          command: ['./migrate', '--direction=up']
          securityContext:
            allowPrivilegeEscalation: false
            readOnlyRootFilesystem: true
            capabilities:
              drop: ["ALL"]
      containers:
        - name: api
          image: myregistry.io/api-server:v3.4.2
          ports:
            - containerPort: 8443
          env:
            - name: DB_HOST
              valueFrom:
                configMapKeyRef:
                  name: api-config
                  key: db_host
            - name: DB_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: api-secrets
                  key: db_password
          securityContext:
            allowPrivilegeEscalation: false
            readOnlyRootFilesystem: true
            capabilities:
              drop: ["ALL"]
          resources:
            requests:
              cpu: "200m"
              memory: "256Mi"
            limits:
              cpu: "1"
              memory: "512Mi"
          volumeMounts:
            - name: tmp
              mountPath: /tmp
      volumes:
        - name: tmp
          emptyDir:
            sizeLimit: "50Mi"   # Prevent an attacker from filling the node's disk
```

**What this demonstrates**: A fully hardened Deployment with: init container for database migrations (runs once, exits), non-root execution with the `nobody` user, read-only root filesystem, all Linux capabilities dropped, Seccomp RuntimeDefault profile, explicit resource limits, and a size-limited `emptyDir` for temporary writes.

### Example 3 — Observing ReplicaSet Behaviour During a Rolling Update

```bash
# Deploy v1
kubectl apply -f deployment-v1.yaml

# Watch Pods in real-time
kubectl get pods -n production -w

# Update the image to v2 (triggers rolling update)
kubectl set image deployment/api-server api=myregistry.io/api-server:v3.5.0 -n backend

# Watch the rollout progress
kubectl rollout status deployment/api-server -n backend

# See both old and new ReplicaSets
kubectl get rs -n backend -l app=api-server

# NAME                          DESIRED   CURRENT   READY   AGE
# api-server-6f8d9c7b44         3         3         3       2m    ← new RS (v3.5.0)
# api-server-5b7c8a6d33         0         0         0       10m   ← old RS (v3.4.2)

# Roll back if something goes wrong
kubectl rollout undo deployment/api-server -n backend
```

**What this demonstrates**: How Deployments orchestrate zero-downtime updates by managing multiple ReplicaSets, and how `rollout undo` provides instant rollback capability.

## ⚠️ Common Misconfigurations & Risks

| Misconfiguration | Risk Assessment | Secure Approach |
|------------------|----------------|-----------------|
| Running bare Pods without a controller (Deployment/StatefulSet) | Pod disappears permanently on node failure; no rollback mechanism; no scaling | Always use a Deployment or StatefulSet as the workload controller |
| Using `:latest` image tag | Image is mutable — a compromised registry push replaces it silently on the next Pod restart | Use immutable tags (e.g., `v1.2.3`) or image digests (`@sha256:...`) |
| No resource `requests`/`limits` set | A single compromised container can consume all node CPU/memory, causing cluster-wide DoS | Always set both `requests` and `limits` for CPU and memory |
| `automountServiceAccountToken` left as default (`true`) | Every Pod gets a JWT that can authenticate to the K8s API server — free credential for attackers | Set to `false` at Pod or ServiceAccount level unless the app needs API access |
| Running containers as `root` (UID 0) | Container escape vulnerabilities are far more severe when the process is root — immediate node-level access | Set `runAsNonRoot: true` and `runAsUser` to a non-root UID |
| `readOnlyRootFilesystem` not set | Attacker can write binaries, cron jobs, or SSH keys into the container filesystem | Set `readOnlyRootFilesystem: true`, mount writable `emptyDir` only where needed |
| Labels not protected by RBAC | An attacker who can patch labels can orphan Pods from their ReplicaSet controller | Restrict `patch`/`update` verbs on Pod resources via RBAC |
| Setting `shareProcessNamespace: true` without need | All containers share PIDs — attacker can `kill` other processes or read `/proc/*/environ` secrets | Only enable process namespace sharing when explicitly required |

   - Does it have resource limits?
   - Is it using the default service account?

3. **Contain**: Scale the Deployment to 0 replicas to stop the bleeding. Then quarantine the namespace:
   ```bash
   kubectl scale deployment suspicious-workload -n compromised-ns --replicas=0
   ```

4. **Remediate**: Write a hardened version of this Deployment that:
   - Uses a minimal base image (e.g., `distroless`)
   - Has explicit resource limits
   - Runs as non-root with read-only filesystem
   - Has all capabilities dropped
   - Has the service account token disabled

5. **Document**: Write a brief incident report covering what was wrong, what you did, and what controls would have prevented this.

**Hint**: Use `kubectl describe deployment`, `kubectl get pods -o yaml`, and `kubectl top pods` (if metrics-server is installed) to gather evidence before containment.

## 🗒️ Quick CLI Cheat Sheet

```bash
# Create a namespace
kubectl create namespace myns

# Apply a manifest
kubectl apply -f deployment.yaml

# List all Pods in a namespace
kubectl get pods -n myns

# Watch Pods in real-time
kubectl get pods -n myns -w

# Describe a specific Pod (events, status, containers)
kubectl describe pod <pod-name> -n myns

# View Pod logs
kubectl logs <pod-name> -n myns
kubectl logs <pod-name> -c <container-name> -n myns  # specific container

# Exec into a running container (debugging only!)
kubectl exec -it <pod-name> -n myns -- /bin/sh

# Scale a Deployment
kubectl scale deployment <name> -n myns --replicas=5

# Trigger a rolling update
kubectl set image deployment/<name> <container>=<new-image> -n myns

# Check rollout status
kubectl rollout status deployment/<name> -n myns

# View rollout history
kubectl rollout history deployment/<name> -n myns

# Rollback to previous version
kubectl rollout undo deployment/<name> -n myns

# View ReplicaSets
kubectl get rs -n myns

# Delete a resource
kubectl delete deployment <name> -n myns

# View the full YAML of a running Pod
kubectl get pod <pod-name> -n myns -o yaml
```
