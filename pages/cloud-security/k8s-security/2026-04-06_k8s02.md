## 🐳 Step 0 — Dockerfile: Packaging `web.jar` the Right Way

Before anything goes into Kubernetes, the image needs to be production-hardened. A common mistake is using a fat JDK base image and running the JAR as root — both are unnecessary attack surface.

### Multi-Stage Build with Non-Root User

```dockerfile
# ── Stage 1: Extract JAR layers for better caching ──────────────────────────
FROM eclipse-temurin:21-jre-alpine AS builder

WORKDIR /app

# Copy the fat JAR built by Maven/Gradle
COPY target/web.jar web.jar

# Use Spring Boot's built-in layer extraction for optimised Docker layers
RUN java -Djarmode=layertools -jar web.jar extract

# ── Stage 2: Minimal runtime image ──────────────────────────────────────────
FROM eclipse-temurin:21-jre-alpine

# Security: create a dedicated non-root user and group
RUN addgroup -S appgroup && adduser -S appuser -G appgroup

WORKDIR /app

# Copy layered application from builder (better caching: deps change rarely)
COPY --from=builder --chown=appuser:appgroup /app/dependencies/ ./
COPY --from=builder --chown=appuser:appgroup /app/spring-boot-loader/ ./
COPY --from=builder --chown=appuser:appgroup /app/snapshot-dependencies/ ./
COPY --from=builder --chown=appuser:appgroup /app/application/ ./

# Switch to non-root user
USER appuser

# JVM tuning for containers (respect cgroup memory limits)
ENV JAVA_OPTS="-XX:+UseContainerSupport \
               -XX:MaxRAMPercentage=75.0 \
               -Djava.security.egd=file:/dev/./urandom \
               -Dspring.backgroundpreinitializer.ignore=true"

EXPOSE 8080

# Health check baked into the image
HEALTHCHECK --interval=30s --timeout=5s --start-period=60s --retries=3 \
  CMD wget -qO- http://localhost:8080/actuator/health || exit 1

ENTRYPOINT ["sh", "-c", "java $JAVA_OPTS org.springframework.boot.loader.launch.JarLauncher"]
```

**Build & Push**:

```bash
# Build the image
docker build -t myregistry.io/web-app:v1.0.0 .

# Verify non-root user
docker run --rm myregistry.io/web-app:v1.0.0 id
# Expected: uid=100(appuser) gid=101(appgroup)

# Push to registry
docker push myregistry.io/web-app:v1.0.0

# Get the image digest (use this in your K8s manifests for immutability)
docker inspect --format='{{index .RepoDigests 0}}' myregistry.io/web-app:v1.0.0
```

> **Security note**: `eclipse-temurin:21-jre-alpine` is ~190MB and contains only the JRE, not the JDK. No `javac`, no `jshell`, no compilation tools for an attacker to use.

## 📦 Step 1 — Namespace, ResourceQuota & LimitRange (`00-namespace.yaml`)

Always give your production workload its own namespace with hard resource ceilings. Without these, a single runaway Pod can OOM-kill everything on the node.

```yaml
# 00-namespace.yaml
---
apiVersion: v1
kind: Namespace
metadata:
  name: web-prod
  labels:
    env: production
    # Enforce the "restricted" Pod Security Standard (Day 0x0D/0x0E topic)
    pod-security.kubernetes.io/enforce: restricted
    pod-security.kubernetes.io/enforce-version: latest
    pod-security.kubernetes.io/warn: restricted
    pod-security.kubernetes.io/warn-version: latest

---
# Hard ceiling for the entire namespace
apiVersion: v1
kind: ResourceQuota
metadata:
  name: web-prod-quota
  namespace: web-prod
spec:
  hard:
    requests.cpu: "4"
    requests.memory: "4Gi"
    limits.cpu: "8"
    limits.memory: "8Gi"
    pods: "20"
    services: "10"
    persistentvolumeclaims: "5"
    secrets: "20"
    configmaps: "20"

---
# Default min/max per container — prevents Pods without resource specs from starting
apiVersion: v1
kind: LimitRange
metadata:
  name: web-prod-limits
  namespace: web-prod
spec:
  limits:
    - type: Container
      default:           # applied if the container has no limits set
        cpu: "500m"
        memory: "512Mi"
      defaultRequest:    # applied if the container has no requests set
        cpu: "100m"
        memory: "256Mi"
      max:
        cpu: "2"
        memory: "2Gi"
      min:
        cpu: "50m"
        memory: "64Mi"
```

```bash
kubectl apply -f 00-namespace.yaml

# Verify
kubectl describe namespace web-prod
kubectl describe resourcequota web-prod-quota -n web-prod
kubectl describe limitrange web-prod-limits -n web-prod
```

## ⚙️ Step 2 — ConfigMap & Secret (`01-configmap.yaml` / `02-secret.yaml`)

Never bake configuration into your image. Separate non-sensitive config (ConfigMap) from credentials (Secret).

### ConfigMap — Non-Sensitive Application Settings

```yaml
# 01-configmap.yaml
apiVersion: v1
kind: ConfigMap
metadata:
  name: web-app-config
  namespace: web-prod
data:
  # Spring Boot application properties
  SPRING_PROFILES_ACTIVE: "prod"
  SERVER_PORT: "8080"
  MANAGEMENT_SERVER_PORT: "8081"      # Actuator on a separate port (better isolation)
  MANAGEMENT_ENDPOINTS_WEB_EXPOSURE_INCLUDE: "health,info,prometheus"
  DB_HOST: "postgres.database.svc.cluster.local"
  DB_PORT: "5432"
  DB_NAME: "webapp"
  LOG_LEVEL: "INFO"
  # JVM options passed via env (overrides Dockerfile ENV at runtime)
  JAVA_OPTS: >-
    -XX:+UseContainerSupport
    -XX:MaxRAMPercentage=75.0
    -Djava.security.egd=file:/dev/./urandom
```

### Secret — Sensitive Credentials

```bash
# Create the Secret imperatively (avoids base64 in YAML committed to git)
kubectl create secret generic web-app-secrets \
  --from-literal=DB_USERNAME=webapp_user \
  --from-literal=DB_PASSWORD='S3cur3P@ssw0rd!' \
  --from-literal=JWT_SECRET='your-256-bit-jwt-secret-here' \
  --namespace web-prod

# Or if you must use a YAML (values are base64-encoded):
```

```yaml
# 02-secret.yaml  (DO NOT commit real values to git — use sealed-secrets or External Secrets Operator)
apiVersion: v1
kind: Secret
metadata:
  name: web-app-secrets
  namespace: web-prod
type: Opaque
stringData:           # stringData lets you write plain text; K8s encodes it automatically
  DB_USERNAME: "webapp_user"
  DB_PASSWORD: "S3cur3P@ssw0rd!"
  JWT_SECRET: "your-256-bit-jwt-secret-here"
```

```bash
kubectl apply -f 01-configmap.yaml
kubectl apply -f 02-secret.yaml

# Verify (values are base64 encoded in the output)
kubectl get secret web-app-secrets -n web-prod -o yaml
```

> **⚠️ Production tip**: Never commit real secrets in YAML. Use [Sealed Secrets](https://sealed-secrets.netlify.app/) or [External Secrets Operator](https://external-secrets.io/) to sync from AWS Secrets Manager / HashiCorp Vault.

## 🚀 Step 3 — Deployment: 2 Replicas, Production-Hardened (`03-deployment.yaml`)

This is the centrepiece. Every field is deliberately chosen.

```yaml
# 03-deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: web-app
  namespace: web-prod
  labels:
    app: web-app
    version: v1.0.0
    team: platform
  annotations:
    deployment.kubernetes.io/revision: "1"
spec:
  replicas: 2                       # Two Pods for immediate HA

  selector:
    matchLabels:
      app: web-app

  # Zero-downtime rolling update strategy
  strategy:
    type: RollingUpdate
    rollingUpdate:
      maxSurge: 1          # Temporarily allow 3 Pods during update (2+1)
      maxUnavailable: 0    # Never bring a Pod down before the new one is Ready

  revisionHistoryLimit: 5  # Keep last 5 ReplicaSets for rollback

  template:
    metadata:
      labels:
        app: web-app
        version: v1.0.0
      annotations:
        # Force Pod restart when ConfigMap/Secret content changes
        checksum/config: "{{ sha256 of configmap content }}"

    spec:
      # ── Identity & Token ──────────────────────────────────────────────────
      serviceAccountName: web-app-sa
      automountServiceAccountToken: false   # App doesn't need K8s API access

      # ── Scheduling: spread Pods across nodes ──────────────────────────────
      topologySpreadConstraints:
        - maxSkew: 1
          topologyKey: kubernetes.io/hostname
          whenUnsatisfiable: DoNotSchedule
          labelSelector:
            matchLabels:
              app: web-app

      # ── Pod-level Security Context ────────────────────────────────────────
      securityContext:
        runAsNonRoot: true
        runAsUser: 100               # matches the 'appuser' UID in the Dockerfile
        runAsGroup: 101
        fsGroup: 101
        seccompProfile:
          type: RuntimeDefault       # Block dangerous syscalls at the kernel level

      # ── Graceful Shutdown ─────────────────────────────────────────────────
      terminationGracePeriodSeconds: 60    # Give Spring Boot time to drain connections

      containers:
        - name: web-app
          image: myregistry.io/web-app:v1.0.0    # Always pin to a specific tag/digest
          imagePullPolicy: Always

          ports:
            - name: http
              containerPort: 8080
              protocol: TCP
            - name: actuator
              containerPort: 8081
              protocol: TCP

          # ── Environment Variables from ConfigMap ──────────────────────────
          envFrom:
            - configMapRef:
                name: web-app-config

          # ── Sensitive env vars from Secret ────────────────────────────────
          env:
            - name: DB_USERNAME
              valueFrom:
                secretKeyRef:
                  name: web-app-secrets
                  key: DB_USERNAME
            - name: DB_PASSWORD
              valueFrom:
                secretKeyRef:
                  name: web-app-secrets
                  key: DB_PASSWORD
            - name: JWT_SECRET
              valueFrom:
                secretKeyRef:
                  name: web-app-secrets
                  key: JWT_SECRET

          # ── Resource Requests & Limits ────────────────────────────────────
          resources:
            requests:
              cpu: "250m"      # 0.25 core guaranteed to the Pod
              memory: "512Mi"
            limits:
              cpu: "1"         # Burst up to 1 core
              memory: "1Gi"    # OOM-killed if it exceeds 1GiB

          # ── Container-level Security Context ─────────────────────────────
          securityContext:
            allowPrivilegeEscalation: false
            readOnlyRootFilesystem: true
            capabilities:
              drop:
                - ALL

          # ── Writable Volume Mounts (for read-only root filesystem) ────────
          volumeMounts:
            - name: tmp
              mountPath: /tmp
            - name: app-logs
              mountPath: /app/logs

          # ── Startup Probe — wait for JVM & Spring context to boot ─────────
          startupProbe:
            httpGet:
              path: /actuator/health
              port: actuator         # 8081 — actuator port
            initialDelaySeconds: 20  # Spring Boot apps typically take 10–30s
            periodSeconds: 5
            failureThreshold: 12     # 20s + (12 * 5s) = 80s max startup window

          # ── Liveness Probe — restart if the app is deadlocked ─────────────
          livenessProbe:
            httpGet:
              path: /actuator/health/liveness
              port: actuator
            initialDelaySeconds: 0   # startupProbe gates this; no extra delay needed
            periodSeconds: 10
            failureThreshold: 3
            timeoutSeconds: 5

          # ── Readiness Probe — stop sending traffic if app is overloaded ───
          readinessProbe:
            httpGet:
              path: /actuator/health/readiness
              port: actuator
            initialDelaySeconds: 0
            periodSeconds: 5
            failureThreshold: 3
            timeoutSeconds: 3

          # ── Graceful Shutdown Hook ────────────────────────────────────────
          lifecycle:
            preStop:
              exec:
                # Wait 15s to let the load balancer drain connections before SIGTERM
                command: ["/bin/sh", "-c", "sleep 15"]

      # ── Volumes ───────────────────────────────────────────────────────────
      volumes:
        - name: tmp
          emptyDir:
            sizeLimit: "100Mi"
        - name: app-logs
          emptyDir:
            sizeLimit: "500Mi"
```

```bash
kubectl apply -f 03-deployment.yaml

# Watch the rollout
kubectl rollout status deployment/web-app -n web-prod

# Verify both Pods are running
kubectl get pods -n web-prod -l app=web-app -o wide

# Check that Pods land on DIFFERENT nodes (topology spread)
kubectl get pods -n web-prod -l app=web-app \
  -o custom-columns='NAME:.metadata.name,NODE:.spec.nodeName'
```

### Spring Boot Actuator Configuration (required for probes)

Add to `application-prod.yaml` in your JAR:

```yaml
# src/main/resources/application-prod.yaml
management:
  server:
    port: 8081
  endpoint:
    health:
      enabled: true
      probes:
        enabled: true        # Enables /actuator/health/liveness and /readiness
      show-details: never    # Don't expose sensitive internals externally
  endpoints:
    web:
      exposure:
        include: health,info,prometheus

server:
  shutdown: graceful         # Spring Boot drains in-flight requests before shutdown
spring:
  lifecycle:
    timeout-per-shutdown-phase: 30s
```

## 🔗 Step 4 — Service: Internal Load Balancer (`04-service.yaml`)

The Service is what gives you the stable DNS name and distributes traffic across both Pods via kube-proxy iptables rules (or IPVS in high-traffic clusters).

```yaml
# 04-service.yaml
apiVersion: v1
kind: Service
metadata:
  name: web-app-svc
  namespace: web-prod
  labels:
    app: web-app
spec:
  type: ClusterIP   # Internal only — Ingress will be the external entry point

  selector:
    app: web-app    # Targets all Pods with this label (both replicas)

  ports:
    - name: http
      port: 80            # Port the Service listens on
      targetPort: http    # Named port 'http' on the container (8080)
      protocol: TCP

  # Optional: enable sticky sessions (clients stick to the same Pod)
  # Useful if your app has server-side session state (avoid if stateless/JWT)
  # sessionAffinity: ClientIP
  # sessionAffinityConfig:
  #   clientIP:
  #     timeoutSeconds: 3600
```

```bash
kubectl apply -f 04-service.yaml

# Verify endpoints — should show 2 IPs (one per Pod)
kubectl get endpoints web-app-svc -n web-prod

# Test internal connectivity from a debug pod
kubectl run debug --image=curlimages/curl:8.7.1 -it --rm -n web-prod -- \
  sh -c 'for i in $(seq 1 6); do curl -s http://web-app-svc/actuator/health | jq .status; done'
# You can observe traffic spreading across both Pods in the application logs
```

**How load balancing works**: Each `ClusterIP` Service gets a virtual IP. `kube-proxy` on every node writes iptables rules that randomly distribute connections to one of the endpoint Pod IPs. With 2 Pods, each gets ~50% of traffic. No additional configuration is needed — this works out of the box.

## 🌐 Step 5 — Ingress with TLS (`05-ingress.yaml`)

The Ingress is what terminates HTTPS from the internet and routes the decrypted HTTP traffic to the Service.

```yaml
# 05-ingress.yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: web-app-ingress
  namespace: web-prod
  annotations:
    # Nginx-specific: tweak upstream behaviour
    nginx.ingress.kubernetes.io/proxy-read-timeout: "60"
    nginx.ingress.kubernetes.io/proxy-send-timeout: "60"
    nginx.ingress.kubernetes.io/proxy-body-size: "10m"

    # Rate limiting: 50 requests/second per IP (DDoS mitigation)
    nginx.ingress.kubernetes.io/limit-rps: "50"
    nginx.ingress.kubernetes.io/limit-connections: "20"

    # Security headers
    nginx.ingress.kubernetes.io/configuration-snippet: |
      add_header X-Frame-Options "SAMEORIGIN" always;
      add_header X-Content-Type-Options "nosniff" always;
      add_header X-XSS-Protection "1; mode=block" always;
      add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
      add_header Referrer-Policy "strict-origin-when-cross-origin" always;

    # cert-manager: automatically provision & renew a Let's Encrypt certificate
    cert-manager.io/cluster-issuer: "letsencrypt-prod"

spec:
  ingressClassName: nginx

  tls:
    - hosts:
        - api.example.com
      secretName: web-app-tls   # cert-manager will create this Secret automatically

  rules:
    - host: api.example.com
      http:
        paths:
          - path: /
            pathType: Prefix
            backend:
              service:
                name: web-app-svc
                port:
                  name: http
```

```bash
kubectl apply -f 05-ingress.yaml

# Check certificate issuance (cert-manager must be installed)
kubectl get certificate -n web-prod
kubectl describe certificate web-app-tls -n web-prod

# Test the live HTTPS endpoint
curl -v https://api.example.com/actuator/health
```

## 📈 Step 6 — HorizontalPodAutoscaler (`06-hpa.yaml`)

The HPA watches CPU and memory metrics and automatically scales the Deployment between a minimum of 2 and a maximum of 10 replicas. The fixed `replicas: 2` in the Deployment becomes the **floor** once the HPA takes over.

```yaml
# 06-hpa.yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: web-app-hpa
  namespace: web-prod
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web-app

  minReplicas: 2      # Never drop below 2 — instant HA guarantee
  maxReplicas: 10     # Controlled cost ceiling

  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70    # Scale out when average CPU crosses 70%

    - type: Resource
      resource:
        name: memory
        target:
          type: Utilization
          averageUtilization: 80    # Scale out when average memory crosses 80%

  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60     # Wait 60s before scaling up again
      policies:
        - type: Pods
          value: 2                        # Add at most 2 Pods per 60s burst
          periodSeconds: 60
    scaleDown:
      stabilizationWindowSeconds: 300    # Wait 5 minutes before scaling down (avoid flapping)
      policies:
        - type: Pods
          value: 1                        # Remove at most 1 Pod per 120s
          periodSeconds: 120
```

```bash
kubectl apply -f 06-hpa.yaml

# Check HPA status (metrics-server must be installed)
kubectl get hpa -n web-prod

# Generate some load to trigger scale-out
kubectl run load-gen --image=busybox:1.36 -it --rm -n web-prod -- \
  sh -c 'while true; do wget -qO- http://web-app-svc/; done'

# Watch the HPA react
kubectl get hpa web-app-hpa -n web-prod --watch
```

## 🛡️ Step 7 — PodDisruptionBudget (`07-pdb.yaml`)

Without a PDB, a node drain (for maintenance or upgrades) can evict all your Pods simultaneously, causing a complete outage. The PDB tells Kubernetes: "never take more than 1 Pod offline at a time."

```yaml
# 07-pdb.yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: web-app-pdb
  namespace: web-prod
spec:
  selector:
    matchLabels:
      app: web-app

  # At least 1 Pod must always be available during voluntary disruptions
  minAvailable: 1

  # Alternatively, express as "never make more than 1 Pod unavailable":
  # maxUnavailable: 1
```

```bash
kubectl apply -f 07-pdb.yaml

# Verify
kubectl get pdb -n web-prod
kubectl describe pdb web-app-pdb -n web-prod

# Simulate a node drain — the PDB will protect 1 Pod
kubectl drain <node-name> --ignore-daemonsets --delete-emptydir-data
# K8s will evict one Pod at a time, respecting the PDB min=1
```

## 🔒 Step 8 — NetworkPolicy: Least-Privilege Traffic (`08-netpol.yaml`)

By default, all Pods in a cluster can freely talk to each other — a serious security risk. NetworkPolicies act like a firewall at the Pod level. We enforce that:

1. Only the Ingress controller can send traffic to the web-app Pods
2. The web-app Pods can only reach the database on port 5432
3. All other ingress/egress is denied

```yaml
# 08-netpol.yaml
---
# Default-deny ALL ingress and egress for the namespace
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: web-prod
spec:
  podSelector: {}     # Applies to ALL Pods in the namespace
  policyTypes:
    - Ingress
    - Egress

---
# Allow Ingress controller → web-app on port 8080
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-ingress-to-web-app
  namespace: web-prod
spec:
  podSelector:
    matchLabels:
      app: web-app
  policyTypes:
    - Ingress
  ingress:
    - from:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: ingress-nginx   # Ingress controller namespace
          podSelector:
            matchLabels:
              app.kubernetes.io/name: ingress-nginx        # Ingress controller Pod label
      ports:
        - protocol: TCP
          port: 8080

---
# Allow web-app → external DB on port 5432 (egress)
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-web-app-to-db
  namespace: web-prod
spec:
  podSelector:
    matchLabels:
      app: web-app
  policyTypes:
    - Egress
  egress:
    - to:
        - ipBlock:
            cidr: 10.0.10.5/32    # Replace with your DB IP or CIDR
      ports:
        - protocol: TCP
          port: 5432

---
# Allow web-app → kube-dns (required for any DNS resolution)
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-dns
  namespace: web-prod
spec:
  podSelector:
    matchLabels:
      app: web-app
  policyTypes:
    - Egress
  egress:
    - to:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: kube-system
          podSelector:
            matchLabels:
              k8s-app: kube-dns
      ports:
        - protocol: UDP
          port: 53
        - protocol: TCP
          port: 53
```

```bash
kubectl apply -f 08-netpol.yaml

# Verify policies are in place
kubectl get networkpolicy -n web-prod
```

## ⚠️ Production Gotchas & Common Mistakes

| Issue | Consequence | Fix |
|-------|-------------|-----|
| No `startupProbe` on a slow JVM | Liveness probe fires before Spring context loads → Pod restart loop | Add `startupProbe` with generous `failureThreshold` (80–90s window) |
| `maxUnavailable: 1` in rolling update | Traffic spikes to only 1 Pod during deploys | Set `maxUnavailable: 0` and `maxSurge: 1` for zero-downtime |
| Missing `preStop` sleep hook | Load balancer still sends requests to a terminating Pod → 502 errors | Add `preStop: sleep 15` to allow LB to drain |
| No PDB defined | Node drain evicts all Pods simultaneously → complete outage | Always define PDB with `minAvailable: 1` |
| `topologySpreadConstraints` missing | Both Pods land on the same node → node failure = full outage | Spread Pods across nodes and/or availability zones |
| Secrets in plain YAML committed to Git | Credentials exposed in version history | Use Sealed Secrets, External Secrets Operator, or SOPS |
| No NetworkPolicy | Any compromised Pod in the cluster can call your app or DB directly | Default-deny all, explicitly allow only required paths |
| `:latest` image tag in Deployment | Pods on different nodes may run different image versions silently | Pin to a versioned tag or digest |
