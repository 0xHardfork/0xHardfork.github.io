## 🧠 Concept Overview

Every Pod in a Kubernetes cluster gets its own IP address. That sounds great on paper until you realise Pods are **ephemeral** — they get created, killed, rescheduled, and replaced constantly. A Deployment with three replicas today might have three totally different Pod IPs tomorrow (or even five minutes from now, after a rolling update). If your frontend container is trying to talk to your backend container by hardcoding a Pod IP, you're going to have a very bad day.

This is the fundamental problem that **Kubernetes Services** solve. A Service is a stable abstraction — essentially a virtual "front door" — that provides a **persistent DNS name and IP address** to reach a set of backing Pods. The backing Pods can come and go, scale up and down, move across nodes — the Service doesn't care. It discovers them via **label selectors** and keeps an up-to-date registry of healthy endpoints.

### The Kubernetes Networking Model — First Principles

Before we talk about Service types, let's lay the groundwork. Kubernetes enforces a very specific networking model with these non-negotiable rules:

1. **Every Pod gets its own unique IP address** — no NAT between Pods on the same cluster.
2. **All Pods can communicate with all other Pods** without NAT (by default, with no NetworkPolicies applied).
3. **All Nodes can communicate with all Pods** (and vice versa) without NAT.
4. **The IP that a Pod sees itself as is the same IP that others see it as** — no address translation tricks.

These rules mean Kubernetes creates a **flat network** where everything can talk to everything else by default. From a security perspective, this is terrifying. Imagine a traditional corporate network where every single server could reach every other server on any port with zero firewall rules. That's essentially what a default Kubernetes cluster looks like internally.

This is why understanding Services and networking is a prerequisite for the network security topics we'll cover in Phase 4 (NetworkPolicies, CNI plugins, etc.). You need to know what the default "wide-open" state looks like before you can lock it down.

### How the Network Actually Works Under the Hood

The networking magic is implemented by a **Container Network Interface (CNI)** plugin. When you set up a cluster (whether it's `kubeadm`, `minikube`, `kind`, or a managed service like EKS/GKE/AKS), you choose (or get assigned) a CNI plugin. Common ones include:

- **Flannel** — Simple overlay network. Good for learning, not feature-rich.
- **Calico** — Supports both overlay and direct routing. Adds NetworkPolicy enforcement.
- **Cilium** — eBPF-based, very powerful. We'll deep-dive into this in Phase 4.
- **Weave Net** — Another overlay network option.

The CNI plugin is responsible for assigning Pod IPs from a **Pod CIDR** range and setting up routes so that packets flow correctly between Pods across nodes. The key thing to understand is: the CNI plugin handles Pod-to-Pod communication. Services add a layer on top of that.

### kube-proxy — The Traffic Director

On every node in the cluster, a component called **kube-proxy** runs. Its job is to watch the Kubernetes API server for Service and Endpoint objects and then program the node's networking stack to route traffic to the correct Pod backends.

kube-proxy has three modes:

| Mode | Mechanism | Notes |
|------|-----------|-------|
| **iptables** (default) | Programs `iptables` rules on each node | Good performance, most common. Uses random selection for load balancing. |
| **IPVS** | Uses Linux IPVS (IP Virtual Server) kernel module | Better performance at scale, supports round-robin, least-connections, etc. |
| **userspace** (legacy) | Proxies traffic through a userspace process | Slow, essentially deprecated. |

When you create a Service, kube-proxy on every node sets up rules so that any traffic sent to the Service's virtual IP (the **ClusterIP**) gets redirected to one of the backing Pods. This redirection happens at the kernel level (in iptables/IPVS modes), so it's fast.

### Endpoints and EndpointSlices

When a Service is created with a selector, the API server automatically creates an associated **Endpoints** object (or **EndpointSlice** in newer clusters). This object contains the list of Pod IPs that match the Service's selector AND are passing their readiness probes.

```bash
# See the Endpoints for a service
kubectl get endpoints my-service

# In newer clusters, EndpointSlices provide the same info with better scalability
kubectl get endpointslices -l kubernetes.io/service-name=my-service
```

This is a critical concept: **only ready Pods appear in the Endpoints list**. If a Pod's readiness probe fails, kube-proxy removes it from the routing table. This is how Services provide basic health-aware load balancing.

### Service Types — The Four Flavours

Kubernetes has four Service types, each building on the one before it:

#### 1. ClusterIP (default)

This is the most common and most important Service type. It assigns an **internal-only virtual IP** (from the Service CIDR range) that's only reachable from within the cluster.

```
    ┌──────────────────────────────────────────┐
    │            Kubernetes Cluster             │
    │                                          │
    │   ┌─────────┐      ┌──────────────────┐ │
    │   │ Pod A    │─────▶│  Service (CIP)   │ │
    │   │ (client) │      │  10.96.0.100:80  │ │
    │   └─────────┘      └────────┬─────────┘ │
    │                       ┌─────┴─────┐      │
    │                       ▼           ▼      │
    │                 ┌─────────┐ ┌─────────┐  │
    │                 │ Pod B   │ │ Pod C   │  │
    │                 │ backend │ │ backend │  │
    │                 └─────────┘ └─────────┘  │
    │                                          │
    └──────────────────────────────────────────┘
    Outside world CANNOT reach 10.96.0.100.
```

**DNS resolution**: Kubernetes runs a DNS server (usually CoreDNS). When you create a Service named `my-service` in namespace `default`, it's automatically resolvable at:
- `my-service` (from the same namespace)
- `my-service.default` (from any namespace)
- `my-service.default.svc.cluster.local` (fully qualified)

This DNS-backed discovery is how microservices find each other inside the cluster without hardcoding IPs.

**Security perspective**: ClusterIP is excellent from a security standpoint because it restricts access to cluster-internal consumers. No external traffic can directly hit a ClusterIP Service. If your backend API doesn't need external exposure, **always use ClusterIP**.

#### 2. NodePort

A NodePort Service extends ClusterIP by additionally opening a **static port** (in the range 30000–32767) on **every node** in the cluster. Traffic hitting `<any-node-ip>:<nodePort>` gets forwarded to the Service, which then routes to a backing Pod.

```
    External Client
         │
         ▼
    ┌─────────────────────────────────────────────────┐
    │  Node A (192.168.1.10)   Node B (192.168.1.11)  │
    │  Port 30080 OPEN         Port 30080 OPEN        │
    │         │                       │                │
    │         └───────────┬───────────┘                │
    │                     ▼                            │
    │            Service (NodePort)                    │
    │            ClusterIP: 10.96.0.100                │
    │            NodePort: 30080                       │
    │                     │                            │
    │              ┌──────┴──────┐                     │
    │              ▼             ▼                     │
    │         ┌─────────┐  ┌─────────┐                │
    │         │ Pod B   │  │ Pod C   │                │
    │         └─────────┘  └─────────┘                │
    └─────────────────────────────────────────────────┘
```

**Security perspective**: NodePort is a significant escalation in exposure. It opens a port on **every** node, regardless of whether that node runs a backing Pod. An attacker scanning your node IPs will find these ports. In production, NodePort services are generally discouraged — prefer LoadBalancer or Ingress instead. If you must use NodePort, restrict access with external firewall rules (cloud security groups, etc.).

**Important detail — `externalTrafficPolicy`**: By default, kube-proxy can route NodePort traffic to a Pod on **any node** (not just the node receiving the traffic). This adds an extra hop and also obscures the client's real source IP (since the traffic gets SNAT'd). Setting `externalTrafficPolicy: Local` restricts routing to Pods on the **same node** that received the traffic, preserving the client's IP but potentially causing uneven load distribution.

#### 3. LoadBalancer

A LoadBalancer Service extends NodePort by provisioning an **external load balancer** from the cloud provider (AWS ELB/ALB, GCP Network LB, Azure LB, etc.). The cloud LB gets a public or internal IP and forwards traffic to the NodePorts on your nodes.

```
    Internet
       │
       ▼
    ┌───────────────┐
    │  Cloud Load   │
    │  Balancer     │
    │  (public IP)  │
    └───────┬───────┘
            │
    ┌───────┴──────────────────────────────────┐
    │  Node A            Node B                │
    │  NodePort 30080    NodePort 30080        │
    │       │                 │                │
    │       └───────┬─────────┘                │
    │               ▼                          │
    │         Service (LB)                     │
    │               │                          │
    │         ┌─────┴─────┐                    │
    │         ▼           ▼                    │
    │     ┌──────┐   ┌──────┐                  │
    │     │Pod B │   │Pod C │                  │
    │     └──────┘   └──────┘                  │
    └──────────────────────────────────────────┘
```

This is the most common way to expose a production service to the internet. The cloud provider's LB handles TLS termination, health checks, and distributes traffic across your nodes.

**Security perspective**: A LoadBalancer Service creates a **public internet endpoint** by default. This is the highest-exposure Service type. You should always:
- Ensure the backing Pods are hardened (Security Contexts, non-root, read-only filesystem, etc.)
- Use TLS (either at the LB or via the application)
- Apply NetworkPolicies to restrict which Pods can be accessed
- Consider using internal-only load balancers (`service.beta.kubernetes.io/aws-load-balancer-internal: "true"` on AWS) if external access isn't needed

**Cost note**: Each LoadBalancer Service typically provisions a separate cloud LB, which costs money. If you have 20 services, that's 20 load balancers. This is one reason **Ingress** (which we'll cover on Day 0x14) exists — it lets you share a single LB across many services.

#### 4. ExternalName

This is a special type that maps a Service to an external DNS name. It doesn't create any proxy rules or ClusterIP. It just configures CoreDNS to return a CNAME record.

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-external-db
  namespace: default
spec:
  type: ExternalName
  externalName: db.example.com
```

Pods in the cluster can now resolve `my-external-db.default.svc.cluster.local` and get back `db.example.com`. It's essentially a DNS alias.

**Security perspective**: ExternalName services can be an exfiltration vector if not monitored. An attacker who gains the ability to create Services could create an ExternalName pointing to their own server, enabling DNS-based data exfiltration. Monitor Service creation with audit logging (we'll cover this in Day 0x19).

### Headless Services — When You Don't Want a Virtual IP

Sometimes you don't want load balancing at all — you want to discover all the individual Pod IPs. This is common for **StatefulSets** (databases, message queues) where each Pod has a unique identity.

A headless Service is created by setting `clusterIP: None`:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-db
spec:
  clusterIP: None
  selector:
    app: postgres
  ports:
    - port: 5432
```

Instead of returning a single ClusterIP, a DNS lookup for `my-db` returns **all the Pod IPs** as A/AAAA records. The client gets the full list and decides which one to talk to.

For StatefulSets, each Pod also gets a unique DNS entry like `pod-0.my-db.default.svc.cluster.local`, which is incredibly useful for database replication setups where you need to address specific replicas.

### DNS in Kubernetes — CoreDNS Deep Dive

CoreDNS is the default cluster DNS server (it replaced kube-dns a while back). It runs as a Deployment in the `kube-system` namespace and is configured via a ConfigMap.

```bash
# Check CoreDNS is running
kubectl get pods -n kube-system -l k8s-app=kube-dns

# View the CoreDNS configuration
kubectl get configmap coredns -n kube-system -o yaml
```

A typical CoreDNS Corefile looks like:

```
.:53 {
    errors
    health {
        lameduck 5s
    }
    ready
    kubernetes cluster.local in-addr.arpa ip6.arpa {
        pods insecure
        fallthrough in-addr.arpa ip6.arpa
        ttl 30
    }
    prometheus :9153
    forward . /etc/resolv.conf {
        max_concurrent 1000
    }
    cache 30
    loop
    reload
    loadbalance
}
```

The key plugin here is `kubernetes`, which tells CoreDNS to resolve cluster-internal names. The `pods insecure` setting means Pod DNS records are created without verification — this can be tightened to `pods verified` in security-conscious environments.

**The `forward` directive** is security-critical. It tells CoreDNS to forward non-cluster queries to the upstream DNS servers listed in `/etc/resolv.conf` on the node. This means **Pods can resolve external DNS names by default**. From a security perspective, you might want to restrict external DNS resolution for certain namespaces — but that requires network-level controls we'll cover in Phase 4.

**DNS Policy for Pods**: Each Pod has a `dnsPolicy` field that controls how DNS resolution works:

| Policy | Behavior |
|--------|----------|
| `ClusterFirst` (default) | Use CoreDNS for cluster domains, then forward external queries upstream |
| `Default` | Use the node's DNS settings directly (bypasses CoreDNS) |
| `None` | No auto-configuration — you must specify `dnsConfig` manually |
| `ClusterFirstWithHostNet` | Same as ClusterFirst, but for Pods using `hostNetwork: true` |

### Port Concepts — Untangling the Confusion

One of the most confusing things for Kubernetes beginners is the different "port" fields. Let's settle this once and for all:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-web
spec:
  type: NodePort
  selector:
    app: web
  ports:
    - port: 80          # ← Service port (what clients inside the cluster use)
      targetPort: 8080   # ← Container port (what the app actually listens on)
      nodePort: 30080    # ← Node port (what external clients use, NodePort/LB only)
      protocol: TCP
```

- **`port`**: The port on the Service's ClusterIP. In-cluster clients connect to `my-web:80`.
- **`targetPort`**: The port on the actual Pod/container. The app is listening on `8080`.
- **`nodePort`**: The port opened on every node. External clients connect to `<node-ip>:30080`.

Traffic flow: `external:30080` → `Service:80` → `Pod:8080`

You can also use **named ports**. If your container spec names a port:

```yaml
# In your Deployment's pod template
containers:
  - name: web
    ports:
      - name: http
        containerPort: 8080
```

Then your Service can reference it by name:

```yaml
# In your Service spec
ports:
  - port: 80
    targetPort: http  # references the named port
```

This is handy when the actual port number might change between environments — the name stays the same.

### Session Affinity — Sticky Sessions

By default, kube-proxy distributes traffic randomly (iptables mode) or according to configurable algorithms (IPVS mode). If you need a client to always hit the **same Pod** (e.g., for stateful web sessions), you can enable session affinity:

```yaml
apiVersion: v1
kind: Service
metadata:
  name: my-app
spec:
  selector:
    app: web
  sessionAffinity: ClientIP
  sessionAffinityConfig:
    clientIP:
      timeoutSeconds: 10800  # 3 hours
  ports:
    - port: 80
      targetPort: 8080
```

kube-proxy will then route all traffic from the same client IP to the same Pod for the configured timeout. This is rough and not as sophisticated as cookie-based affinity (which requires an Ingress controller), but it works at L4.

### The Security Implications — Why This All Matters

Now that we've covered the mechanics, let's connect the dots to security:

**1. Default-open networking is dangerous**
Every Pod can talk to every other Pod. Every Service is reachable from any namespace. A compromised Pod in a low-security namespace can reach your database Service in the production namespace. This is the #1 reason NetworkPolicies exist (Day 0x11-0x12).

**2. NodePort services expand the attack surface**
Every open NodePort is essentially a hole in your cluster's perimeter. Attackers scanning your infrastructure will enumerate these ports. Always audit what NodePort services exist:

```bash
kubectl get svc --all-namespaces -o wide | grep NodePort
```

**3. LoadBalancer services face the internet**
Any misconfigured LoadBalancer Service is directly internet-facing. A common mistake is creating a LoadBalancer Service for an admin panel or debug endpoint. Always audit:

```bash
kubectl get svc --all-namespaces -o wide | grep LoadBalancer
```

**4. DNS can be abused**
An attacker with Service creation privileges can create ExternalName services pointing to malicious hosts. DNS tunneling is also possible if egress isn't restricted. CoreDNS logs should be monitored.

**5. `externalTrafficPolicy` affects security logging**
With the default `Cluster` policy, you lose the client's real source IP. Setting `Local` preserves it but may cause availability issues if Pods aren't evenly distributed. This is a security vs. availability tradeoff that must be made consciously.

**6. Service selectors are trust boundaries**
If an attacker can modify a Service's selector (or create a new Service with the same name in a different namespace), they can redirect traffic to malicious Pods. RBAC must restrict who can create/modify Service objects.

---

## 🔑 Key Points

- **Services provide stable networking for ephemeral Pods** — They abstract away Pod IP changes via label selectors, DNS names, and virtual IPs.
- **ClusterIP is the secure default** — It's cluster-internal only. Always prefer it unless external access is truly required.
- **NodePort and LoadBalancer widen your attack surface** — Each exposes cluster workloads beyond the cluster boundary. Audit these regularly.
- **Kubernetes networking is flat and open by default** — All Pods can reach all other Pods and Services. This requires explicit NetworkPolicies to restrict.
- **CoreDNS is the backbone of service discovery** — Misconfigured DNS policies or unrestricted external resolution can be exploited for data exfiltration.

## 💻 Code & Configuration Examples

### Example 1 — Creating a ClusterIP Service

```yaml
# file: clusterip-service.yaml
# A basic internal-only Service for a backend API
apiVersion: v1
kind: Service
metadata:
  name: backend-api
  namespace: default
  labels:
    app: backend
    tier: api
spec:
  type: ClusterIP         # Default — cluster-internal only
  selector:
    app: backend           # Matches Pods with label app=backend
    tier: api
  ports:
    - name: http
      port: 80             # Service port (clients use this)
      targetPort: 8080     # Pod/container port (app listens on this)
      protocol: TCP
```

```bash
# Deploy the Service
kubectl apply -f clusterip-service.yaml

# Verify it was created with a ClusterIP
kubectl get svc backend-api
# NAME          TYPE        CLUSTER-IP     EXTERNAL-IP   PORT(S)   AGE
# backend-api   ClusterIP   10.96.45.123   <none>        80/TCP    5s

# Check the endpoints — which Pods are backing this Service?
kubectl get endpoints backend-api
# NAME          ENDPOINTS                         AGE
# backend-api   10.244.1.5:8080,10.244.2.8:8080   5s

# Test connectivity from another Pod
kubectl run test-client --rm -it --image=curlimages/curl -- \
  curl -s http://backend-api.default.svc.cluster.local/health
```

**What this demonstrates**: A ClusterIP Service that load-balances traffic across backend Pods. The Service is only reachable from within the cluster, making it invisible to external attackers. Using meaningful labels (`app`, `tier`) ensures the selector precisely targets the right Pods.

### Example 2 — NodePort Service with externalTrafficPolicy

```yaml
# file: nodeport-service.yaml
# A NodePort Service that preserves client source IPs
apiVersion: v1
kind: Service
metadata:
  name: web-frontend
  namespace: default
  labels:
    app: web
spec:
  type: NodePort
  externalTrafficPolicy: Local   # Preserve client source IP
  selector:
    app: web
  ports:
    - name: http
      port: 80
      targetPort: 3000
      nodePort: 30080            # Fixed node port (range: 30000-32767)
      protocol: TCP
```

```bash
# Deploy
kubectl apply -f nodeport-service.yaml

# Verify — note the NodePort in the PORT(S) column
kubectl get svc web-frontend
# NAME           TYPE       CLUSTER-IP    EXTERNAL-IP   PORT(S)        AGE
# web-frontend   NodePort   10.96.0.201   <none>        80:30080/TCP   3s

# Access from outside the cluster (replace with your node's IP)
curl http://<node-ip>:30080

# Find your node IPs
kubectl get nodes -o wide
```

**What this demonstrates**: A NodePort service with `externalTrafficPolicy: Local` set, which preserves the original client's source IP in the packet headers. This is critical for security logging and IP-based access controls, but note that traffic will only route to Pods running on the node that receives the request. If no matching Pod exists on that node, the connection will be refused.

### Example 3 — LoadBalancer Service with Internal-Only Annotation

```yaml
# file: internal-lb-service.yaml
# An internal-only LoadBalancer (AWS example)
# This creates a cloud LB that is NOT internet-facing
apiVersion: v1
kind: Service
metadata:
  name: internal-api
  namespace: production
  annotations:
    # AWS-specific: create an internal NLB
    service.beta.kubernetes.io/aws-load-balancer-internal: "true"
    service.beta.kubernetes.io/aws-load-balancer-type: "nlb"
    # GCP equivalent:
    # networking.gke.io/load-balancer-type: "Internal"
    # Azure equivalent:
    # service.beta.kubernetes.io/azure-load-balancer-internal: "true"
spec:
  type: LoadBalancer
  selector:
    app: internal-api
  ports:
    - name: https
      port: 443
      targetPort: 8443
      protocol: TCP
```

```bash
# Deploy
kubectl apply -f internal-lb-service.yaml

# Check — EXTERNAL-IP will be a private VPC IP, not a public one
kubectl get svc internal-api -n production
# NAME           TYPE           CLUSTER-IP    EXTERNAL-IP    PORT(S)         AGE
# internal-api   LoadBalancer   10.96.1.50    10.0.5.123     443:31234/TCP   30s
```

**What this demonstrates**: A LoadBalancer Service with cloud-provider annotations that force an **internal** load balancer. The resulting LB gets a private IP within your VPC, not a public internet IP. This is a critical security pattern for services that need to be reachable from outside the cluster but only from within your private network (e.g., from other VPC resources or via VPN).

### Example 4 — Headless Service for StatefulSet Discovery

```yaml
# file: headless-service.yaml
# A headless Service for a PostgreSQL StatefulSet
apiVersion: v1
kind: Service
metadata:
  name: postgres-headless
  namespace: database
spec:
  clusterIP: None               # ← This makes it headless
  selector:
    app: postgres
  ports:
    - name: postgres
      port: 5432
      targetPort: 5432
---
# The corresponding StatefulSet (abbreviated)
apiVersion: apps/v1
kind: StatefulSet
metadata:
  name: postgres
  namespace: database
spec:
  serviceName: postgres-headless  # ← Links to our headless Service
  replicas: 3
  selector:
    matchLabels:
      app: postgres
  template:
    metadata:
      labels:
        app: postgres
    spec:
      containers:
        - name: postgres
          image: postgres:16-alpine
          ports:
            - containerPort: 5432
              name: postgres
```

```bash
# Deploy
kubectl apply -f headless-service.yaml

# DNS lookup returns all Pod IPs (no ClusterIP)
kubectl run dns-test --rm -it --image=busybox:1.36 -- \
  nslookup postgres-headless.database.svc.cluster.local

# Each StatefulSet Pod gets a unique DNS entry:
# postgres-0.postgres-headless.database.svc.cluster.local
# postgres-1.postgres-headless.database.svc.cluster.local
# postgres-2.postgres-headless.database.svc.cluster.local
```

**What this demonstrates**: A headless Service combined with a StatefulSet gives each Pod a stable, unique DNS identity. This is essential for database clusters where the primary replica needs to be addressed directly (e.g., for write operations) while read replicas are addressed individually for read-heavy workloads.

### Example 5 — Multi-Port Service

```yaml
# file: multi-port-service.yaml
# A Service exposing multiple ports (common for apps with metrics endpoints)
apiVersion: v1
kind: Service
metadata:
  name: my-app
  namespace: default
spec:
  type: ClusterIP
  selector:
    app: my-app
  ports:
    - name: http           # Name is REQUIRED when multiple ports are defined
      port: 80
      targetPort: 8080
      protocol: TCP
    - name: metrics        # Prometheus scrape endpoint
      port: 9090
      targetPort: 9090
      protocol: TCP
    - name: grpc           # gRPC endpoint
      port: 9000
      targetPort: 9000
      protocol: TCP
```

**What this demonstrates**: A single Service exposing multiple ports. This is common for applications that expose both a main endpoint and a metrics/health endpoint. From a security perspective, be cautious — exposing a metrics port (like Prometheus metrics) via a NodePort or LoadBalancer could leak sensitive operational data. Keep metrics endpoints on ClusterIP-only Services.

### Example 6 — Auditing All Services in the Cluster

```bash
#!/bin/bash
# audit-services.sh
# Quick script to audit all services and flag security concerns

echo "=== NodePort Services (potential external exposure) ==="
kubectl get svc --all-namespaces -o json | \
  jq -r '.items[] | select(.spec.type == "NodePort") |
  "\(.metadata.namespace)/\(.metadata.name) — Ports: \([.spec.ports[] | "\(.nodePort)→\(.targetPort)"] | join(", "))"'

echo ""
echo "=== LoadBalancer Services (internet-facing) ==="
kubectl get svc --all-namespaces -o json | \
  jq -r '.items[] | select(.spec.type == "LoadBalancer") |
  "\(.metadata.namespace)/\(.metadata.name) — External IP: \(.status.loadBalancer.ingress[0].ip // .status.loadBalancer.ingress[0].hostname // "pending")"'

echo ""
echo "=== ExternalName Services (DNS aliasing — potential exfiltration) ==="
kubectl get svc --all-namespaces -o json | \
  jq -r '.items[] | select(.spec.type == "ExternalName") |
  "\(.metadata.namespace)/\(.metadata.name) → \(.spec.externalName)"'

echo ""
echo "=== Services without selectors (manual endpoint management) ==="
kubectl get svc --all-namespaces -o json | \
  jq -r '.items[] | select(.spec.selector == null and .spec.type != "ExternalName") |
  "\(.metadata.namespace)/\(.metadata.name) — Type: \(.spec.type)"'
```

**What this demonstrates**: A security audit script that categorises all Services in the cluster by risk level. NodePort and LoadBalancer services represent external exposure, ExternalName services could be used for DNS-based attacks, and services without selectors might have manually-configured endpoints pointing to arbitrary IPs.

## ⚠️ Common Misconfigurations & Risks

| Misconfiguration | Risk Assessment | Secure Approach |
|------------------|----------------|-----------------|
| Using `LoadBalancer` type for internal-only services | Creates an internet-facing cloud LB, exposing the service to the public internet. Attackers can scan and reach the service directly. | Use `ClusterIP` for internal services. If cloud LB access is needed but only from your VPC, use internal LB annotations (`aws-load-balancer-internal: "true"`). |
| Leaving `externalTrafficPolicy: Cluster` (default) on NodePort/LB services | Client source IPs are masked via SNAT. Security logs, rate-limiters, and IP-based ACLs see the node's IP instead of the client's real IP. | Set `externalTrafficPolicy: Local` when client IP preservation is important for security logging or WAF rules. Accept the tradeoff of potentially uneven Pod distribution. |
| Exposing Prometheus metrics port via NodePort or LoadBalancer | Leaks internal operational data (memory usage, request rates, error counts, Go runtime stats) to anyone who can reach the port. | Keep metrics ports on ClusterIP-only Services. Use a ServiceMonitor or Port Forward for Prometheus scraping. Never expose `/metrics` externally. |
| Not auditing ExternalName Services | An attacker with Service creation RBAC can create ExternalName services pointing to attacker-controlled DNS, enabling data exfiltration via DNS. | Restrict Service creation with RBAC. Monitor the `CREATE` verb on `services` resources via API Server audit logs. Alert on ExternalName Service creation. |
| Using default Service sans NetworkPolicy | Any Pod in any namespace can reach any Service in any other namespace. A compromised workload in a dev namespace can access production databases. | Implement default-deny NetworkPolicies per namespace and explicitly allow only necessary cross-namespace traffic (covered in Phase 4). |
| Hardcoding NodePort values without documenting them | Port collisions across teams, undocumented external entry points, and difficulty tracking what's exposed. Over time, nobody knows which NodePorts are intentional. | Maintain a registry of NodePort allocations. Better yet, avoid NodePort entirely — use an Ingress controller with a single LoadBalancer for a unified entry point. |

### Exercise 6 — Source IP Exploitation via SNAT (Hard Escalation)

2. Expose it via a NodePort Service with the default `externalTrafficPolicy: Cluster`.

## 🗒️ Quick CLI Cheat Sheet

```bash
# ── Service Management ──
kubectl get svc                                            # List services in current namespace
kubectl get svc --all-namespaces                           # List ALL services across the cluster
kubectl get svc -o wide                                    # Show selectors and additional info
kubectl describe svc <name>                                # Detailed info including endpoints
kubectl get endpoints <name>                               # See which Pods back this service
kubectl get endpointslices -l kubernetes.io/service-name=<name>  # EndpointSlice (newer)

# ── Service Creation (imperative) ──
kubectl expose deployment <deploy> --port=80 --target-port=8080  # Quick ClusterIP Service
kubectl expose deployment <deploy> --type=NodePort --port=80     # Quick NodePort Service

kubectl run dns-debug --rm -it --image=busybox:1.36 -- nslookup <service>.<namespace>
kubectl get configmap coredns -n kube-system -o yaml       # View CoreDNS config
kubectl logs -n kube-system -l k8s-app=kube-dns            # CoreDNS logs

# ── Connectivity Testing ──
kubectl run curl-test --rm -it --image=curlimages/curl -- curl -s http://<service>:<port>
kubectl port-forward svc/<service> <local-port>:<svc-port> # Secure local access, no exposure

# ── Security Auditing ──
kubectl get svc -A -o json | jq '.items[] | select(.spec.type == "NodePort")'     # Find NodePorts
kubectl get svc -A -o json | jq '.items[] | select(.spec.type == "LoadBalancer")'  # Find LBs
kubectl get svc -A -o json | jq '.items[] | select(.spec.type == "ExternalName")'  # Find ExternalNames

# ── kube-proxy ──
kubectl get pods -n kube-system -l k8s-app=kube-proxy      # Check kube-proxy pods
kubectl logs -n kube-system -l k8s-app=kube-proxy           # kube-proxy logs
```
