## 🏗️ Full Traffic Flow Architecture

Before diving into each component, let's visualise the complete journey of a single HTTP request from the outside world all the way down to your Spring Boot container:

```
External Browser
       │
       │  curl http://api.127.0.0.1.nip.io:32165/hello
       ▼
┌──────────────────────────────────────────────────────────────────┐
│  Node (Bare Metal / VM)                                         │
│  NodePort :32165 ──► Ingress Controller (Nginx)                 │
│                      ┌─────────────────────────────────────┐    │
│                      │  1. Match Host: api.127.0.0.1.nip.io│    │
│                      │  2. Match Path: / (Prefix)          │    │
│                      │  3. Lookup Endpoints of web-app-svc │    │
│                      │  4. Direct proxy to Pod IP:8080     │    │
│                      └─────────────┬───────────────────────┘    │
│                                    │                             │
│                    ┌───────────────┼───────────────┐             │
│                    ▼                               ▼             │
│              ┌───────────┐                  ┌───────────┐        │
│              │  Pod A    │                  │  Pod B    │        │
│              │ 10.1.1.2  │                  │ 10.1.1.3  │        │
│              │  :8080    │                  │  :8080    │        │
│              │  web.jar  │                  │  web.jar  │        │
│              └───────────┘                  └───────────┘        │
│                                                                  │
│              Namespace: web-prod                                 │
└──────────────────────────────────────────────────────────────────┘
```

> **Key insight:** Nginx Ingress does **not** route traffic through the Service (ClusterIP). It reads the Service's Endpoints list and proxies directly to Pod IPs. The Service acts purely as a **service-discovery mechanism** in this flow.

## 1. Ingress — The Layer 7 Front Door

### 1.1 What Is Ingress?

An **Ingress** resource is a set of HTTP/HTTPS routing rules that tells a reverse proxy (the Ingress Controller) how to dispatch external traffic to internal Services. Think of it as the **reception desk** at the entrance of a corporate building.

Without Ingress, your Services are of type `ClusterIP` — reachable only inside the cluster. Ingress is the component that exposes them to the outside world through domain names, URL paths, and TLS certificates.

### 1.2 Anatomy of an Ingress Resource

```yaml
apiVersion: networking.k8s.io/v1
kind: Ingress
metadata:
  name: web-app-ingress
  namespace: web-prod
  annotations:
    nginx.ingress.kubernetes.io/proxy-read-timeout: "60"
    nginx.ingress.kubernetes.io/proxy-send-timeout: "60"
    nginx.ingress.kubernetes.io/proxy-body-size: "10m"
    nginx.ingress.kubernetes.io/load-balance: "least_conn"
    nginx.ingress.kubernetes.io/limit-rps: "50"
    nginx.ingress.kubernetes.io/limit-connections: "20"
    nginx.ingress.kubernetes.io/ssl-redirect: "false"
spec:
  ingressClassName: nginx

  tls:
    - hosts:
        - api.127.0.0.1.nip.io
      secretName: web-app-tls

  rules:
    - host: api.127.0.0.1.nip.io    # Step 1: Match the domain
      http:
        paths:
          - path: /                  # Step 2: Match the URL path (prefix)
            pathType: Prefix
            backend:
              service:
                name: web-app-svc   # Step 3: Look up this Service
                port:
                  name: http        # Step 4: Use this named port
```

The routing logic follows a strict chain:

| Step | What Happens | YAML Field |
|------|-------------|------------|
| 1 | Nginx checks the `Host` header of the incoming request | `rules[].host` |
| 2 | Nginx matches the URL path against the prefix | `rules[].http.paths[].path` |
| 3 | Nginx looks up the Service named `web-app-svc` | `backend.service.name` |
| 4 | Nginx reads the **Endpoints** behind that Service | `backend.service.port.name` |
| 5 | Nginx proxies **directly to a Pod IP** | *(internal — bypasses ClusterIP)* |

### 1.3 How Ingress Connects to Internal Services

Here is the critical truth that most beginners miss:

**Ingress does NOT send traffic to the Service's ClusterIP.** Instead, it uses the Service purely as a phone book (Endpoints lookup). When a request arrives, Nginx:

1. Queries the Kubernetes API for Endpoints associated with `web-app-svc`.
2. Discovers the live Pod IPs (e.g. `10.1.1.2:8080`, `10.1.1.3:8080`).
3. Writes an internal `upstream` block and proxies directly to those Pods.

This is why Nginx Ingress performs better than a naive Service-chaining approach — it eliminates one hop of network translation.

### 1.4 Load Balancing at the Ingress Layer

Since Nginx handles the actual proxying, **it controls the load-balancing algorithm** for external traffic. You can configure this via annotations:

| Annotation Value | Algorithm | Description |
|-----------------|-----------|-------------|
| `round_robin` (default) | Round-Robin | Requests are distributed evenly, one by one |
| `least_conn` | Least Connections | Sends to the Pod with the fewest active connections |
| `ewma` | EWMA | Exponentially Weighted Moving Average — sends to the fastest Pod |

```yaml
annotations:
  nginx.ingress.kubernetes.io/load-balance: "least_conn"
```

### 1.5 Automatic Failover

When a node suddenly dies:

1. **Millisecond-level (Nginx retry):** If Nginx sends a request to a dead Pod, it detects the failure instantly and **retries** the request against the next healthy Pod — all within the same client connection. The end user never sees an error.
2. **Second-level (Watch API push):** The Kubernetes API Server maintains a persistent Watch connection with the Ingress Controller. As soon as the Endpoints list changes (dead Pod removed), the API Server **pushes** the update. Nginx hot-reloads its upstream list in memory without restarting.

## 2. Service — The Layer 4 Load Balancer and Service Discovery

### 2.1 What Is a Service?

A **Service** provides a stable network identity for a dynamic set of Pods. Pods are ephemeral — they can crash, restart, or be rescheduled at any time, each time receiving a brand new IP address. Without a Service, any consumer would need to track every Pod IP change in real time.

A Service solves this by:
- **Label-based discovery:** Using a `selector` to dynamically track all Pods matching a given label (e.g. `app: web-app`).
- **Stable virtual IP:** Allocating a single ClusterIP that never changes, regardless of Pod churn.
- **Internal DNS registration:** Automatically creating a DNS record (e.g. `web-app-svc.web-prod.svc.cluster.local`) via CoreDNS.

### 2.2 Anatomy of a Service

```yaml
apiVersion: v1
kind: Service
metadata:
  name: web-app-svc
  namespace: web-prod
spec:
  type: ClusterIP          # Only reachable inside the cluster
  selector:
    app: web-app           # "Find all Pods with this label"
  ports:
    - name: http           # A symbolic name — referenced by Ingress
      port: 80             # The port the Service listens on
      targetPort: http     # Forward to the Pod's named port "http" (8080)
      protocol: TCP
```

### 2.3 The Dual Role of a Service

A Service plays **different roles** depending on who is calling it:

| Caller | Role of Service | How It Works |
|--------|----------------|--------------|
| **Ingress** (external traffic) | Pure service discovery (phone book) | Ingress reads Endpoints and proxies directly to Pod IPs, bypassing ClusterIP entirely |
| **Internal microservice** (e.g. `payment-pod`) | Real L4 load balancer | Traffic hits the ClusterIP → `kube-proxy` intercepts and distributes to Pod IPs |

### 2.4 Service and LVS: The Kernel-Level Connection

Under the hood, `kube-proxy` — the component on every node that materialises Service rules — operates in one of two modes:

| Mode | Technology | Lookup Performance | Load-Balancing Algorithms Available |
|------|-----------|-------------------|--------------------------------------|
| **iptables** (default) | Linux Netfilter rules | O(n) — linear chain walk | Random probability only |
| **IPVS** | Linux Virtual Server (LVS kernel module) | O(1) — hash table lookup | `rr`, `lc`, `dh`, `sh`, `sed`, `nq` |

For large-scale production clusters (thousands of Services), switching to IPVS mode is strongly recommended. IPVS is literally the same technology that powers LVS — one of the oldest and most battle-tested Layer 4 load balancers in the Linux kernel.

### 2.5 Session Affinity

By default, Services distribute traffic randomly (iptables) or round-robin (IPVS). If your application stores session state in memory, you can enable **sticky sessions**:

```yaml
spec:
  sessionAffinity: ClientIP
  sessionAffinityConfig:
    clientIP:
      timeoutSeconds: 3600   # Remember the binding for 1 hour
```

This ensures that the same client IP is always routed to the same Pod.

### 2.6 Why Can't Deployment Replace Service?

A common question: **"Deployment already manages Pods with IPs. Why do we need a Service?"**

The answer is that **Deployment does not have an IP.** Only Pods do. And Pod IPs are:

- **Ephemeral:** Crash → restart → new IP.
- **Unpredictable:** HPA scales from 2 to 10 Pods → 8 brand new IPs appear.
- **Invisible to consumers:** No one outside the cluster can discover them.

The Service is the **dynamic registry** that tracks these ever-changing IPs and exposes them through a single stable identity (ClusterIP + DNS name). Without it, every consumer would need to implement its own service-discovery mechanism.

### 2.7 Internal DNS: CoreDNS

When a Service is created, Kubernetes automatically registers a DNS record via **CoreDNS**:

```
web-app-svc.web-prod.svc.cluster.local
```

Any Pod inside the cluster can reach the Service using just its short name:

```java
// No need to hardcode IPs — DNS handles everything
String url = "http://web-app-svc:80/hello";
```

Even if the Service is deleted and recreated (receiving a new ClusterIP), the DNS name stays the same, and consumers never need to change their code.

## 3. HorizontalPodAutoscaler (HPA) — Automatic Scaling

### 3.1 What Is HPA?

The **HorizontalPodAutoscaler** automatically adjusts the number of Pod replicas based on observed resource metrics (CPU, memory, or custom metrics). It is the "HR Director" that hires (scales up) and fires (scales down) workers based on workload.

### 3.2 Configuration Breakdown

```yaml
apiVersion: autoscaling/v2
kind: HorizontalPodAutoscaler
metadata:
  name: web-app-hpa
  namespace: web-prod
spec:
  scaleTargetRef:
    apiVersion: apps/v1
    kind: Deployment
    name: web-app           # Takes over replica control from the Deployment

  minReplicas: 2            # Never go below 2 (high availability)
  maxReplicas: 10           # Never exceed 10 (cost control)

  metrics:
    - type: Resource
      resource:
        name: cpu
        target:
          type: Utilization
          averageUtilization: 70   # Scale up when average CPU > 70%

  behavior:
    scaleUp:
      stabilizationWindowSeconds: 60
      policies:
        - type: Pods
          value: 2
          periodSeconds: 60        # Add at most 2 Pods per minute
    scaleDown:
      stabilizationWindowSeconds: 300
      policies:
        - type: Pods
          value: 1
          periodSeconds: 120       # Remove at most 1 Pod every 2 minutes
```

### 3.3 The Stabilization Window: Preventing Flapping

Without the `stabilizationWindowSeconds`, HPA can enter a destructive oscillation loop:

```
Traffic spike → scale up → traffic dips → scale down → traffic returns → scale up again...
```

The **300-second scale-down window** forces HPA to observe for a full 5 minutes of sustained low usage before removing any Pod. This prevents premature downsizing and the associated cold-start penalties.

### 3.4 Prerequisite: metrics-server

HPA requires the `metrics-server` component to be installed in the cluster. Without it, `kubectl get hpa` will show `<unknown>` for CPU utilisation, and HPA will never trigger a scaling event.

```bash
kubectl apply -f https://github.com/kubernetes-sigs/metrics-server/releases/latest/download/components.yaml
```

## 4. PodDisruptionBudget (PDB) — Zero-Downtime Maintenance

### 4.1 What Is PDB?

A **PodDisruptionBudget** is a safety net that prevents **voluntary disruptions** (node drains, cluster upgrades, maintenance) from killing too many Pods simultaneously. It is the "Union Protection Law" for your workers.

### 4.2 What It Protects Against

PDB guards against **voluntary disruptions** — actions initiated by administrators:

| Voluntary (PDB protects) | Involuntary (PDB cannot protect) |
|--------------------------|----------------------------------|
| `kubectl drain node-1` | Node hardware failure |
| Cluster version upgrade | Kernel panic / power loss |
| Node pool resizing | OOM kill by the kernel |

### 4.3 How It Works

```yaml
apiVersion: policy/v1
kind: PodDisruptionBudget
metadata:
  name: web-app-pdb
  namespace: web-prod
spec:
  selector:
    matchLabels:
      app: web-app
  minAvailable: 1   # At least 1 Pod must remain running at ALL times
```

When an administrator runs `kubectl drain node-1`:

1. Kubernetes checks all PDBs that apply to Pods on that node.
2. If evicting a Pod would violate `minAvailable: 1` (e.g. only 1 Pod left and it's on this node), **the drain operation is blocked**.
3. The administrator must wait for the Deployment to schedule a replacement Pod on a different node first.
4. Only after the new Pod passes its readiness probe and is fully operational does the drain proceed.

This mechanism guarantees **zero-downtime** during planned maintenance events.

## 5. NetworkPolicy — Zero-Trust Network Security

### 5.1 The Default Problem

By default, Kubernetes networking is **completely flat and open**. Every Pod can communicate with every other Pod across all namespaces. This is a security nightmare — if an attacker compromises a single weak container, they can laterally move across the entire cluster.

**NetworkPolicy** implements a **Zero-Trust** posture: deny everything by default, then explicitly allow only the traffic you need.

### 5.2 The Three Iron Rules of NetworkPolicy

Understanding NetworkPolicy requires abandoning the mental model of traditional firewalls (top-to-bottom rule evaluation with priorities). Kubernetes NetworkPolicy follows a fundamentally different logic:

| Rule | Description |
|------|-------------|
| **Rule 1: No policy = full freedom** | If no NetworkPolicy selects a Pod, it can communicate freely with everything |
| **Rule 2: Once selected = total lockdown** | The moment *any* NetworkPolicy's `podSelector` matches a Pod, that Pod enters isolation. All traffic is denied by default |
| **Rule 3: Allow rules are additive (union)** | All matching NetworkPolicies' allow rules are merged. If *any* rule permits the traffic, it passes. There is **no deny rule, no ordering, no priority** |

### 5.3 Default Deny All

The foundation of any secure namespace begins with a blanket lockdown:

```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: default-deny-all
  namespace: web-prod
spec:
  podSelector: {}      # Matches ALL Pods in this namespace
  policyTypes:
    - Ingress
    - Egress
  # No ingress/egress rules = nothing is allowed
```

From this point on, every Pod in `web-prod` is completely isolated. No traffic in, no traffic out — not even DNS resolution.

### 5.4 Punching Holes: Allow Specific Traffic

After the lockdown, you selectively open channels:

**Allow Ingress Controller to reach web-app:**
```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-ingress-to-web-app
  namespace: web-prod
spec:
  podSelector:
    matchLabels:
      app: web-app
  policyTypes:
    - Ingress
  ingress:
    - from:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: ingress-nginx
          podSelector:
            matchLabels:
              app.kubernetes.io/name: ingress-nginx
      ports:
        - protocol: TCP
          port: 8080
```

**Allow web-app to reach the database:**
```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-web-app-to-db
  namespace: web-prod
spec:
  podSelector:
    matchLabels:
      app: web-app
  policyTypes:
    - Egress
  egress:
    - to:
        - ipBlock:
            cidr: 10.0.10.5/32    # Exact database IP
      ports:
        - protocol: TCP
          port: 5432              # PostgreSQL only
```

**Allow DNS resolution (essential):**
```yaml
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-dns
  namespace: web-prod
spec:
  podSelector:
    matchLabels:
      app: web-app
  policyTypes:
    - Egress
  egress:
    - to:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: kube-system
          podSelector:
            matchLabels:
              k8s-app: kube-dns
      ports:
        - protocol: UDP
          port: 53
        - protocol: TCP
          port: 53
```

### 5.5 Intra-Namespace Microservice Communication

When multiple services coexist within the same namespace under a `default-deny-all` policy, **they cannot communicate with each other by default.** You must create explicit bidirectional allow rules.

Example: Allow `app1` to call `app2` on port 8080:

```yaml
---
# Egress: Allow app1 to send traffic to app2
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-app1-to-app2
  namespace: web-prod
spec:
  podSelector:
    matchLabels:
      app: app1
  policyTypes:
    - Egress
  egress:
    - to:
        - podSelector:
            matchLabels:
              app: app2
      ports:
        - protocol: TCP
          port: 8080

---
# Ingress: Allow app2 to accept traffic from app1
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-app2-from-app1
  namespace: web-prod
spec:
  podSelector:
    matchLabels:
      app: app2
  policyTypes:
    - Ingress
  ingress:
    - from:
        - podSelector:
            matchLabels:
              app: app1
      ports:
        - protocol: TCP
          port: 8080
```

> **Both rules are required.** If only the Egress rule exists, `app1` can leave its own Pod but gets blocked at `app2`'s door. If only the Ingress rule exists, `app2` is willing to accept visitors but `app1` can never leave home.

### 5.6 Cross-Namespace Communication

To allow `web-app` in `web-prod` to reach `payment-api` in `pay-prod`, use `namespaceSelector`:

```yaml
---
# In web-prod: allow web-app to reach pay-prod/payment-api
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-web-to-pay
  namespace: web-prod
spec:
  podSelector:
    matchLabels:
      app: web-app
  policyTypes:
    - Egress
  egress:
    - to:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: pay-prod
          podSelector:
            matchLabels:
              app: payment-api
      ports:
        - protocol: TCP
          port: 8080

---
# In pay-prod: allow payment-api to accept traffic from web-prod/web-app
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-pay-from-web
  namespace: pay-prod
spec:
  podSelector:
    matchLabels:
      app: payment-api
  policyTypes:
    - Ingress
  ingress:
    - from:
        - namespaceSelector:
            matchLabels:
              kubernetes.io/metadata.name: web-prod
          podSelector:
            matchLabels:
              app: web-app
      ports:
        - protocol: TCP
          port: 8080
```

> **Critical YAML syntax note:** When `namespaceSelector` and `podSelector` are at the **same indentation level** (under a single `-`), they form an **AND** condition — both must match. If you accidentally add a second `-` before `podSelector`, they become an **OR** condition, which would be a devastating security hole.

### 5.7 Security Zoning with Labels

For organisations with dozens of namespaces, writing individual cross-namespace rules becomes unmanageable. The solution is **label-based security zoning**.

**Step 1:** Assign security-level labels to namespaces:
```bash
kubectl label ns frontend-prod security-level=L2
kubectl label ns api-prod      security-level=L3
kubectl label ns payment-prod  security-level=L4
kubectl label ns database-prod security-level=L5
```

**Step 2:** Write zone-based policies using `matchExpressions`:
```yaml
# Allow L3 services to accept traffic from L2 and L3 namespaces
apiVersion: networking.k8s.io/v1
kind: NetworkPolicy
metadata:
  name: allow-from-l2-l3
  namespace: api-prod
spec:
  podSelector: {}
  policyTypes:
    - Ingress
  ingress:
    - from:
        - namespaceSelector:
            matchExpressions:
              - key: security-level
                operator: In
                values: ["L2", "L3"]   # Accept from same level or above
```

This approach scales beautifully: any new namespace labelled `security-level: L2` automatically gains access — no policy modifications needed.

### 5.8 Enterprise-Grade: Calico / Cilium Tiered Policies

Native Kubernetes NetworkPolicy has one critical limitation: **there is no concept of priority or precedence.** Any developer who can create a NetworkPolicy can potentially open a hole that the security team didn't intend.

Enterprise CNI plugins like **Calico** and **Cilium** solve this with **tiered policies**:

| Tier | Owner | Purpose | Priority |
|------|-------|---------|----------|
| **SecOps** | Security team | Hard deny rules (e.g. L2 can never reach L5) | Highest — cannot be overridden |
| **Platform** | Infrastructure team | DNS, monitoring, logging access | Medium |
| **Application** | Developers | Fine-grained app-to-app rules | Lowest |

Traffic is evaluated from the highest tier downward. If the SecOps tier explicitly denies a connection, no lower-tier rule can override it. This is the **only** way to implement true hierarchical security governance in Kubernetes.

### 5.9 CNI Plugin Prerequisite

> **⚠️ Critical Warning:** NetworkPolicy resources can be created on any cluster, but they are **only enforced** if the cluster's CNI plugin supports them. Default CNI plugins like **Flannel** do **not** enforce NetworkPolicy. You must use a policy-aware CNI such as **Calico**, **Cilium**, or **Weave Net** for these rules to have any real effect.

### Component Responsibility Summary

| Component | Layer | Scope | Primary Responsibility |
|-----------|-------|-------|----------------------|
| **Ingress** | L7 (HTTP) | Cluster-wide | Domain routing, TLS, rate-limiting, external LB |
| **Service** | L4 (TCP/UDP) | Namespace | Service discovery, internal LB (via kube-proxy/LVS), DNS registration |
| **HPA** | Control plane | Per-Deployment | Auto-scaling based on resource metrics |
| **PDB** | Control plane | Per-Deployment | Guaranteeing minimum availability during maintenance |
| **NetworkPolicy** | L3/L4 (IP/Port) | Namespace | Zero-trust micro-segmentation firewall |
| **CoreDNS** | L7 (DNS) | Cluster-wide | Internal service name resolution |
| **CNI (Calico/Flannel)** | L2/L3 | Cluster-wide | Pod IP allocation, network fabric, policy enforcement |
| **kube-proxy** | L4 | Per-Node | Materialising Service rules as iptables/IPVS rules |

## 7. Key Takeaways

1. **Ingress and Service are complementary, not redundant.** Ingress handles L7 concerns (domains, paths, TLS). Service handles L4 concerns (stable IPs, internal DNS, endpoint tracking).

2. **Service has a dual personality.** For Ingress, it's a phone book. For internal microservices, it's a real L4 load balancer backed by iptables or IPVS (LVS).

3. **Pod IPs are ephemeral.** Never hardcode them. Service + CoreDNS provide the stable abstraction layer.

4. **HPA takes over replica count.** Once HPA is active, the `replicas` field in your Deployment is effectively ignored.

5. **PDB protects against careless admins**, not hardware failures. It ensures at least N Pods survive voluntary disruptions like node drains.

6. **NetworkPolicy has no ordering or priority.** Rules are purely additive. The only "deny" is the implicit isolation that occurs when a Pod is first selected by any policy.

7. **NetworkPolicy requires a capable CNI.** Without Calico, Cilium, or equivalent, your policies are silently ignored.

8. **Cross-namespace communication requires bilateral rules** — Egress from the source namespace AND Ingress in the destination namespace.
