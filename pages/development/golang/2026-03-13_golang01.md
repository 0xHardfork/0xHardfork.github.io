# Golang

## 🧠 Concept Overview

Today we tackle two of Go's most defining features: **Concurrency** and **Error Handling**. 

As an experienced engineer, you're likely familiar with threading (e.g., POSIX pthreads, Java Threads) or async/await models (e.g., Node.js, C#, Rust). Go takes a radically different approach: **Communicating Sequential Processes (CSP)**. 

### 1. Goroutines and the OS Thread M:N Scheduler
A `goroutine` is not an OS thread. It's a lightweight, user-space thread managed by the Go runtime scheduler. When a Go program starts, it spins up a pool of OS threads (typically one per CPU core, defined by `GOMAXPROCS`). The Go scheduler then multiplexes potentially millions of goroutines onto this small fixed number of OS threads. 
- **Stakes**: A Java thread might take 1MB+ of stack space. A goroutine starts at ~2KB and grows/shrinks dynamically.
- **Context Switching**: Switching OS threads involves crossing the kernel boundary (expensive). Switching goroutines is a fast, user-space pointer swap within the Go runtime.
- **Blocking**: If a goroutine blocks (e.g., waiting for I/O), the Go scheduler transparently detaches it from the OS thread and assigns a runnable goroutine to that thread. The OS thread itself never blocks.

### 2. Channels: "Do not communicate by sharing memory; instead, share memory by communicating."
Channels are typed conduits for passing data between goroutines. They provide synchronization natively.
- **Unbuffered Channels**: Synchronous. Both sender and receiver must be ready at the same time. The sender blocks until the receiver takes the data, and vice versa.
- **Buffered Channels**: Asynchronous up to the buffer size. The sender only blocks if the buffer is full.

### 3. Error Philosophy: Errors are Values
Go explicitly rejects exceptions (`try/catch`). Exceptions create invisible control flow paths and treat errors as catastrophic events. In Go, an `error` is just a standard interface type: `type error interface { Error() string }`.
Errors are treated as normal returned values that must be explicitly checked and handled. 
- **Wrapping & Unwrapping**: As of Go 1.13, you can wrap errors using `fmt.Errorf("...: %w", err)` and inspect error chains using `errors.Is(err, target)` (value equality) and `errors.As(err, &target)` (type assertion).
- **`defer`, `panic`, and `recover`**: Go *does* have a mechanism for catastrophic failures (`panic`). However, it shouldn't be used for standard control flow. `defer` schedules a function call to run immediately before the enclosing function returns (great for cleanup).

## 🔑 Key Points

- **Goroutines**: Started with the `go` keyword. Extremely cheap.
- **Channels**: `make(chan Type)` (unbuffered) or `make(chan Type, capacity)` (buffered). 
- **Select**: The `select` statement lets a goroutine wait on multiple channel operations simultaneously.
- **Closure Iteration Bug (Pre Go 1.22)**: Be careful capturing loop variables in goroutines!
- **Error Chaining**: Always use `%w` in `fmt.Errorf` when you want callers to be able to inspect the original underlying error.

## 💻 Code Examples

### Example 1 — The Worker Pool Pattern (Channels & Select)

This example demonstrates how to fan-out work to multiple concurrent workers and aggregate the results, preventing resource exhaustion.

```go
// File: example_worker_pool.go
package main

import (
	"fmt"
	"time"
)

// worker processes jobs and sends results back
func worker(id int, jobs <-chan int, results chan<- int) {
	// Ranges over the channel until it's closed
	for j := range jobs {
		fmt.Printf("Worker %d started job %d\n", id, j)
		time.Sleep(time.Millisecond * 500) // Simulate work
		fmt.Printf("Worker %d finished job %d\n", id, j)
		results <- j * 2 
	}
}

func main() {
	const numJobs = 5
	// Buffered channels prevent goroutines from blocking when submitting
	jobs := make(chan int, numJobs)
	results := make(chan int, numJobs)

	// Spin up 3 workers
	for w := 1; w <= 3; w++ {
		go worker(w, jobs, results)
	}

	// Send jobs
	for j := 1; j <= numJobs; j++ {
		jobs <- j
	}
	// Crucial: close the jobs channel so workers know to exit their range loop
	close(jobs)

	// Collect results
	for a := 1; a <= numJobs; a++ {
		res := <-results
		fmt.Printf("Result: %d\n", res)
	}
}
```

**What this demonstrates**: How buffered channels and range loops interact. Closing `jobs` is the idiomatic way to signal workers that no more data is coming, allowing them to cleanly exit.

### Example 2 — Advanced Error Handling (Wrapping & Inspection)

```go
// File: example_errors.go
package main

import (
	"errors"
	"fmt"
)

// Define a custom sentinel error
var ErrDatabaseConnection = errors.New("database connection lost")

// Custom error type to hold structured data
type QueryError struct {
	Query string
	Err   error
}

func (e *QueryError) Error() string {
	return fmt.Sprintf("query '%s' failed: %v", e.Query, e.Err)
}

func (e *QueryError) Unwrap() error { return e.Err }

func executeQuery(query string) error {
	// Simulate an upstream failure
	underlyingErr := ErrDatabaseConnection
	
	// Wrap the error with contextual data
	return &QueryError{
		Query: query,
		Err:   underlyingErr,
	}
}

func main() {
	err := executeQuery("SELECT * FROM users")
	if err != nil {
		fmt.Println("Raw Error:", err)

		// 1. errors.Is: Check if a specific sentinel error is anywhere in the chain
		if errors.Is(err, ErrDatabaseConnection) {
			fmt.Println("Action: Attempting database reconnect...")
		}

		// 2. errors.As: Extract a specific error type from the chain
		var qErr *QueryError
		if errors.As(err, &qErr) {
			fmt.Printf("Alert: Query %q failed!\n", qErr.Query)
		}
	}
}
```

**What this demonstrates**: The modern (Go 1.13+) way to handle errors. We don't just return strings; we construct an error chain that retains context (the SQL query) while still allowing the caller to definitively check the root cause (`ErrDatabaseConnection`).

## ⚠️ Common Mistakes

| Mistake | Why it's wrong | Correct approach |
|---------|---------------|-----------------|
| Capturing loop variables: `for i := 0; i < 3; i++ { go func() { print(i) }() }` | (Pre Go 1.22) All goroutines capture the *same* memory address of `i`. They will likely all print `3`. | Pass it as an argument: `go func(val int) { print(val) }(i)` |
| Forgetting to close channels | Receivers ranging over a channel will block forever if the channel is never closed, causing a memory leak (goroutine leak). | Sender is responsible for closing the channel when done. |
| `fmt.Errorf("failed: %v", err)` | Uses `%v` instead of `%w`. The original error is converted to a string, destroying the error chain. `errors.Is` will fail. | Always use `%w` to wrap: `fmt.Errorf("failed: %w", err)` |
