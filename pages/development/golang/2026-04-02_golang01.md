# Golang

## 🧠 Concept Overview

Interfaces are arguably the single most elegant feature in Go's type system — and they work completely differently from what you're used to in Java, C#, or C++. There's no `implements` keyword. No class hierarchies. No explicit declarations that a type satisfies an interface. If your type has the methods the interface requires, it satisfies it. Period. This is called **implicit interface satisfaction**, and it changes how you think about abstraction and polymorphism.

### What Is an Interface?

An interface in Go is a **set of method signatures**. It defines *behavior*, not data. Any concrete type (struct, custom type, etc.) that implements all the methods listed in the interface automatically satisfies it — no annotation required.

```go
type Speaker interface {
    Speak() string
}
```

That's it. Any type that has a `Speak() string` method is a `Speaker`. You don't write `type Dog implements Speaker` anywhere. The compiler figures it out structurally.

```go
type Dog struct {
    Name string
}

func (d Dog) Speak() string {
    return d.Name + " says: Woof!"
}

type Cat struct {
    Name string
}

func (c Cat) Speak() string {
    return c.Name + " says: Meow!"
}
```

Both `Dog` and `Cat` satisfy `Speaker` because they both have a `Speak() string` method. Now you can write a function that accepts *any* `Speaker`:

```go
func Greet(s Speaker) {
    fmt.Println(s.Speak())
}

func main() {
    Greet(Dog{Name: "Rex"})    // Rex says: Woof!
    Greet(Cat{Name: "Mimi"})   // Mimi says: Meow!
}
```

### Why Implicit Satisfaction Matters

In Java, you'd have to write `class Dog implements Speaker`. That creates a hard dependency between the implementing type and the interface definition. In Go, the interface and the type can live in completely separate packages that know nothing about each other. This is called **decoupling** and it's enormously powerful for building modular, testable systems.

Consider: You write a package `animals` with `Dog` and `Cat`. Someone else writes a package `speech` with the `Speaker` interface. Neither package imports the other. Yet `Dog` and `Cat` satisfy `Speaker`. This is duck typing — "if it walks like a duck and quacks like a duck, it's a duck" — but with full compile-time safety.

### Interface Values — What's Under the Hood

When you assign a concrete value to an interface variable, Go creates a two-word data structure internally:

1. **A pointer to the type descriptor** (what concrete type is stored)
2. **A pointer to the actual data** (the concrete value itself)

```go
var s Speaker
s = Dog{Name: "Rex"}
```

Under the hood, `s` now holds `(type=Dog, value=&Dog{Name:"Rex"})`. This is sometimes called an **interface boxing** — the concrete value is "boxed" into the interface wrapper. Understanding this is critical for performance-sensitive code and for understanding nil interface gotchas (which we'll get to).

### The Empty Interface: `interface{}` and `any`

The empty interface has zero method requirements:

```go
type anything interface{}
```

Since every type implements zero methods, **every type satisfies the empty interface**. This is Go's way of saying "any type." Starting in Go 1.18, there's an alias:

```go
var x any = 42
var y any = "hello"
var z any = []int{1, 2, 3}
```

`any` is literally defined as `type any = interface{}` in the `builtin` package. You'll see `interface{}` in older codebases and `any` in newer ones — they are identical.

The empty interface is used extensively in:
- `fmt.Println(a ...any)` — accepts anything
- JSON unmarshaling into `map[string]any`
- Generic containers (before Go 1.18 generics)

But here's the catch: once a value is an `any`, you lose type safety. You can't call `.Name` on an `any` without first asserting what concrete type it actually is. That's where type assertions come in.

### Type Assertions

A type assertion extracts the concrete value from an interface:

```go
var s Speaker = Dog{Name: "Rex"}

// Single-return form — panics if wrong type!
d := s.(Dog)
fmt.Println(d.Name) // "Rex"

// Two-return form — safe, no panic
d, ok := s.(Dog)
if ok {
    fmt.Println("It's a dog:", d.Name)
} else {
    fmt.Println("Not a dog")
}
```

**Always use the two-return form** in production code. The single-return form panics if the assertion fails, which will crash your program. The two-return form gives you a boolean `ok` flag instead.

You can also assert to another interface:

```go
type Runner interface {
    Run() string
}

// Does s also satisfy Runner?
if r, ok := s.(Runner); ok {
    fmt.Println(r.Run())
}
```

### Type Switches

When you need to branch on multiple possible types, use a type switch:

```go
func Describe(i interface{}) string {
    switch v := i.(type) {
    case int:
        return fmt.Sprintf("integer: %d", v)
    case string:
        return fmt.Sprintf("string: %q (len=%d)", v, len(v))
    case bool:
        return fmt.Sprintf("boolean: %t", v)
    case Dog:
        return fmt.Sprintf("dog named %s", v.Name)
    case nil:
        return "nil value"
    default:
        return fmt.Sprintf("unknown type: %T", v)
    }
}
```

Inside each `case`, `v` is already the correctly-typed value — no further assertion needed. The `%T` format verb prints the type name, which is handy for debugging.

### Interface Composition

Go interfaces compose beautifully through embedding:

```go
type Reader interface {
    Read(p []byte) (n int, err error)
}

type Writer interface {
    Write(p []byte) (n int, err error)
}

// ReadWriter is the combination of both
type ReadWriter interface {
    Reader
    Writer
}
```

This is exactly how the standard library's `io.ReadWriter` is defined. A type that has both `Read` and `Write` methods satisfies `ReadWriter`. This encourages small, focused interfaces — a key Go design principle.

The Go proverb says: **"The bigger the interface, the weaker the abstraction."** Interfaces with one or two methods are the sweet spot. The standard library is full of powerful single-method interfaces:

| Interface | Method | Package | Purpose |
|-----------|--------|---------|---------|
| `io.Reader` | `Read([]byte) (int, error)` | `io` | Anything you can read bytes from |
| `io.Writer` | `Write([]byte) (int, error)` | `io` | Anything you can write bytes to |
| `fmt.Stringer` | `String() string` | `fmt` | Custom string representation |
| `error` | `Error() string` | `builtin` | Error values |
| `sort.Interface` | `Len()`, `Less()`, `Swap()` | `sort` | Custom sorting |
| `io.Closer` | `Close() error` | `io` | Anything with a cleanup step |

### Pointer Receivers and Interface Satisfaction

This is a subtle but critical point. Consider:

```go
type Counter struct {
    count int
}

// Method with POINTER receiver
func (c *Counter) Increment() {
    c.count++
}

type Incrementer interface {
    Increment()
}
```

A `*Counter` satisfies `Incrementer`, but a `Counter` (non-pointer) does **not**:

```go
var i Incrementer

i = &Counter{}  // ✅ *Counter has Increment()
i = Counter{}   // ❌ Compile error! Counter does not have Increment()
```

Why? Because if you stored a copy of `Counter` in the interface, calling `Increment()` would modify the copy, not the original. Go prevents this silently incorrect behavior at compile time. The rule: **if any method has a pointer receiver, you must use a pointer to satisfy the interface**.

### The nil Interface Gotcha

This is the single most infamous trap in Go:

```go
var s Speaker       // nil interface
fmt.Println(s == nil) // true

var d *Dog          // nil pointer of type *Dog
s = d               // assign nil *Dog to Speaker
fmt.Println(s == nil) // FALSE! 😱
```

Wait, what? `s` holds a nil `*Dog`, but `s` itself is not nil? Yes. Remember the two-word interface structure: `(type, value)`. After `s = d`:
- type = `*Dog` (not nil — we know the concrete type)
- value = `nil` (the pointer itself is nil)

Since the type descriptor is set, the interface is not nil. An interface is only nil when **both** its type and value are nil. This bites everyone at least once. The defensive pattern:

```go
func DoSomething(s Speaker) {
    if s == nil {
        return // handles nil interface
    }
    // But a non-nil interface with nil underlying value can still panic!
    // Use reflect if you must check:
    // reflect.ValueOf(s).IsNil()
}
```

## Error Handling — Go's Controversial Superpower

Go doesn't have exceptions. No `try/catch`. No `throw`. Instead, errors are **values** — regular return values that you check explicitly. This is one of Go's most debated design decisions and one of its greatest strengths once you internalize it.

### The `error` Interface

The built-in `error` type is just an interface:

```go
type error interface {
    Error() string
}
```

That's it. Any type with an `Error() string` method is an error. The simplest way to create an error:

```go
import "errors"

err := errors.New("something went wrong")
fmt.Println(err.Error()) // "something went wrong"
fmt.Println(err)         // same — fmt.Println calls Error() automatically
```

Or with formatting:

```go
import "fmt"

name := "config.yaml"
err := fmt.Errorf("failed to open file: %s", name)
```

### The `if err != nil` Pattern

This is the bread and butter of Go error handling:

```go
result, err := SomeFunction()
if err != nil {
    // Handle the error: log it, return it, wrap it
    return fmt.Errorf("calling SomeFunction: %w", err)
}
// Happy path continues here
fmt.Println(result)
```

Yes, you will write `if err != nil` hundreds of times. It feels verbose at first, but it forces you to think about failure at every step. There's no hidden exception that silently bubbles up through 15 stack frames and crashes your program at 3 AM.

### Custom Error Types

Since `error` is an interface, you can create rich, domain-specific error types:

```go
type ValidationError struct {
    Field   string
    Message string
}

func (e *ValidationError) Error() string {
    return fmt.Sprintf("validation failed on %s: %s", e.Field, e.Message)
}

func ValidateAge(age int) error {
    if age < 0 || age > 150 {
        return &ValidationError{
            Field:   "age",
            Message: fmt.Sprintf("value %d is out of range [0, 150]", age),
        }
    }
    return nil
}
```

Returning `nil` means "no error" — the operation succeeded. Callers can type-assert to get the structured error:

```go
err := ValidateAge(-5)
if err != nil {
    var ve *ValidationError
    if errors.As(err, &ve) {
        fmt.Printf("Invalid field: %s\n", ve.Field)
    }
}
```

### Error Wrapping with `%w`

Go 1.13 introduced error wrapping. Use `%w` in `fmt.Errorf` to wrap an underlying error while adding context:

```go
func ReadConfig(path string) ([]byte, error) {
    data, err := os.ReadFile(path)
    if err != nil {
        return nil, fmt.Errorf("ReadConfig(%s): %w", path, err)
    }
    return data, nil
}

func LoadApp() error {
    data, err := ReadConfig("/etc/app/config.yaml")
    if err != nil {
        return fmt.Errorf("LoadApp: %w", err)
    }
    // use data...
    _ = data
    return nil
}
```

The error chain might look like: `LoadApp: ReadConfig(/etc/app/config.yaml): open /etc/app/config.yaml: no such file or directory`. Each layer adds context while preserving the original error.

### `errors.Is` and `errors.As` — Unwrapping Errors

When errors are wrapped, you can't compare them with `==` directly. Use `errors.Is` to check for a specific error value anywhere in the chain:

```go
_, err := os.Open("/nonexistent")
if errors.Is(err, os.ErrNotExist) {
    fmt.Println("File doesn't exist!")
}
```

Use `errors.As` to check for a specific error *type* in the chain:

```go
var pathErr *os.PathError
if errors.As(err, &pathErr) {
    fmt.Printf("Failed path: %s\n", pathErr.Path)
}
```

The difference: `errors.Is` checks for a specific **value** (like a sentinel error), while `errors.As` checks for a specific **type** and extracts the value.

### Sentinel Errors

Sentinel errors are pre-declared error values used as constants:

```go
var (
    ErrNotFound     = errors.New("resource not found")
    ErrUnauthorized = errors.New("unauthorized access")
    ErrTimeout      = errors.New("operation timed out")
)

func FindUser(id int) (*User, error) {
    // ...
    if user == nil {
        return nil, ErrNotFound
    }
    return user, nil
}

// Caller:
user, err := FindUser(42)
if errors.Is(err, ErrNotFound) {
    // handle missing user specifically
}
```

The standard library has many sentinel errors: `io.EOF`, `os.ErrNotExist`, `os.ErrPermission`, `sql.ErrNoRows`, etc. By convention, they start with `Err` (for values) or end with `Error` (for types).

### Multiple Error Returns — Go's Pattern

Go functions routinely return `(result, error)`:

```go
func Divide(a, b float64) (float64, error) {
    if b == 0 {
        return 0, errors.New("division by zero")
    }
    return a / b, nil
}
```

This extends to multiple results:

```go
func ParseCoordinate(s string) (lat, lon float64, err error) {
    // Named return values can be used for documentation
    parts := strings.Split(s, ",")
    if len(parts) != 2 {
        return 0, 0, fmt.Errorf("expected 'lat,lon', got %q", s)
    }
    lat, err = strconv.ParseFloat(strings.TrimSpace(parts[0]), 64)
    if err != nil {
        return 0, 0, fmt.Errorf("invalid latitude: %w", err)
    }
    lon, err = strconv.ParseFloat(strings.TrimSpace(parts[1]), 64)
    if err != nil {
        return 0, 0, fmt.Errorf("invalid longitude: %w", err)
    }
    return lat, lon, nil
}
```

### `panic` and `recover` — The Emergency Exits

`panic` immediately stops the current function, unwinds the stack, and crashes the program (unless caught by `recover`). It's for truly unrecoverable situations:

```go
func MustGetEnv(key string) string {
    val := os.Getenv(key)
    if val == "" {
        panic(fmt.Sprintf("required environment variable %s is not set", key))
    }
    return val
}
```

`recover` can catch a panic, but **only inside a deferred function**:

```go
func SafeExecute(fn func()) (err error) {
    defer func() {
        if r := recover(); r != nil {
            err = fmt.Errorf("panic recovered: %v", r)
        }
    }()
    fn()
    return nil
}
```

The `defer` keyword schedules a function to run when the enclosing function exits. Combined with `recover`, it's Go's version of `try/catch` — but it's intentionally awkward to discourage overuse.

**When to use panic**:
- During initialization if a critical dependency is missing (`MustGetEnv`, `regexp.MustCompile`)
- Programming errors that should never happen (index out of bounds in trusted internal code)
- In tests

**When NOT to use panic**:
- For expected errors (file not found, network timeout, invalid input)
- In library code that other packages will import (return errors instead)
- As a flow control mechanism (this isn't Python/Java exceptions)

### `defer` in Detail

`defer` is essential for cleanup — closing files, releasing locks, flushing buffers:

```go
func ProcessFile(path string) error {
    f, err := os.Open(path)
    if err != nil {
        return err
    }
    defer f.Close() // Guaranteed to run when ProcessFile returns

    // ... read and process the file ...
    scanner := bufio.NewScanner(f)
    for scanner.Scan() {
        fmt.Println(scanner.Text())
    }
    return scanner.Err()
}
```

Key defer behaviors:
- **LIFO order** — multiple defers execute in **reverse** order (last in, first out)
- **Arguments evaluated immediately** — `defer fmt.Println(x)` captures x's value at the defer statement, not at execution time
- **Deferred functions run even on panic** — that's how `recover` works

```go
func DeferOrder() {
    defer fmt.Println("1st defer") // runs third
    defer fmt.Println("2nd defer") // runs second
    defer fmt.Println("3rd defer") // runs first
    fmt.Println("main body")
}
// Output:
// main body
// 3rd defer
// 2nd defer
// 1st defer
```

### The `errors.Join` Pattern (Go 1.20+)

When you need to collect multiple errors (e.g., validating multiple fields):

```go
func ValidateUser(u User) error {
    var errs []error

    if u.Name == "" {
        errs = append(errs, errors.New("name is required"))
    }
    if u.Email == "" {
        errs = append(errs, errors.New("email is required"))
    }
    if u.Age < 0 {
        errs = append(errs, errors.New("age cannot be negative"))
    }

    return errors.Join(errs...)
}
```

`errors.Join` returns nil if all errors are nil, a single error if there's exactly one, or a combined error that `errors.Is` and `errors.As` can unwrap individually.

## 🔑 Key Points

- **Interfaces are implicit** — no `implements` keyword; if a type has the methods, it satisfies the interface. This enables extreme decoupling.
- **Keep interfaces small** — one or two methods is ideal. The standard library's most powerful interfaces (`io.Reader`, `error`, `fmt.Stringer`) all have a single method.
- **Errors are values, not exceptions** — handle them explicitly with `if err != nil`. Use `%w` to wrap, `errors.Is`/`errors.As` to unwrap.
- **The nil interface trap** — an interface holding a nil pointer is NOT a nil interface. The `(type, value)` pair must both be nil.
- **`panic` is the emergency brake** — use it only for truly unrecoverable programmer errors, never for expected runtime failures.

## 💻 Code Examples

### Example 1 — Polymorphism with Interfaces

```go
// File: shapes.go
package main

import (
	"fmt"
	"math"
)

// Shape defines an interface for geometric shapes
type Shape interface {
	Area() float64
	Perimeter() float64
}

// Stringer composability — shapes should also describe themselves
type Describer interface {
	Shape
	Describe() string
}

type Circle struct {
	Radius float64
}

func (c Circle) Area() float64      { return math.Pi * c.Radius * c.Radius }
func (c Circle) Perimeter() float64 { return 2 * math.Pi * c.Radius }
func (c Circle) Describe() string {
	return fmt.Sprintf("Circle(r=%.2f)", c.Radius)
}

type Rectangle struct {
	Width, Height float64
}

func (r Rectangle) Area() float64      { return r.Width * r.Height }
func (r Rectangle) Perimeter() float64 { return 2 * (r.Width + r.Height) }
func (r Rectangle) Describe() string {
	return fmt.Sprintf("Rectangle(%.2f×%.2f)", r.Width, r.Height)
}

// TotalArea works with any collection of Shapes — no coupling to concrete types
func TotalArea(shapes []Shape) float64 {
	total := 0.0
	for _, s := range shapes {
		total += s.Area()
	}
	return total
}

// PrintDetails works with the composed Describer interface
func PrintDetails(d Describer) {
	fmt.Printf("%s → area=%.2f, perimeter=%.2f\n",
		d.Describe(), d.Area(), d.Perimeter())
}

func main() {
	shapes := []Shape{
		Circle{Radius: 5},
		Rectangle{Width: 3, Height: 7},
		Circle{Radius: 2.5},
	}

	fmt.Printf("Total area: %.2f\n", TotalArea(shapes))

	// Type assertion to access the richer Describer interface
	for _, s := range shapes {
		if d, ok := s.(Describer); ok {
			PrintDetails(d)
		}
	}
}
```

**What this demonstrates**: Implicit interface satisfaction with composed interfaces. `TotalArea` is completely decoupled from `Circle` and `Rectangle` — it works with any future type that satisfies `Shape`. The type assertion shows how to safely "upgrade" from a narrow interface to a wider one.

### Example 2 — Robust Error Handling Chain

```go
// File: errchain.go
package main

import (
	"encoding/json"
	"errors"
	"fmt"
	"os"
)

// Domain-specific error types
type ConfigError struct {
	File    string
	Field   string
	Message string
}

func (e *ConfigError) Error() string {
	return fmt.Sprintf("config error in %s: field %q — %s", e.File, e.Field, e.Message)
}

// Sentinel errors
var (
	ErrMissingField = errors.New("required field is missing")
	ErrInvalidValue = errors.New("field has an invalid value")
)

type AppConfig struct {
	Port     int    `json:"port"`
	LogLevel string `json:"log_level"`
}

func ReadConfig(path string) (*AppConfig, error) {
	data, err := os.ReadFile(path)
	if err != nil {
		// Wrap the OS error with context
		return nil, fmt.Errorf("ReadConfig: reading %s: %w", path, err)
	}

	var cfg AppConfig
	if err := json.Unmarshal(data, &cfg); err != nil {
		return nil, fmt.Errorf("ReadConfig: parsing JSON: %w", err)
	}

	// Validate
	if cfg.Port == 0 {
		return nil, fmt.Errorf("ReadConfig: %w: %w", ErrMissingField,
			&ConfigError{File: path, Field: "port", Message: "port must be > 0"})
	}
	if cfg.Port < 1 || cfg.Port > 65535 {
		return nil, fmt.Errorf("ReadConfig: %w: %w", ErrInvalidValue,
			&ConfigError{File: path, Field: "port", Message: fmt.Sprintf("port %d out of range", cfg.Port)})
	}

	return &cfg, nil
}

func main() {
	cfg, err := ReadConfig("app.json")
	if err != nil {
		// Check by sentinel value
		if errors.Is(err, os.ErrNotExist) {
			fmt.Println("Config file not found — using defaults")
			return
		}
		if errors.Is(err, ErrMissingField) {
			fmt.Println("A required field is missing!")
		}

		// Check by error type
		var cfgErr *ConfigError
		if errors.As(err, &cfgErr) {
			fmt.Printf("Bad config field: %s in file %s\n", cfgErr.Field, cfgErr.File)
		}

		// Always print the full chain for logging
		fmt.Printf("Full error: %v\n", err)
		os.Exit(1)
	}

	fmt.Printf("Config loaded — port=%d, log_level=%s\n", cfg.Port, cfg.LogLevel)
}
```

**What this demonstrates**: A complete error handling pipeline — wrapping errors with context using `%w`, creating custom error types with structured data, and using both `errors.Is` (for sentinels) and `errors.As` (for structured types) to handle errors at call sites.

### Example 3 — The Nil Interface Trap & defer/recover

```go
// File: traps.go
package main

import "fmt"

type Logger interface {
	Log(msg string)
}

type FileLogger struct {
	Path string
}

func (f *FileLogger) Log(msg string) {
	fmt.Printf("[%s] %s\n", f.Path, msg)
}

func main() {
	// === The nil interface trap ===
	var l Logger
	fmt.Printf("nil interface:  l == nil? %t\n", l == nil) // true

	var fl *FileLogger // nil pointer
	l = fl             // assign nil *FileLogger to Logger interface
	fmt.Printf("nil-ptr in iface: l == nil? %t\n", l == nil) // FALSE!

	// This would panic: l.Log("hello") — because fl is nil
	// Always guard:
	if l != nil {
		// Even this isn't fully safe! The underlying *FileLogger is nil.
		// Calling l.Log() would dereference nil if Log touches fields.
		fmt.Println("l is not nil, but underlying value might be!")
	}

	// === defer + recover ===
	fmt.Println("\nTesting panic recovery:")
	result, err := safeDiv(10, 0)
	if err != nil {
		fmt.Printf("Caught: %v\n", err)
	} else {
		fmt.Printf("Result: %.2f\n", result)
	}

	result, err = safeDiv(10, 3)
	fmt.Printf("Result: %.2f, err: %v\n", result, err)
}

func safeDiv(a, b int) (result float64, err error) {
	defer func() {
		if r := recover(); r != nil {
			err = fmt.Errorf("panic in safeDiv: %v", r)
		}
	}()

	if b == 0 {
		panic("division by zero!")
	}
	return float64(a) / float64(b), nil
}
```

**What this demonstrates**: The subtle nil interface trap where `l == nil` is `false` even though the underlying pointer is nil, plus a practical `defer`/`recover` pattern that converts panics into returned errors — useful for wrapping third-party code that might panic.

## ⚠️ Common Mistakes

| Mistake | Why it's wrong | Correct approach |
|---------|---------------|-----------------|
| Using `==` to compare wrapped errors | `err == os.ErrNotExist` fails when the error is wrapped with `%w` | Use `errors.Is(err, os.ErrNotExist)` to traverse the chain |
| Ignoring returned errors with `_` | `f, _ := os.Open(path)` — if Open fails, `f` is nil, and every subsequent call on `f` panics | Always check: `f, err := os.Open(path); if err != nil { ... }` |
| Returning a typed nil pointer as `error` | `return (*MyError)(nil)` creates a non-nil interface — callers' `err != nil` is unexpectedly true | Return the plain `nil`: `return nil` |
| Huge interfaces with 10+ methods | Impossible for other packages to implement; defeats the purpose of abstraction | Split into small, composable interfaces (1–3 methods each) |
| Using `panic` for expected errors | Callers can't handle it gracefully; the program crashes | Return errors and let callers decide how to handle them |
| Not wrapping errors with context | `return err` gives no clue about where the error originated in a deep call stack | `return fmt.Errorf("FunctionName: %w", err)` adds breadcrumbs |

### Exercise 1 — Implement `fmt.Stringer` (Easy)

**Goal**: Create a `Book` struct that satisfies the `fmt.Stringer` interface for pretty printing.

**Requirements**:
- Define a `Book` struct with `Title`, `Author`, and `Year` fields
- Implement the `String() string` method so `fmt.Println(book)` prints something like `"The Go Programming Language" by Donovan & Kernighan (2015)`
- Create a slice of 3 books and print them in a loop

**Starter code**:

```go
package main

import "fmt"

type Book struct {
    // TODO: define fields
}

// TODO: implement Stringer interface

func main() {
    books := []Book{
        // TODO: create 3 books
    }
    for _, b := range books {
        fmt.Println(b) // Should use your String() method
    }
```
