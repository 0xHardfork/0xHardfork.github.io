# Golang

## 🧠 Concept Overview

Concurrency is one of Go's defining superpowers, and it's baked into the language at the syntax level — not bolted on via a library. To understand Go's concurrency model deeply, you need to grasp two primitives that work in concert: **goroutines** and **channels**. Together they implement a philosophy popularized by Tony Hoare's 1978 paper on Communicating Sequential Processes (CSP): *"Do not communicate by sharing memory; instead, share memory by communicating."* That single sentence is Go's entire concurrency worldview. Let's unpack what it means with the rigor it deserves.

---

### Goroutines — Lightweight, Multiplexed Threads

In C++ or Java, creating a new thread typically allocates 1–8 MB of stack space by default. The OS scheduler tracks each thread. Context-switching between threads is a kernel-mode operation. All of that is expensive.

A **goroutine** is Go's answer to "what if threads were cheap?" A goroutine starts with just **2–8 KB** of stack space (the exact amount changes between Go versions — in Go 1.22 it's 2KB). Critically, that stack is **growable**: the Go runtime will automatically expand it as needed (up to a configurable limit, default 1 GB), then shrink it when the stack frames are no longer needed. You can easily run millions of goroutines in a single program.

The Go runtime has its own **M:N scheduler**, where M goroutines are multiplexed onto N OS threads. The scheduler is described internally as a "work-stealing scheduler" running in its own goroutine (called `g0`). When a goroutine blocks on I/O (e.g., a network read), the runtime parks that goroutine and puts another runnable goroutine on the same OS thread — all transparently. This is what makes Go so strong for writing high-throughput servers.

You launch a goroutine with the `go` keyword followed by a function call:

```go
go myFunction(arg1, arg2)
go func() { fmt.Println("anonymous goroutine") }()
```

That's it. No `Thread thread = new Thread(...)`. No `thread.start()`. No `pthread_create`. Just `go`.

**Critical detail**: The `main` function itself runs as a goroutine. When `main` returns, the entire program exits — **all other goroutines are killed immediately**, regardless of what they're doing. This is a very common rookie mistake: firing off goroutines and wondering why they never print anything. If `main` exits first, they're dead. You need to use `sync.WaitGroup` (covered in Day 5) or block on a channel to keep `main` alive.

---

### Channels — Typed Pipelines Between Goroutines

A **channel** is a typed conduit for communication between goroutines. Think of it like a Unix pipe, but strongly-typed and deeply integrated into the language.

```go
ch := make(chan int)      // unbuffered channel of ints
ch <- 42                   // send 42 into the channel
value := <-ch              // receive from the channel
```

The `chan` keyword defines the channel type. `make(chan T)` creates an **unbuffered** channel. Unbuffered channels have a capacity of zero — a send on an unbuffered channel **blocks** until a receiver is ready, and a receive **blocks** until a sender puts something in. This makes unbuffered channels a perfect synchronization point: both sender and receiver must be simultaneously ready. In CSP terms, this is a "rendezvous."

#### Buffered Channels

```go
ch := make(chan int, 5) // buffered channel, capacity 5
```

A buffered channel has an internal queue. Sends don't block until the buffer is full; receives don't block unless the buffer is empty. Think of it like a bounded queue. You use buffered channels when you want to decouple the sender and receiver, allowing the producer to run ahead without waiting for the consumer.

**When to use buffered vs. unbuffered?** Use unbuffered when you want explicit synchronization between goroutines (the handshake model). Use buffered when you want to smooth out bursts, allow async sends, or implement worker pool patterns.

#### Directional Channels

When you pass a channel to a function, you can restrict its direction for safety:

```go
func producer(ch chan<- int) { ch <- 42 }   // send-only
func consumer(ch <-chan int) { v := <-ch }   // receive-only
```

This is purely a compile-time restriction that documents your intent and prevents bugs. The underlying channel is still bidirectional; we're just narrowing the view.

#### Closing a Channel

```go
close(ch)
```

Closing a channel signals to receivers that no more values will be sent. After a channel is closed, receives return the zero value immediately (with a second `ok` boolean of `false`). You can check if a channel was closed:

```go
v, ok := <-ch
if !ok {
    fmt.Println("channel was closed")
}
```

**Critical rules about closing**:
1. Only the **sender** should close a channel, never the receiver.
2. **Never close a channel twice** — this causes a panic.
3. **Never send on a closed channel** — this also causes a panic.
4. Closing is about signaling *completion*, not about freeing memory (channels are GC'd).

#### Ranging Over a Channel

The idiomatic way to consume all values from a channel until it's closed:

```go
for v := range ch {
    fmt.Println(v)
}
// loop exits when ch is closed
```

### The `select` Statement — Non-Deterministic Multiplexing

`select` is one of the most powerful constructs in the language. It's like a `switch` statement, but for channel operations. It waits until one of its cases can proceed, then executes that case. If multiple cases are ready simultaneously, **it picks one at random** — this non-determinism is intentional and prevents starvation.

```go
select {
case msg := <-ch1:
    fmt.Println("received from ch1:", msg)
case msg := <-ch2:
    fmt.Println("received from ch2:", msg)
case ch3 <- "hello":
    fmt.Println("sent to ch3")
default:
    fmt.Println("no channel ready, doing something else")
}
```

The `default` case makes `select` non-blocking: if no channel is ready right now, execute default and move on. Without `default`, `select` blocks until at least one case is ready.

**Common patterns with `select`**:
- **Timeout**: Combine with `time.After(duration)` to avoid waiting forever.
- **Cancellation**: Combine with a `done` or `ctx.Done()` channel to stop a goroutine gracefully.
- **Fan-in**: Merge multiple channels into one using `select` in a loop.

---

### The Go Runtime Scheduler in More Detail (for the curious)

The scheduler has 3 key concepts: **G** (goroutine), **M** (machine/OS thread), and **P** (processor, a logical CPU slot). P holds a local run queue of goroutines. M is associated with a P and actually runs goroutines. When an M is blocked (e.g., waiting for a syscall), Go creates a new M to allow the blocked P to keep running goroutines — this is *hand-off*. Work-stealing lets P2 steal goroutines from P1's local queue when P2 runs out. The number of Ps is controlled by `GOMAXPROCS`, which defaults to the number of CPU cores.

---

### Common Concurrency Pitfalls

**Goroutine leak**: You spawn a goroutine but never provide a way for it to exit. It loops forever or blocks on a channel no one sends to. Over time, leaked goroutines accumulate and eat memory. Always ensure every goroutine you spawn has a clean exit path — usually via a `done` or `ctx.Done()` channel.

**Data race**: Two goroutines access the same memory without synchronization, and at least one access is a write. Go ships with a built-in race detector: `go run -race main.go`. Use it. Always. Data races are undefined behavior in Go's memory model.

**Deadlock**: Two goroutines each wait for the other to send/receive. The Go runtime can detect whole-program deadlocks (when all goroutines are blocked) and panics with `all goroutines are asleep - deadlock!`. But partial deadlocks involving only some goroutines are not detected and will silently hang.

**Closing a nil channel**: Panics immediately. Always initialize channels with `make`.

## 🔑 Key Points

- A goroutine starts with ~2KB stack, grows automatically — you can run millions concurrently
- `go f()` launches a goroutine; `main` exiting kills all goroutines
- **Unbuffered** channels synchronize sender + receiver (rendezvous); **buffered** channels decouple them
- Only the **sender** closes a channel; sending to or double-closing a closed channel panics
- `for v := range ch` loops until channel is closed — the clean consumer pattern
- `select` multiplexes channel ops, randomly picks a ready case, supports `default` for non-blocking
- Use `-race` flag to detect data races during development

## 💻 Code Examples

### Example 1 — Basic Goroutine & Unbuffered Channel Handshake

```go
// File: example1.go
package main

import "fmt"

// square sends the square of n into the channel, then returns
func square(n int, ch chan<- int) {
	ch <- n * n // blocks until main is ready to receive
}

func main() {
	ch := make(chan int) // unbuffered: zero capacity

	// Launch a goroutine to compute 7²
	go square(7, ch)

	// Receive the result — this unblocks the goroutine
	result := <-ch
	fmt.Printf("7² = %d\n", result) // 7² = 49
}
```

**What this demonstrates**: The goroutine blocks on `ch <- n*n` until `main` executes `result := <-ch`. This is a perfect rendezvous: sender and receiver meet at the channel boundary.

---

### Example 2 — Pipeline Pattern: Generator → Squarer → Printer

```go
// File: example2.go
package main

import "fmt"

// generate sends integers 1..n to the returned channel, then closes it
func generate(n int) <-chan int {
	ch := make(chan int)
	go func() {
		defer close(ch) // always close when done producing
		for i := 1; i <= n; i++ {
			ch <- i
		}
	}()
	return ch
}

// square reads from in, squares each value, sends to the returned channel
func square(in <-chan int) <-chan int {
	out := make(chan int)
	go func() {
		defer close(out)
		for v := range in { // exits when in is closed
			out <- v * v
		}
	}()
	return out
}

func main() {
	// Pipeline: generate(5) → square → print
	nums := generate(5)
	squares := square(nums)

	for v := range squares {
		fmt.Println(v) // 1, 4, 9, 16, 25
	}
}
```

**What this demonstrates**: The classic CSP pipeline. Each stage is a goroutine. Data flows through channels. The `defer close(ch)` pattern ensures downstream stages terminate cleanly via `range`.

### Example 3 — Buffered Channel as a Semaphore (Limit Concurrency)

```go
// File: example3.go
package main

import (
	"fmt"
	"time"
)

// worker simulates an HTTP call that takes some time
func worker(id int, sem chan struct{}, results chan<- string) {
	sem <- struct{}{} // acquire slot (blocks if full)
	defer func() { <-sem }() // release slot when done

	// Simulate work
	time.Sleep(10 * time.Millisecond)
	results <- fmt.Sprintf("worker %d done", id)
}

func main() {
	const maxConcurrent = 3
	const totalWorkers = 10

	sem := make(chan struct{}, maxConcurrent) // semaphore: max 3 concurrent
	results := make(chan string, totalWorkers)

	for i := 1; i <= totalWorkers; i++ {
		go worker(i, sem, results)
	}

	for i := 0; i < totalWorkers; i++ {
		fmt.Println(<-results)
	}
}
```

**What this demonstrates**: A buffered channel of `struct{}` (zero-size type) acts as a counting semaphore. At most 3 goroutines run at once. This pattern is essential for rate-limiting concurrent API calls or database queries.

### Example 4 — `select` with Timeout and Cancellation

```go
// File: example4.go
package main

import (
	"fmt"
	"time"
)

// slowOperation returns a channel that delivers a result after a delay
func slowOperation(delay time.Duration) <-chan string {
	ch := make(chan string, 1)
	go func() {
		time.Sleep(delay)
		ch <- "result ready"
	}()
	return ch
}

func main() {
	done := make(chan struct{})
	result := slowOperation(200 * time.Millisecond)

	// Simulate a cancellation after 100ms
	go func() {
		time.Sleep(100 * time.Millisecond)
		close(done) // broadcast cancellation to all listeners
	}()

	select {
	case r := <-result:
		fmt.Println("got:", r)
	case <-time.After(500 * time.Millisecond):
		fmt.Println("timeout: operation took too long")
	case <-done:
		fmt.Println("cancelled: done signal received") // this one fires
	}
}
```

**What this demonstrates**: `select` racing three conditions — a result, a timeout, and a cancellation. The `close(done)` broadcasts to all goroutines watching `done` simultaneously (closed channels are always immediately readable, delivering zero values).

### Example 5 — Fan-Out / Fan-In Pattern

```go
// File: example5.go
package main

import (
	"fmt"
	"sync"
)

// fanOut distributes jobs to n workers
func fanOut(jobs <-chan int, n int) []<-chan int {
	outs := make([]<-chan int, n)
	for i := 0; i < n; i++ {
		out := make(chan int)
		outs[i] = out
		go func(ch chan<- int) {
			for j := range jobs {
				ch <- j * j // square the job
			}
			close(ch)
		}(out)
	}
	return outs
}

// fanIn merges multiple channels into one
func fanIn(channels ...<-chan int) <-chan int {
	merged := make(chan int)
	var wg sync.WaitGroup
	for _, ch := range channels {
		wg.Add(1)
		go func(c <-chan int) {
			defer wg.Done()
			for v := range c {
				merged <- v
			}
		}(ch)
	}
	go func() {
		wg.Wait()
		close(merged) // close merged when all inputs are drained
	}()
	return merged
}

func main() {
	jobs := make(chan int, 10)
	for i := 1; i <= 10; i++ {
		jobs <- i
	}
	close(jobs)

	// Fan out to 3 workers, then fan in results
	workers := fanOut(jobs, 3)
	results := fanIn(workers...)

	for r := range results {
		fmt.Println(r)
	}
}
```

**What this demonstrates**: The fan-out/fan-in pattern — a standard approach for parallel workload distribution. Multiple goroutines consume from a shared job queue, and their outputs are merged into a single result channel.

## ⚠️ Common Mistakes

| Mistake | Why it's wrong | Correct approach |
|---------|----------------|-----------------|
| `main` exits before goroutines finish | All goroutines are killed immediately when `main` returns | Block `main` with `sync.WaitGroup`, a channel receive, or `select {}` |
| Sending on a closed channel | Causes a runtime panic | Only the sender closes; use `sync.Once` if multiple senders exist |
| Closing a channel from the receiver | Violates the ownership model; sender may still try to send | Only the sender (or a coordinator) closes |
| Using an unbuffered channel in a single goroutine | Sending blocks forever with no receiver → deadlock | Use a separate goroutine or use a buffered channel |
| Goroutine leak via infinite loop with no exit | Goroutine runs forever, leaking memory | Always pass a `done chan struct{}` or `ctx.Done()` so the goroutine can exit |
| Data race on a shared variable | Undefined behavior, silently corrupts data | Protect with `sync.Mutex` or communicate via channel instead |
| Ranging over an unclosed channel | `for range ch` blocks forever if channel is never closed | Always `close(ch)` after the last send, typically with `defer close(ch)` |

## 🗒️ Quick Cheat Sheet

```go
// Create channels
ch := make(chan int)        // unbuffered
ch := make(chan int, 5)     // buffered, cap 5

// Send / receive
ch <- value                 // send (blocks if unbuffered and no receiver)
value := <-ch               // receive (blocks if empty)
value, ok := <-ch           // receive with closed check

// Directional in function signatures
func producer(out chan<- int) {}   // send-only
func consumer(in <-chan int)  {}   // receive-only

// Close (sender only!)
close(ch)

// Consume until closed
for v := range ch { /* ... */ }

// Goroutine
go func() { /* ... */ }()

// Select
select {
case v := <-ch1:            // receive
case ch2 <- val:            // send
case <-time.After(1*time.Second): // timeout
default:                    // non-blocking fallback
}

// Semaphore pattern
sem := make(chan struct{}, N)
sem <- struct{}{}           // acquire
<-sem                       // release
```
