# Golang

## 🧠 Concept Overview

Go's concurrency model — goroutines and channels — is elegant, but it doesn't cover every synchronization scenario you'll hit in production code. Channels are great for communicating data between goroutines, but sometimes you just need plain old mutual exclusion to guard a shared counter, or a mechanism to say "wait until all these workers are done." That's where the `sync` package steps in. And when you need to propagate deadlines, cancellation signals, and request-scoped values across an entire call tree of goroutines, `context.Context` is the standard tool. Together, `sync` and `context` form the backbone of writing safe, well-behaved concurrent Go programs.

### Why `sync` Exists When We Already Have Channels

Channels are a higher-level abstraction. Under the hood, they use locks themselves. If all you need is to protect a critical section — say, incrementing a shared map or counter — spinning up a channel-based pipeline is overkill. The `sync` package gives you low-level, zero-allocation (in many cases) primitives that are closer to what you'd find in C++ (`std::mutex`) or Java (`synchronized`), but designed to play nicely with Go's runtime scheduler.

The key primitives are:

| Primitive | Purpose |
|-----------|---------|
| `sync.Mutex` | Mutual exclusion lock — only one goroutine holds it at a time |
| `sync.RWMutex` | Reader/writer lock — multiple concurrent readers OR one exclusive writer |
| `sync.WaitGroup` | A counter that blocks until it reaches zero — "wait for N goroutines" |
| `sync.Once` | Execute a function exactly once, regardless of how many goroutines call it |
| `sync.Map` | A concurrent-safe map optimized for specific access patterns |
| `sync.Pool` | A pool of temporary objects to reduce GC pressure |
| `sync.Cond` | A condition variable for goroutine signaling (rarely used in practice) |

Let's go through each in depth.

### `sync.Mutex` — The Basic Lock

A `sync.Mutex` has two methods: `Lock()` and `Unlock()`. When a goroutine calls `Lock()`, it blocks until no other goroutine holds the lock. Once it has the lock, it enters the critical section. It **must** call `Unlock()` when it's done — and `defer` is your best friend here.

```go
var mu sync.Mutex
var count int

mu.Lock()
count++       // critical section — only one goroutine at a time
mu.Unlock()
```

Why `defer mu.Unlock()` is idiomatic: if your critical section panics or has early returns, the deferred unlock guarantees you don't leave the mutex locked forever (which would deadlock every other goroutine trying to acquire it).

```go
mu.Lock()
defer mu.Unlock()
// ... complex logic with possible early returns ...
count++
```

**Internal behavior**: Go's mutex implementation uses a combination of spinning (for short waits on multicore systems) and OS-level semaphore sleeping (for longer waits). Starting from Go 1.9, the mutex has a "starvation mode" — if a goroutine has been waiting for the lock for more than 1ms, the mutex switches from a "normal" (barging) mode to a "starvation" mode where the lock is handed directly to the longest-waiting goroutine. This prevents tail-latency blowups.

**A mutex is NOT reentrant.** If the same goroutine calls `Lock()` twice without an `Unlock()` in between, it deadlocks itself. Go made this choice deliberately — reentrant locks (like Java's `ReentrantLock`) hide design problems. If you need reentrancy, restructure your code so the locking happens at a single layer.

**Zero value is ready to use.** You don't need `mu = new(sync.Mutex)` — just declare `var mu sync.Mutex` and start using it. This is a recurring theme in Go: zero values should be useful.

**Never copy a mutex.** A `sync.Mutex` must not be copied after first use. If you embed a mutex in a struct and pass that struct by value, you're copying the lock state — instant data-race bug. Always pass structs containing a mutex by pointer. The `go vet` tool catches this.

### `sync.RWMutex` — Multiple Readers, Single Writer

When your workload is read-heavy (many goroutines reading, few writing), a plain `Mutex` is too coarse — it serializes all access, even reads that don't conflict with each other. `sync.RWMutex` distinguishes between:

- `RLock()` / `RUnlock()` — acquire/release a read lock. Multiple goroutines can hold read locks simultaneously.
- `Lock()` / `Unlock()` — acquire/release a write lock. A write lock is exclusive — no other readers or writers can proceed.

```go
var rw sync.RWMutex
var cache map[string]string

// Reader
rw.RLock()
v := cache["key"]
rw.RUnlock()

// Writer
rw.Lock()
cache["key"] = "value"
rw.Unlock()
```

**When to use vs plain `Mutex`**: if your read:write ratio is at least 10:1, `RWMutex` can give you measurably better throughput. If reads and writes are roughly balanced, the overhead of the more complex lock isn't worth it — just use `Mutex`.

**Writer starvation caveat**: In Go's `RWMutex`, once a writer calls `Lock()`, new readers calling `RLock()` are blocked (they queue behind the writer). This prevents writer starvation. However, existing readers that already hold read locks will continue until they `RUnlock()`. So a long-running read operation can still delay a writer.

### `sync.WaitGroup` — Fan-Out Coordination

`sync.WaitGroup` is the most commonly used sync primitive after `Mutex`. It's a thread-safe counter with three operations:

- `Add(delta int)` — increment the counter by `delta`
- `Done()` — decrement the counter by 1 (equivalent to `Add(-1)`)
- `Wait()` — block until the counter reaches zero

The canonical pattern: launch N goroutines, call `wg.Add(N)` before launching, have each goroutine `defer wg.Done()`, and call `wg.Wait()` in the main goroutine to block until all work is finished.

```go
var wg sync.WaitGroup

for i := 0; i < 10; i++ {
    wg.Add(1)
    go func(id int) {
        defer wg.Done()
        fmt.Printf("Worker %d done\n", id)
    }(i)
}

wg.Wait() // blocks until all 10 goroutines call Done()
```

**Critical rule**: Call `wg.Add()` **before** launching the goroutine, not inside it. If you call `Add` inside the goroutine, there's a race between the main goroutine reaching `Wait()` and the new goroutine calling `Add()`. The counter might be zero when `Wait()` checks, causing it to return prematurely.

**Don't reuse across cycles naïvely.** You can reuse a `WaitGroup` after `Wait()` returns and the counter is zero, but be careful about overlapping cycles. If a new `Add()` happens while `Wait()` from the previous cycle is still in progress (because of scheduling delays), you get a panic. In practice, just create a fresh `WaitGroup` per batch.

### `sync.Once` — One-Time Initialization

`sync.Once` guarantees that a function is executed exactly once, no matter how many goroutines call it. This is the Go equivalent of `std::call_once` in C++ or the "initialization-on-demand holder" pattern in Java.

```go
var once sync.Once
var instance *Database

func GetDB() *Database {
    once.Do(func() {
        instance = connectToDatabase()
    })
    return instance
}
```

Even if 100 goroutines call `GetDB()` simultaneously, `connectToDatabase()` runs exactly once. The other 99 goroutines block inside `once.Do()` until the initializer completes, then they all see the initialized `instance`.

**Gotcha**: If the function passed to `Do` panics, `Once` considers it "done." Subsequent calls won't retry. If you need retry-on-failure semantics, you'll need a different pattern (e.g., `sync.Mutex` + a boolean flag).

**The `Once.Do(f)` function must not call `Do(f)` recursively** — that deadlocks, because the inner `Do` tries to acquire the same internal mutex that the outer `Do` already holds.

### `sync.Map` — The Concurrent Map (Use Sparingly)

Go's built-in `map` type is **not** safe for concurrent access. If two goroutines read and write the same map simultaneously without synchronization, you get a fatal runtime panic (not just corrupted data — the runtime actively detects and kills the program).

`sync.Map` is a concurrent-safe map with these methods:

```go
var m sync.Map

m.Store("key", "value")          // set
v, ok := m.Load("key")           // get
m.Delete("key")                  // remove
v, loaded := m.LoadOrStore("key", "default")  // get-or-set atomically
m.Range(func(k, v interface{}) bool {         // iterate
    fmt.Println(k, v)
    return true // continue iteration
})
```

**When to use it**: `sync.Map` is optimized for two specific patterns:
1. Keys are written once and then read many times (like a cache that's populated at startup).
2. Multiple goroutines read/write disjoint sets of keys.

For the general case (same keys being read and written frequently), a plain `map` + `sync.RWMutex` is usually faster, because `sync.Map` uses `interface{}` everywhere (boxing overhead) and has a more complex internal structure (read-only map + dirty map with promotion).

**No generics**: As of Go 1.21+, `sync.Map` still uses `any` (a.k.a. `interface{}`) for both keys and values. If you want type safety, wrap it in a generic struct or just use `map[K]V` + `sync.RWMutex`.

### `sync.Pool` — Temporary Object Reuse

`sync.Pool` is a cache for temporary objects that reduces GC pressure. You `Put` objects back when you're done with them, and `Get` retrieves a previously pooled object (or creates a new one via the `New` function).

```go
var bufPool = sync.Pool{
    New: func() interface{} {
        return new(bytes.Buffer)
    },
}

func process() {
    buf := bufPool.Get().(*bytes.Buffer)
    buf.Reset()
    defer bufPool.Put(buf)

    buf.WriteString("hello")
    // use buf...
}
```

**Key behavior**: The pool can be cleared at any GC cycle. You must never assume that an object you `Put` will still be there on the next `Get`. A pool is NOT a free list or a connection pool — it's a performance optimization for reducing allocations, nothing more.

**Real-world usage**: The standard library uses `sync.Pool` internally for `fmt.Fprintf` (to pool `pp` printer structs), `encoding/json` (to pool encoder buffers), and many other hot paths.

### `context.Context` — Cancellation, Deadlines & Request-Scoped Values

The `context` package was introduced in Go 1.7 and is now woven into virtually every Go API — HTTP handlers, database queries, gRPC calls, you name it. A `context.Context` carries:

1. **Cancellation signal** — "this request was cancelled; stop work"
2. **Deadline** — "this request must complete by time T"
3. **Values** — request-scoped key-value pairs (use sparingly)

The interface is minimal:

```go
type Context interface {
    Deadline() (deadline time.Time, ok bool)
    Done() <-chan struct{}
    Err() error
    Value(key interface{}) interface{}
}
```

- `Done()` returns a channel that's closed when the context is cancelled or the deadline expires.
- `Err()` returns `context.Canceled` or `context.DeadlineExceeded` after `Done()` is closed.
- `Value(key)` retrieves a value associated with the key.

#### Creating Contexts

You never implement `Context` yourself. You derive new contexts from existing ones:

```go
// The root — use at the top of main(), tests, or init
ctx := context.Background()

// A root for when you're unsure which context to use (the linter will nag you)
ctx := context.TODO()

// With cancellation
ctx, cancel := context.WithCancel(parentCtx)
defer cancel()  // ALWAYS defer cancel to avoid leaking goroutines

// With a deadline (absolute time)
ctx, cancel := context.WithDeadline(parentCtx, time.Now().Add(5*time.Second))
defer cancel()

// With a timeout (relative duration) — syntactic sugar over WithDeadline
ctx, cancel := context.WithTimeout(parentCtx, 5*time.Second)
defer cancel()

// With a value (request-scoped data)
ctx := context.WithValue(parentCtx, myKey, myValue)
```

**The context tree**: Contexts form a tree. Cancelling a parent context cancels all its children. This is how an HTTP server can cancel all downstream work when a client disconnects — the `http.Request.Context()` is cancelled, and every goroutine checking that context gets the signal.

#### Using `ctx.Done()` in a `select`

The idiomatic way to respect cancellation in a long-running goroutine:

```go
func worker(ctx context.Context, jobs <-chan Job) {
    for {
        select {
        case <-ctx.Done():
            fmt.Println("cancelled:", ctx.Err())
            return
        case job := <-jobs:
            process(job)
        }
    }
}
```

The `select` listens on both the cancellation channel and the work channel. When `ctx.Done()` fires, the goroutine exits cleanly.

#### Context Values — The Controversy

`context.WithValue` is designed for request-scoped data that transits process boundaries — things like trace IDs, authentication tokens, and request deadlines. It is **not** a replacement for function parameters. Abusing `context.Value` turns your code into an untyped, key-value bag of magic — you lose compile-time type checking and make code impossible to reason about.

**Best practice**: Define unexported key types to avoid collisions:

```go
type ctxKey struct{}

func WithRequestID(ctx context.Context, id string) context.Context {
    return context.WithValue(ctx, ctxKey{}, id)
}

func RequestID(ctx context.Context) string {
    id, _ := ctx.Value(ctxKey{}).(string)
    return id
}
```

Using an unexported struct type as the key makes it impossible for other packages to accidentally clash with your key.

#### `context.AfterFunc` (Go 1.21+)

Go 1.21 added `context.AfterFunc(ctx, f)`, which registers a function `f` to be called in its own goroutine after `ctx` is done. This is useful for cleanup:

```go
stop := context.AfterFunc(ctx, func() {
    conn.Close()
})
// If you no longer need the cleanup, call stop() to unregister
```

#### `context.WithoutCancel` (Go 1.21+)

Sometimes you want a context that inherits values from a parent but is NOT cancelled when the parent is cancelled — for example, logging or auditing that should complete even after the request is done:

```go
auditCtx := context.WithoutCancel(requestCtx)
go writeAuditLog(auditCtx, event)
```

### Combining `sync` and `context` — Real-World Patterns

In production, you'll rarely use just one primitive. A typical pattern:

1. Create a `context.WithTimeout` for the overall operation.
2. Launch goroutines with `sync.WaitGroup` for fan-out.
3. Protect shared results with `sync.Mutex`.
4. Each goroutine checks `ctx.Done()` in its `select` loop.

```go
func fetchAll(ctx context.Context, urls []string) map[string]string {
    ctx, cancel := context.WithTimeout(ctx, 10*time.Second)
    defer cancel()

    var mu sync.Mutex
    results := make(map[string]string)

    var wg sync.WaitGroup
    for _, url := range urls {
        wg.Add(1)
        go func(u string) {
            defer wg.Done()
            req, _ := http.NewRequestWithContext(ctx, "GET", u, nil)
            resp, err := http.DefaultClient.Do(req)
            if err != nil {
                return
            }
            defer resp.Body.Close()
            body, _ := io.ReadAll(resp.Body)

            mu.Lock()
            results[u] = string(body)
            mu.Unlock()
        }(url)
    }
    wg.Wait()
    return results
}
```

This function fans out HTTP requests, respects a 10-second timeout, safely writes to a shared map, and waits for all goroutines to finish.

### Data Races and the Race Detector

Go ships with a built-in race detector. Run your tests or program with `-race`:

```bash
go test -race ./...
go run -race main.go
```

The race detector instruments memory accesses at compile time and reports any unsynchronized concurrent access. It's not a static analysis tool — it detects races that actually happen during execution. So maximize your test coverage.

**Performance overhead**: The race detector slows programs by ~2-10x and increases memory usage by ~5-10x. Never ship a `-race` binary to production, but always run your CI tests with `-race`.

## 🔑 Key Points

- **`sync.Mutex`** is the workhorse lock — use it with `defer Unlock()` to protect critical sections. It is NOT reentrant, and must never be copied.
- **`sync.WaitGroup`** coordinates fan-out — always call `Add()` before launching the goroutine, and `defer wg.Done()` inside.
- **`sync.Once`** guarantees one-time initialization — safe for lazy singleton initialization across goroutines.
- **`context.Context`** propagates cancellation, deadlines, and values down a call tree — always `defer cancel()` and check `ctx.Done()` in long-running work.
- **Always run `go test -race`** — the race detector catches real concurrency bugs that are invisible to code review.

## 💻 Code Examples

### Example 1 — Thread-Safe Counter with `sync.Mutex`

```go
// File: example1.go
package main

import (
	"fmt"
	"sync"
)

type SafeCounter struct {
	mu    sync.Mutex
	count int
}

func (c *SafeCounter) Increment() {
	c.mu.Lock()
	defer c.mu.Unlock()
	c.count++
}

func (c *SafeCounter) Value() int {
	c.mu.Lock()
	defer c.mu.Unlock()
	return c.count
}

func main() {
	counter := &SafeCounter{}
	var wg sync.WaitGroup

	for i := 0; i < 1000; i++ {
		wg.Add(1)
		go func() {
			defer wg.Done()
			counter.Increment()
		}()
	}

	wg.Wait()
	fmt.Println("Final count:", counter.Value()) // Always 1000
}
```

**What this demonstrates**: Embedding a `sync.Mutex` inside a struct to create a self-contained, goroutine-safe data type. The mutex protects both reads and writes. Running this with `-race` produces zero warnings.

### Example 2 — Fan-Out with `sync.WaitGroup` and Rate-Bounded Workers

```go
// File: example2.go
package main

import (
	"fmt"
	"sync"
	"time"
)

func worker(id int, wg *sync.WaitGroup, sem chan struct{}) {
	defer wg.Done()
	sem <- struct{}{}        // acquire a slot (blocks if full)
	defer func() { <-sem }() // release the slot

	fmt.Printf("Worker %d starting\n", id)
	time.Sleep(100 * time.Millisecond) // simulate work
	fmt.Printf("Worker %d done\n", id)
}

func main() {
	const numJobs = 20
	const maxConcurrency = 5

	var wg sync.WaitGroup
	sem := make(chan struct{}, maxConcurrency) // semaphore

	for i := 0; i < numJobs; i++ {
		wg.Add(1)
		go worker(i, &wg, sem)
	}

	wg.Wait()
	fmt.Println("All workers finished")
}
```

**What this demonstrates**: Combining `sync.WaitGroup` with a buffered channel as a semaphore to limit concurrency to 5 workers at a time while launching 20 goroutines. This is a production-grade pattern for controlling resource usage (database connections, API rate limits, etc.).

### Example 3 — `sync.Once` for Lazy Initialization

```go
// File: example3.go
package main

import (
	"fmt"
	"sync"
)

type Config struct {
	DBHost string
	DBPort int
}

var (
	config     *Config
	configOnce sync.Once
)

func loadConfig() *Config {
	configOnce.Do(func() {
		fmt.Println("Loading config (this prints exactly once)...")
		config = &Config{
			DBHost: "localhost",
			DBPort: 5432,
		}
	})
	return config
}

func main() {
	var wg sync.WaitGroup
	for i := 0; i < 10; i++ {
		wg.Add(1)
		go func(id int) {
			defer wg.Done()
			cfg := loadConfig()
			fmt.Printf("Goroutine %d got config: %+v\n", id, cfg)
		}(i)
	}
	wg.Wait()
}
```

**What this demonstrates**: `sync.Once` ensures `loadConfig` only executes the initialization logic once, even when called concurrently by 10 goroutines. All goroutines see the same `*Config` pointer.

### Example 4 — `context.WithTimeout` for Deadline-Bounded Work

```go
// File: example4.go
package main

import (
	"context"
	"fmt"
	"time"
)

func slowOperation(ctx context.Context) (string, error) {
	select {
	case <-time.After(3 * time.Second): // simulate slow work
		return "result", nil
	case <-ctx.Done():
		return "", ctx.Err()
	}
}

func main() {
	// Give the operation only 1 second
	ctx, cancel := context.WithTimeout(context.Background(), 1*time.Second)
	defer cancel()

	result, err := slowOperation(ctx)
	if err != nil {
		fmt.Println("Operation failed:", err) // context deadline exceeded
		return
	}
	fmt.Println("Got result:", result)
}
```

**What this demonstrates**: A context with a 1-second timeout killing a 3-second operation. The `select` in `slowOperation` races the work timer against `ctx.Done()`. This is how every HTTP client, DB driver, and gRPC call in Go respects timeouts.

### Example 5 — `context.WithCancel` for Parent-Child Cancellation

```go
// File: example5.go
package main

import (
	"context"
	"fmt"
	"sync"
	"time"
)

func infiniteWorker(ctx context.Context, id int, wg *sync.WaitGroup) {
	defer wg.Done()
	for {
		select {
		case <-ctx.Done():
			fmt.Printf("Worker %d: shutting down (%v)\n", id, ctx.Err())
			return
		case <-time.After(200 * time.Millisecond):
			fmt.Printf("Worker %d: tick\n", id)
		}
	}
}

func main() {
	ctx, cancel := context.WithCancel(context.Background())
	var wg sync.WaitGroup

	for i := 1; i <= 3; i++ {
		wg.Add(1)
		go infiniteWorker(ctx, i, &wg)
	}

	time.Sleep(1 * time.Second)
	fmt.Println("Cancelling all workers...")
	cancel() // signal all goroutines to stop

	wg.Wait()
	fmt.Println("All workers stopped")
}
```

**What this demonstrates**: Three infinite-loop goroutines cleanly shut down when the parent context is cancelled. This is the standard graceful-shutdown pattern in Go servers.

### Example 6 — `sync.RWMutex` for Read-Heavy Cache

```go
// File: example6.go
package main

import (
	"fmt"
	"sync"
	"time"
)

type Cache struct {
	mu    sync.RWMutex
	store map[string]string
}

func NewCache() *Cache {
	return &Cache{store: make(map[string]string)}
}

func (c *Cache) Get(key string) (string, bool) {
	c.mu.RLock()
	defer c.mu.RUnlock()
	v, ok := c.store[key]
	return v, ok
}

func (c *Cache) Set(key, value string) {
	c.mu.Lock()
	defer c.mu.Unlock()
	c.store[key] = value
}

func main() {
	cache := NewCache()
	cache.Set("greeting", "hello")

	var wg sync.WaitGroup

	// 100 concurrent readers
	for i := 0; i < 100; i++ {
		wg.Add(1)
		go func(id int) {
			defer wg.Done()
			v, _ := cache.Get("greeting")
			_ = v // use the value
		}(i)
	}

	// 1 writer updating concurrently
	wg.Add(1)
	go func() {
		defer wg.Done()
		time.Sleep(10 * time.Millisecond)
		cache.Set("greeting", "こんにちは")
	}()

	wg.Wait()
	v, _ := cache.Get("greeting")
	fmt.Println("Final value:", v)
}
```

**What this demonstrates**: A `sync.RWMutex` allowing 100 readers to proceed concurrently while a single writer acquires exclusive access. This cache pattern is used everywhere — in-memory caches, config stores, DNS resolution caches.

## ⚠️ Common Mistakes

| Mistake | Why it's wrong | Correct approach |
|---------|---------------|-----------------|
| Forgetting `defer cancel()` after `context.WithTimeout` / `WithCancel` | Leaks a goroutine inside the context package that waits forever. Each leaked context is a goroutine + timer leak. | Always `defer cancel()` immediately after creating the cancellable context. |
| Calling `wg.Add(1)` inside the goroutine instead of before launching it | Race between `wg.Wait()` seeing counter==0 and the new goroutine calling `Add(1)`. Wait can return before all goroutines start. | Call `wg.Add(1)` in the parent goroutine, before `go func(...)`. |
| Copying a `sync.Mutex` or `sync.WaitGroup` by value | The internal state is copied, creating two independent locks/counters. Concurrent access is no longer synchronized. | Always pass `*sync.Mutex` / `*sync.WaitGroup` by pointer, or embed them in structs that are themselves passed by pointer. |
| Using `sync.Map` as a general-purpose concurrent map | `sync.Map` is slower than `map` + `sync.RWMutex` for most access patterns due to `interface{}` boxing and internal complexity. | Use `map[K]V` + `sync.RWMutex` unless your workload is write-once/read-many or involves disjoint key sets. |
| Storing business-logic parameters in `context.Value` | Context values are untyped and invisible to the type checker. Debugging becomes a nightmare. | Only store cross-cutting concerns (trace IDs, auth tokens). Pass business data as explicit function parameters. |
| Locking a `Mutex` and then calling a function that also tries to lock the same `Mutex` | Deadlock — Go's `Mutex` is not reentrant. The goroutine blocks forever waiting for itself. | Restructure code: extract the critical section, or use internal "locked" helper methods that assume the mutex is already held. |
