# Rust

## 🧠 Concept Overview

Ownership is Rust's most unique and defining feature. It enables Rust to make memory safety guarantees without needing a garbage collector. The core of ownership lies in three fundamental rules:

1. **Each value in Rust has an owner.**
2. **There can only be one owner at a time.**
3. **When the owner goes out of scope, the value will be dropped.**

When we pass a variable to a function or assign it to another variable, the default behavior for data stored on the heap (like `String` or `Vec`) is to *move* the ownership. The original variable becomes invalid to prevent double-free errors. 

To use a value without taking ownership, Rust provides **Borrowing**. We borrow values using references (`&` for shared borrowing, `&mut` for mutable borrowing).

The borrowing rules are just as strict:
1. **At any given time, you can have either one mutable reference or any number of immutable references.** (This prevents data races at compile time).
2. **References must always be valid.** (Rust's lifetime system ensures no dangling pointers).

Finally, a **Slice** is a special kind of reference that lets you reference a contiguous sequence of elements in a collection rather than the whole collection.

---

## 🔑 Key Points

- Copy types (types stored entirely on the stack like `i32`, `bool`, `f64`, arrays of copy types) are *copied* instead of *moved* during assignment or function calls.
- Moving ownership invalidates the previous binding; trying to use it afterward won't compile.
- `&T` allows you to view data but not change it. You can have multiple `&T` references concurrently.
- `&mut T` allows you to modify data. You can only have one `&mut T` to a specific piece of data in an active scope, and no other immutable references can exist simultaneously.
- Slices like `&str` or `&[T]` are lightweight references that carry a pointer to the start and the length of the slice.

## 💻 Code Examples

### Example 1 — Ownership and Moves

```rust
// File: main.rs
fn main() {
    let s1 = String::from("hello"); // s1 owns the String on the heap
    let s2 = s1; // Ownership MOVES from s1 to s2. s1 is no longer valid.

    // println!("{}, world!", s1); // ❌ ERROR: value borrowed here after move
    println!("{}, world!", s2); // ✅ SUCCESS: s2 is the valid owner

    let x = 5; // x is an i32, which implements the Copy trait
    let y = x; // x is COPIED to y. Both are valid.
    
    println!("x = {}, y = {}", x, y); // ✅ SUCCESS: both are valid
}
```

**What this demonstrates**: How heap-allocated types (`String`) move ownership to prevent double-frees, while stack-allocated `Copy` types (`i32`) duplicate their data.

### Example 2 — Immutable and Mutable Borrowing

```rust
// File: main.rs
fn calculate_length(s: &String) -> usize {
    s.len() // We have an immutable reference to s, so we can't modify it.
}

fn change(s: &mut String) {
    s.push_str(", world"); // We have a mutable reference, so we can modify the owner's data.
}

fn main() {
    let mut greeting = String::from("Hello");

    let len = calculate_length(&greeting); // Passing an immutable reference (&)
    println!("The length of '{}' is {}.", greeting, len);

    change(&mut greeting); // Passing a mutable reference (&mut)
    println!("Modified greeting: {}", greeting);
    
    // Demonstrating the Borrowing Rules:
    let r1 = &greeting; 
    let r2 = &greeting; 
    println!("{} and {}", r1, r2); // ✅ Multiple immutable references are fine!
    
    // let r3 = &mut greeting; 
    // println!("{}, {}, and {}", r1, r2, r3); // ❌ ERROR: Cannot borrow as mutable because it is also borrowed as immutable
    // (If we use r3 *after* r1 and r2 are no longer used, it is valid due to Non-Lexical Lifetimes).
}
```

**What this demonstrates**: How to borrow data immutably (`&T`) to read it, and mutably (`&mut T`) to modify it without taking ownership, adhering to Rust's aliasing rules.

### Example 3 — Slices

```rust
// File: main.rs
fn first_word(s: &String) -> &str {
    let bytes = s.as_bytes();

    for (i, &item) in bytes.iter().enumerate() {
        if item == b' ' {
            return &s[0..i]; // returning a slice of the string
        }
    }

    &s[..] // no spaces found, return the slice of the whole string
}

fn main() {
    let mut s = String::from("hello world");
    let word = first_word(&s); // word gets an immutable borrow of `s`

    // s.clear(); // ❌ ERROR: cannot borrow `s` as mutable because it is also borrowed as immutable
                  // (clearing the string would invalidate the `word` slice!)
                  
    println!("the first word is: {}", word); 
    // `word` goes out of scope here. After this, `s` could be modified.
}
```

**What this demonstrates**: How slices (`&str`) tie the lifetime of the returned view to the underlying data, preventing the compiler from allowing modifications to the original `String` while the slice is in use.

## ⚠️ Common Mistakes

| Mistake | Why it's wrong | Correct approach |
|---------|---------------|-----------------|
| Using a `String` after passing it to a function `fn foo(s: String)`. | The function takes ownership of `s`, so it's moved and invalidated in the caller. | Pass a reference `fn foo(s: &String)` or `fn foo(s: &str)`. |
| Creating two `&mut T` to the same variable. | Rust prevents data races by allowing only one active mutable reference to a value. | Structure your code so the scopes of mutable references do not overlap. |
| Returning a reference to a variable created inside the function. | The variable goes out of scope at the end of the function, leaving a dangling pointer. | Return the value directly (move ownership out), or pass in a mutable reference to modify. |

## 🗒️ Quick Cheat Sheet

```rust
let x = String::from("heap");       // x owns the data
let y = x;                          // y owns it, x is moved/invalid
let r1 = &y;                        // r1 borrows y immutably (read-only)
let mut z = String::from("mut");    // z is mutable owner
let r2 = &mut z;                    // r2 borrows z mutably (read-write)
```
