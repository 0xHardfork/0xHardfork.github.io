# Rust

## 🧠 Concept Overview

Rust is a systems programming language that runs blazingly fast, prevents segfaults, and guarantees thread safety—all without needing a garbage collector. Unlike dynamically typed languages (Python, JavaScript) or statically typed languages with heavy runtime garbage collectors (Java, Go, C#), Rust achieves safe memory management through an innovative system of *ownership* enforced entirely at compile time.

Today, we're building the foundation. We will tear down your preconceived notions about variables and control flow from other languages and map them to Rust's unyielding compiler.

### Variables and Immutability by Default

In almost every mainstream programming language (C, Java, Go, Python), a variable is mutable by default. You declare `x = 5`, and on the next line you can say `x = 6`.

Rust flips this paradigm on its head. In Rust, **variables are immutable by default**. This is not a suggestion; it's a hard compiler error if you try to mutate one without explicit permission.

```rust
let x = 5;
// x = 6; // THIS WILL NOT COMPILE! x is immutable.
```

Why? Immutability drastically reduces the surface area for bugs, especially in concurrent applications where multiple threads might be trying to read or write the same data. By making data immutable by default, the Rust compiler can mathematically prove that data races are impossible for those variables.

If you *need* a variable to change, you explicitly opt-in with the `mut` keyword.

```rust
let mut y = 10;
y = 11; // Perfectly fine!
```

**Shadowing:**
Even though `let x = 5` is immutable, Rust allows you to "shadow" a variable by using the `let` keyword again with the exact same name.

```rust
let x = 5;
let x = x + 1; // Creates a NEW variable named 'x', hiding the old one.
let x = "Now I am a string!"; // Shadowing allows changing the type entirely!
```
Shadowing is functionally different from mutability (`mut`). Shadowing creates a completely new allocation tied to a new variable name in the current scope, whereas mutation alters the data stored in the original memory address.

### The Type System: Scalars and Compounds

Rust is a statically, strongly typed language. Every value in Rust has a certain data type, preventing you from accidentally multiplying a boolean and a string like you might in Javascript.

The types are grouped into scalars (representing a single value) and compounds (representing multiple values).

**Scalars:**
- **Integers:** `i8`, `i16`, `i32` (default), `i64`, `i128`, and unsigned variants like `u32`. The `isize` and `usize` types depend on the architecture of the computer your program is running on (64-bit on a 64-bit architecture).
- **Floating-point numbers:** `f32` and `f64` (default).
- **Booleans:** `bool` (`true` or `false`).
- **Characters:** `char`. In Rust, a `char` is 4 bytes and represents a Unicode Scalar Value. Emojis are perfectly valid single characters! `let c = '🚀';`

**Compounds:**
- **Tuples:** A fixed-length, immutable grouping of potentially different types.
  `let tup: (i32, f64, u8) = (500, 6.4, 1);`
- **Arrays:** A fixed-length collection of values of the *exact same* type. Arrays in Rust are extremely strict. They are allocated on the stack (unlike dynamically growing vectors allocated on the heap, which we will see in Day 6).
  `let a = [1, 2, 3, 4, 5]; // Type inferred as [i32; 5]`

### Functions: Statements vs. Expressions

Functions in Rust look familiar but have a critical semantic difference from C or Java based on how they return values.

In Rust, the language is fundamentally built around expressions.
- **Statements** perform an action and do not return a value. (e.g., `let y = 6;` is a statement. Function definitions are statements).
- **Expressions** evaluate to a resulting value. (e.g., `5 + 6`, calling a function, a macro, or even a basic math operation).

Crucially, blocks enclosed in curly braces `{}` are expressions in Rust! The final line inside a block determines the value of the block *if the final line omits the trailing semicolon*.

```rust
fn add_five(x: i32) -> i32 {
    x + 5 // NO SEMICOLON! This evaluates to a value and implicitly returns it.
}
```
If you add a semicolon (`x + 5;`), you turn the expression into a statement, and the function will return the "unit type" `()`, which is Rust's version of `void`. The compiler will fiercely yell at you if you specify `-> i32` but return `()`.

### Control Flow: `if` Expressions and `loop`

Because `if` blocks are expressions, you can actually use them on the right side of a `let` statement to assign a result! (This is Rust's highly readable alternative to the ternary operator `? :` found in C/Java).

```rust
let condition = true;
let number = if condition { 5 } else { 6 };
// Both arms must return the same type, otherwise the compiler will panic.
```

**Looping Constructs:**
Rust provides `loop`, `while`, and `mutiple-times-for-iterators`.

- `loop`: An infinite loop unless you explicitly use the `break` keyword. You can even return a value out of a `loop` block!
- `while`: Standard conditional looping.
- `for`: By far the most common, safest, and fastest way to loop through collections.

## 🔑 Key Points

- Variables in Rust are **immutable by default**. Mutation requires the explicit `mut` keyword.
- **Shadowing** lets you reuse a variable name and even change its type, creating a fresh allocation in that scope.
- Rust strongly differentiates between **Statements** (actions without return values) and **Expressions** (blocks that evaluate to a value).
- Omitting the trailing semicolon at the end of a block/function is the idiomatic way to implicitly return that expression's value.
- `if` blocks evaluate as expressions and can be bound directly to variables, functioning as safe ternary operators.
- Rust uses strict stack-bound Arrays `[T; N]` for fixed-size lists and Tuples `(T1, T2)` for fixed-length heterogeneous groupings.

## 💻 Code Examples

### Example 1 — Immutability, Mutation, and Shadowing

```rust
// File: main.rs
fn main() {
    // 1. Immutability
    let config_port = 8080;
    // config_port = 8081; // Error if uncommented: cannot assign twice to immutable variable

    // 2. Opt-in Mutability
    let mut connection_count = 0;
    connection_count += 1; // Valid!

    // 3. Shadowing
    let secret = "hunter2";
    println!("The secret string is: {}", secret);
    
    // We shadow 'secret' to hold its length. We completely changed the type!
    let secret = secret.len(); 
    println!("The secret length is: {}", secret);

    // This is useful because it saves us from inventing names like `secret_str` and `secret_len`.
}
```

**What this demonstrates**: How Rust fundamentally approaches variable assignment, explicitly locking down data by default, and how the concept of Shadowing bypasses type-rigidity when you simply want to refine or transform data without renaming it.

### Example 2 — Functions, Expressions, and Loops

```rust
// File: main.rs

// Functions must declare parameter types and return types explicitly.
fn calculate_growth(base: f64, multiplier: f64) -> f64 {
    let result = base * multiplier;
    // Notice the lack of semicolon below! This is an implicit return.
    result + 1.5 
}

fn main() {
    let growth = calculate_growth(10.0, 1.2);
    
    // Using `if` as an expression
    // Note: Rust has no ternary like `condition ? true : false`.
    let classification = if growth > 15.0 {
        "High"
    } else {
        "Normal"
    };

    println!("Growth is {}, classified as {}", growth, classification);

    // The preferred loop style in Rust using a range. 
    // `1..=3` means inclusive 1 to 3.
    for i in 1..=3 {
        println!("Processing batch {}...", i);
    }

    // You can break out of a `loop` AND return a value simultaneously!
    let mut counter = 0;
    let final_value = loop {
        counter += 1;
        if counter == 10 {
            break counter * 2; // Return this value into `final_value`!
        }
    };
    println!("The loop exited with value: {}", final_value);
}
```

**What this demonstrates**: Idiomatic implicit returns without semicolons, the power of `if` bindings (ternary replacement), iterating over exclusive/inclusive ranges, and Rust's unique ability to return values directly out of an infinite `loop` construct using `break`.

## ⚠️ Common Mistakes

| Mistake | Why it's wrong | Correct approach |
|---------|---------------|-----------------|
| Trying to mutate without `mut` | The compiler strictly enforces immutability to prevent untracked state changes. | Add `mut`: `let mut obj = 5;` |
| Accidentally returning `()` | Putting a semicolon at the end of what you thought was your return value (`x + 5;`) turns it into a statement, yielding the unit type `()`. | Remove the semicolon on the final line: `x + 5` or use explicit `return x + 5;`. |
| Different types in `if`/`else` arms | Because `let x = if cond { A } else { B };` evaluates to a single type for `x`, both inner arms must output the *exact same type*. | Ensure both arms return the same type. You cannot return `"string"` and `0`. |
