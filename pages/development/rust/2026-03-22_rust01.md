# Rust

## 🧠 Concept Overview

Rust is a systems programming language that guarantees memory safety and thread safety **at compile time** — without a garbage collector. Where C and C++ give you raw power but let you shoot yourself in the foot, and Java/Go/Python keep you safe via runtime garbage collection (at a performance cost), Rust finds a third path: a strict type system and borrow checker that prevents entire categories of bugs before your code ever runs. The tradeoff is a steeper learning curve and a compiler that will fight you until you internalize its rules. But once you do, the code that compiles is remarkably reliable.

Let's build up from the absolute basics.

### Variables & Mutability

In Rust, variables are **immutable by default**. This is the opposite of most languages, and it's a deliberate choice — immutability makes code easier to reason about and prevents accidental mutations.

```rust
let x = 5;      // immutable — you cannot reassign x
// x = 10;     // ERROR: cannot assign twice to immutable variable `x`

let mut y = 5;   // mutable — you CAN reassign y
y = 10;          // OK
```

If you come from Python or JavaScript, this might feel restrictive. From C/C++, compare it to `const` being the default:

```c
// In C, mutable is the default
int x = 5;
x = 10;          // fine

const int y = 5;
// y = 10;      // error
```

Rust flips this convention because immutability-by-default prevents an enormous class of bugs in concurrent code (which we'll see later in the curriculum).

#### Shadowing vs. Mutation

Rust lets you **shadow** a variable by using `let` again with the same name. This is NOT the same as mutation — it creates a brand new variable that happens to reuse the name:

```rust
let x = 5;
let x = x + 1;       // shadowing — new variable, same name
let x = x * 2;       // shadowing again
println!("{x}");      // 12

// Shadowing even allows type changes:
let spaces = "   ";          // &str
let spaces = spaces.len();   // usize — different type, same name!
```

Mutation (`mut`) does NOT allow type changes:

```rust
let mut spaces = "   ";
// spaces = spaces.len();    // ERROR: mismatched types
```

This distinction matters in practice: shadowing is great for transforming data step-by-step (like a pipeline), while `mut` is for when you genuinely need to modify a value in place.

### Data Types

Rust is **statically typed** — every variable's type is known at compile time. The compiler can usually **infer** the type, but sometimes you need to annotate explicitly.

#### Scalar Types

| Type | Variants | Default | Size | Notes |
|------|----------|---------|------|-------|
| Integer | `i8`, `i16`, `i32`, `i64`, `i128`, `isize` | `i32` | 1-16 bytes | Signed |
| Unsigned | `u8`, `u16`, `u32`, `u64`, `u128`, `usize` | — | 1-16 bytes | Unsigned |
| Float | `f32`, `f64` | `f64` | 4 or 8 bytes | IEEE 754 |
| Boolean | `bool` | — | 1 byte | `true` / `false` |
| Character | `char` | — | 4 bytes | Unicode scalar value (NOT a byte!) |

`isize` and `usize` match your CPU architecture: 8 bytes on 64-bit systems. `usize` is used for indexing collections — you'll see it everywhere.

**Integer overflow**: In debug mode, Rust panics on overflow. In release mode, it wraps (two's complement). If you want explicit wrapping, use methods like `wrapping_add`, `checked_add`, or `saturating_add`.

```rust
let a: u8 = 255;
// let b = a + 1;  // debug mode: panics! release mode: wraps to 0
let b = a.wrapping_add(1);  // explicitly wraps: b = 0
let c = a.saturating_add(1); // saturates at max: c = 255
let d = a.checked_add(1);    // returns Option<u8>: d = None
```

This is way better than C's silent undefined behavior on signed overflow.

**`char` is 4 bytes**: Unlike C's `char` (1 byte, ASCII), Rust's `char` represents a Unicode Scalar Value. This means it can hold any Unicode character — emoji, CJK characters, etc. — but it's always 4 bytes (the size of a `u32`).

```rust
let c = 'z';
let heart = '❤';
let kanji = '漢';
println!("{} {} {}", c, heart, kanji); // z ❤ 漢
```

#### Compound Types

**Tuples** — fixed-size, heterogeneous collections:

```rust
let tup: (i32, f64, u8) = (500, 6.4, 1);

// Destructuring
let (x, y, z) = tup;
println!("y is {y}"); // 6.4

// Index access (0-based)
let five_hundred = tup.0;
let six_point_four = tup.1;
```

The **unit type** `()` is a special empty tuple — it's Rust's equivalent of `void`. Functions that don't return anything implicitly return `()`.

**Arrays** — fixed-size, homogeneous, stack-allocated:

```rust
let a = [1, 2, 3, 4, 5];        // [i32; 5]
let b = [3; 5];                   // [3, 3, 3, 3, 3] — 5 elements, all 3
let first = a[0];                  // 1
// let crash = a[10];             // PANICS at runtime! Rust checks bounds.
```

Arrays in Rust are **always bounds-checked at runtime**. This is a massive safety improvement over C, where out-of-bounds access is undefined behavior. The performance cost is negligible — the branch predictor almost always gets it right.

For dynamically-sized collections, you'll use `Vec<T>` (Day 6).

### Functions

Functions are declared with `fn`. Rust is expression-based, which means almost everything is an expression that returns a value.

```rust
fn add(x: i32, y: i32) -> i32 {
    x + y    // NO semicolon = this is the return value (expression)
}

fn main() {
    let result = add(5, 3);
    println!("5 + 3 = {result}");  // 5 + 3 = 8
}
```

**The semicolon rule**: This catches every Rust beginner:
- `x + y` (no semicolon) — an **expression** that evaluates to the sum
- `x + y;` (with semicolon) — a **statement** that evaluates to `()` (the unit type)

```rust
fn broken_add(x: i32, y: i32) -> i32 {
    x + y;   // ERROR: expected `i32`, found `()`
             // The semicolon turns the expression into a statement!
}
```

This is one of the most common Rust beginner mistakes. If your function isn't returning what you expect, check for a stray semicolon.

**Early returns**: You can use `return` for explicit early returns:

```rust
fn abs(x: i32) -> i32 {
    if x < 0 {
        return -x;    // explicit early return
    }
    x                 // implicit return (last expression, no semicolon)
}
```

**Type annotations are required** on function signatures. Rust never infers function parameter or return types — this is a deliberate design choice for readability and compile-time performance.

### Statements vs. Expressions

This is fundamental to understanding Rust. In C-family languages, `if` is a statement. In Rust, `if` is an **expression** — it returns a value:

```rust
let condition = true;
let number = if condition { 5 } else { 6 };
// number = 5

// Both arms must return the same type!
// let broken = if condition { 5 } else { "six" };  // ERROR: mismatched types
```

Blocks `{}` are also expressions — the last expression in a block is its value:

```rust
let y = {
    let x = 3;
    x + 1       // no semicolon — this is the block's return value
};
println!("{y}"); // 4
```

This expression-oriented design eliminates the need for ternary operators (`? :` in C/Java) and makes code more composable.

### Control Flow

#### `if` / `else if` / `else`

Conditions in Rust do **NOT** need parentheses (unlike C/Java), but the braces are **mandatory**:

```rust
let number = 7;

if number < 5 {
    println!("less than 5");
} else if number == 5 {
    println!("exactly 5");
} else {
    println!("greater than 5");
}
```

Conditions must be `bool` — no truthy/falsy values. `if 1 { ... }` won't compile. This prevents a whole class of bugs.

#### `loop` — Infinite Loop

```rust
let mut counter = 0;
let result = loop {
    counter += 1;
    if counter == 10 {
        break counter * 2;   // break WITH a value — loop is an expression!
    }
};
println!("{result}"); // 20
```

**Loop labels** — for breaking out of nested loops:

```rust
let mut count = 0;
'counting_up: loop {
    let mut remaining = 10;
    loop {
        if remaining == 9 {
            break;              // breaks inner loop
        }
        if count == 2 {
            break 'counting_up; // breaks OUTER loop
        }
        remaining -= 1;
    }
    count += 1;
}
```

#### `while` Loop

```rust
let mut number = 3;
while number != 0 {
    println!("{number}!");
    number -= 1;
}
println!("LIFTOFF!");
```

#### `for` Loop (the most common)

`for` iterates over anything that implements `Iterator`. It's the most idiomatic loop in Rust:

```rust
let a = [10, 20, 30, 40, 50];
for element in a {
    println!("the value is: {element}");
}

// Range syntax
for number in 1..4 {
    println!("{number}"); // 1, 2, 3 (exclusive upper bound)
}

for number in 1..=4 {
    println!("{number}"); // 1, 2, 3, 4 (inclusive upper bound)
}

// Reverse
for number in (1..4).rev() {
    println!("{number}"); // 3, 2, 1
}
```

Prefer `for` over `while` for iterating collections — it's bounds-safe, can't accidentally go out of bounds, and the compiler can often optimize away the bounds check entirely.

### Type Inference & the Turbofish

Rust's type inference is powerful. Most of the time, you don't need explicit annotations:

```rust
let x = 42;             // inferred as i32
let y = 3.14;            // inferred as f64
let z = true;            // inferred as bool
let s = "hello";         // inferred as &str
```

But sometimes the compiler needs help, especially with generic functions. That's where the **turbofish** syntax (`::<>`) comes in:

```rust
let guess: i32 = "42".parse().expect("Not a number!");
// or equivalently, using the turbofish:
let guess = "42".parse::<i32>().expect("Not a number!");
```

Both are equivalent — whichever you find more readable is fine.

### `println!` and Formatting

`println!` is a **macro** (notice the `!`), not a function. Macros are expanded at compile time.

```rust
let name = "Rustacean";
let age = 3;
let pi = 3.14159;

println!("Hello, {name}!");              // variable capture (Rust 2021+)
println!("Hello, {}!", name);            // positional placeholder
println!("{0} is {1}, {0} is cool", name, age); // indexed
println!("Pi to 2 places: {pi:.2}");     // formatting: 3.14
println!("Binary: {:b}", 42);            // 101010
println!("Hex: {:x}", 255);              // ff
println!("Debug: {:?}", (1, 2, 3));      // (1, 2, 3)
println!("Pretty: {:#?}", vec![1,2,3]);  // multi-line debug
```

The `{:?}` formatter uses the `Debug` trait (most types implement it). The `{}` formatter uses the `Display` trait (you have to implement it yourself for custom types). We'll cover traits deeply on Day 9.

### Constants & Statics

**Constants** (`const`) are inlined at compile time — they have no fixed memory address:

```rust
const MAX_POINTS: u32 = 100_000;  // type annotation required
                                   // underscores in numbers are OK
```

**Statics** (`static`) have a fixed address in memory and live for the entire program:

```rust
static LANGUAGE: &str = "Rust";
static mut COUNTER: u32 = 0;   // mutable statics are unsafe to access!
```

Mutable statics are `unsafe` because accessing them from multiple threads is a data race. Prefer `const` unless you specifically need a fixed memory address.

### A Note on Ownership (Preview)

We'll dive deep into Ownership on Day 2, but you need to know this exists: every value in Rust has exactly one **owner** variable. When the owner goes out of scope, the value is dropped (freed). This is how Rust manages memory without a GC.

```rust
{
    let s = String::from("hello"); // s owns the String
    // use s here...
}// s goes out of scope — String is dropped (memory freed)
```

If you try to use `s` after the block, the compiler will reject it. This is Rust's core innovation — it shifts memory management from runtime (GC or manual) to compile time (ownership + borrow checker).

## 🔑 Key Points

- **Variables are immutable by default** — use `mut` for mutation. Shadowing (`let x = ...` again) creates a new variable and allows type changes; mutation does not.
- **Rust is expression-based** — `if`, blocks, and `loop` all return values. The last expression without a semicolon is the return value. Adding a semicolon turns it into a statement returning `()`.
- **Type annotations are required on function signatures** but usually inferred inside function bodies. The turbofish `::<T>` helps when the compiler can't infer a generic type.
- **All array/slice access is bounds-checked at runtime** — no buffer overflows, unlike C. The performance cost is minimal.
- **Integer overflow panics in debug mode**, wraps in release mode. Use `checked_*`, `wrapping_*`, or `saturating_*` methods for explicit behavior.

## 💻 Code Examples

### Example 1 — Variable Binding, Shadowing, and Type Demo

```rust
// File: main.rs
fn main() {
    // Immutable by default
    let x = 5;
    println!("x = {x}");

    // Shadowing: new variable, same name, can even change type
    let x = x + 1;          // x is now 6 (still i32)
    let x = x as f64 * 1.5; // x is now 9.0 (f64!)
    println!("After shadowing: x = {x}");

    // Mutable variable
    let mut counter = 0;
    for _ in 0..5 {
        counter += 1;
    }
    println!("Counter: {counter}"); // 5

    // Type annotations and turbofish
    let parsed: i64 = "9999".parse().unwrap();
    let also_parsed = "1234".parse::<u16>().unwrap();
    println!("Parsed: {parsed}, Also parsed: {also_parsed}");

    // Destructuring a tuple
    let point = (3.0, 4.0);
    let (px, py) = point;
    let distance = (px * px + py * py).sqrt();
    println!("Distance from origin: {distance:.2}"); // 5.00

    // Integer overflow methods
    let max_u8: u8 = 255;
    println!("Wrapping: {}", max_u8.wrapping_add(1));       // 0
    println!("Saturating: {}", max_u8.saturating_add(10));   // 255
    println!("Checked: {:?}", max_u8.checked_add(1));        // None
}
```

**What this demonstrates**: How shadowing differs from mutation, type conversions with `as`, the turbofish for `parse`, tuple destructuring, and Rust's explicit overflow-handling methods — all core patterns you'll use every day.

### Example 2 — Expression-based Control Flow

```rust
// File: main.rs
fn main() {
    // if as an expression (like ternary in C, but more powerful)
    let temperature = 22;
    let description = if temperature > 30 {
        "hot"
    } else if temperature > 15 {
        "pleasant"
    } else {
        "cold"
    };
    println!("The weather is {description}");

    // loop as an expression with break value
    let mut n = 0;
    let first_square_over_100 = loop {
        n += 1;
        if n * n > 100 {
            break n * n;
        }
    };
    println!("First perfect square over 100: {first_square_over_100}"); // 121

    // Block as an expression
    let area = {
        let width = 10.0_f64;
        let height = 5.5;
        width * height  // no semicolon = this is the block's value
    };
    println!("Area: {area}"); // 55.0

    // for loop with enumerate (like Python's enumerate)
    let fruits = ["apple", "banana", "cherry"];
    for (index, fruit) in fruits.iter().enumerate() {
        println!("{index}: {fruit}");
    }
}
```

**What this demonstrates**: Rust's expression-oriented design — `if`, `loop`, and blocks all produce values that can be bound to variables. This eliminates the need for ternary operators and makes code flow more naturally.

### Example 3 — Functions, Return Values, and Documentation

```rust
// File: main.rs

/// Converts Fahrenheit to Celsius.
/// Formula: C = (F - 32) × 5/9
fn fahrenheit_to_celsius(f: f64) -> f64 {
    (f - 32.0) * 5.0 / 9.0  // implicit return — no semicolon
}

/// Returns the nth Fibonacci number using iterative computation.
/// Panics if n is so large it causes overflow.
fn fibonacci(n: u32) -> u64 {
    if n == 0 { return 0; }     // early return
    if n == 1 { return 1; }

    let mut prev = 0_u64;
    let mut curr = 1_u64;
    for _ in 2..=n {
        let next = prev + curr;
        prev = curr;
        curr = next;
    }
    curr   // implicit return
}

/// Returns true if a number is prime.
fn is_prime(n: u64) -> bool {
    if n < 2 { return false; }
    if n == 2 { return true; }
    if n % 2 == 0 { return false; }

    let limit = (n as f64).sqrt() as u64;
    for i in (3..=limit).step_by(2) {
        if n % i == 0 {
            return false;
        }
    }
    true
}

fn main() {
    // Temperature conversion
    let temps_f = [32.0, 72.0, 100.0, 212.0];
    for f in temps_f {
        let c = fahrenheit_to_celsius(f);
        println!("{f:.1}°F = {c:.1}°C");
    }

    println!();

    // Fibonacci sequence
    for i in 0..=10 {
        println!("fib({i}) = {}", fibonacci(i));
    }

    println!();

    // Prime check
    for n in 1..=20 {
        if is_prime(n) {
            print!("{n} ");
        }
    }
    println!("are the primes up to 20");
}
```

**What this demonstrates**: Complete, idiomatic Rust functions with doc comments (`///`), early returns, implicit returns, iterator methods like `step_by`, and practical numeric computations. Notice how `is_prime`'s final `true` has no semicolon — it's the implicit return.

### Example 4 — Common Compiler Errors (and How to Fix Them)

```rust
// File: main.rs

fn main() {
    // ❌ ERROR 1: Immutable variable reassignment
    // let x = 5;
    // x = 10;  // error[E0384]: cannot assign twice to immutable variable

    // ✅ FIX: use mut
    let mut x = 5;
    x = 10;
    println!("x = {x}");

    // ❌ ERROR 2: Type mismatch in if expression
    // let val = if true { 5 } else { "no" };
    // error[E0308]: `if` and `else` have incompatible types

    // ✅ FIX: both arms must return the same type
    let val = if true { 5 } else { 0 };
    println!("val = {val}");

    // ❌ ERROR 3: Semicolon kills the return value
    // fn add(a: i32, b: i32) -> i32 { a + b; }
    // error[E0308]: expected `i32`, found `()`

    // ✅ FIX: remove the semicolon
    let sum = add(3, 4);
    println!("3 + 4 = {sum}");

    // ❌ ERROR 4: Using a moved value (ownership preview)
    let s1 = String::from("hello");
    let s2 = s1;   // s1 is MOVED to s2
    // println!("{s1}");  // error[E0382]: borrow of moved value: `s1`

    // ✅ FIX: clone if you need both
    let s3 = String::from("hello");
    let s4 = s3.clone();
    println!("s3 = {s3}, s4 = {s4}"); // both valid
}

fn add(a: i32, b: i32) -> i32 {
    a + b  // no semicolon!
}
```

**What this demonstrates**: The four most common beginner mistakes and exactly how to fix them. Understanding these error codes (`E0384`, `E0308`, `E0382`) early will save you hours of frustration.

## ⚠️ Common Mistakes

| Mistake | Why it's wrong | Correct approach |
|---------|---------------|-----------------|
| Adding a semicolon to the last expression in a function: `x + y;` | Turns the expression into a statement returning `()`, causing a type mismatch | Remove the semicolon: `x + y` |
| Using `if x { ... }` where `x` is an integer | Rust has no truthy/falsy — conditions must be `bool` | Use `if x != 0 { ... }` |
| Shadowing confusion: thinking `let x = x + 1` mutates the original | Shadowing creates a NEW variable, the old one is gone | If you need mutation, use `let mut x` |
| Assuming `char` is 1 byte like in C | Rust `char` is 4 bytes (a Unicode scalar value) | Use `u8` for raw bytes, `char` for Unicode characters |
| Ignoring integer overflow in release mode | Release builds silently wrap on overflow — potential logic bugs | Use `checked_*` or `saturating_*` when overflow matters |
| Writing `return x;` on the last line | Not wrong, but unidiomatic — Rust prefers implicit returns | Just write `x` (no return keyword, no semicolon) |

## 🗒️ Quick Cheat Sheet

```rust
let x = 5;                // immutable binding
let mut y = 10;           // mutable binding
let x = x + 1;            // shadowing (new variable)

fn add(a: i32, b: i32) -> i32 { a + b }  // implicit return (no ;)

let val = if cond { a } else { b };       // if-expression
let n = loop { break 42; };              // loop-expression

for i in 0..10 { }         // exclusive range [0, 10)
for i in 0..=10 { }        // inclusive range [0, 10]
for (i, v) in arr.iter().enumerate() { }  // index + value

let arr = [0; 5];           // [0, 0, 0, 0, 0]
```
