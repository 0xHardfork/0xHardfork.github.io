# Rust

## 🧠 Concept Overview

Ownership is *the* defining feature of Rust. It's what makes Rust unique among all mainstream programming languages, and it's the mechanism that gives you memory safety without a garbage collector. Every other language either uses garbage collection (Java, Go, Python, JavaScript) or gives you raw pointers and says "good luck" (C, C++). Rust does neither. Instead, the compiler enforces a set of ownership rules at compile time, and if your code breaks those rules, it simply won't compile. No runtime overhead, no GC pauses, no use-after-free, no double-free, no data races. The cost? You have to learn to think about memory in a way you've never had to before.

### The Three Ownership Rules

Everything in Rust boils down to three rules:

1. **Each value in Rust has exactly one owner** (a variable).
2. **There can only be one owner at a time.**
3. **When the owner goes out of scope, the value is dropped** (its memory is freed).

That's it. These three rules, enforced at compile time, eliminate entire categories of bugs. Let's walk through what each one means in practice.

### Rule 1 & 2: One Owner At a Time

```rust
fn main() {
    let s1 = String::from("hello");  // s1 owns the String
    let s2 = s1;                     // ownership MOVES to s2
    // println!("{}", s1);           // ❌ COMPILE ERROR: s1 is no longer valid
    println!("{}", s2);              // ✅ s2 is the owner now
}
```

If you come from C++, this looks like a copy. But it's not. Rust performs a **move** by default for heap-allocated types. After `let s2 = s1`, the variable `s1` is *invalidated*. The compiler won't let you use it. The actual data on the heap hasn't moved — only the pointer/metadata on the stack was copied, and `s1` was marked as "moved from."

Why does Rust do this? Because if both `s1` and `s2` pointed to the same heap data, and both went out of scope, Rust would call `drop` (its equivalent of a destructor/free) twice — a **double-free** bug. Instead of tracking this with reference counting (like `shared_ptr` in C++) or garbage collection, Rust simply makes it impossible at compile time.

### What Actually Happens in Memory

Let's look under the hood. A `String` in Rust is a struct with three fields on the stack:

```
Stack (s1):
┌──────────┬───────┬──────────┐
│ ptr      │ len   │ capacity │
│ 0x7f3a.. │ 5     │ 5        │
└──────────┴───────┴──────────┘
        │
        ▼
Heap:  [h][e][l][l][o]
```

After `let s2 = s1`:

```
Stack (s1): [INVALIDATED — compiler marks as moved]

Stack (s2):
┌──────────┬───────┬──────────┐
│ ptr      │ len   │ capacity │
│ 0x7f3a.. │ 5     │ 5        │
└──────────┴───────┴──────────┘
        │
        ▼
Heap:  [h][e][l][l][o]    (same allocation, not copied)
```

The heap data is untouched. Only the 24-byte stack metadata was copied, and `s1` was invalidated. This is a **zero-cost abstraction** — no allocation, no deep copy, no runtime check.

### Copy vs Move — The `Copy` Trait

Not all types move. Simple, stack-only types like integers, floats, booleans, and characters implement the `Copy` trait, which means assignment creates a bitwise copy instead of a move:

```rust
fn main() {
    let x = 42;
    let y = x;     // Copy, not move! x is still valid
    println!("x = {}, y = {}", x, y); // ✅ Both valid
}
```

Why do integers copy but Strings move? Because copying an integer is cheap (just copy 4-8 bytes on the stack) and there's no heap allocation that could be double-freed. The `Copy` trait is Rust's way of saying "this type is trivially copiable and has no ownership semantics."

Types that implement `Copy`:
- All integer types (`i8`, `i16`, `i32`, `i64`, `i128`, `isize`, and their unsigned `u*` variants)
- `f32`, `f64`
- `bool`
- `char`
- Tuples, **only if all elements are `Copy`**: `(i32, f64)` is `Copy`, but `(i32, String)` is NOT
- Fixed-size arrays of `Copy` types: `[i32; 5]` is `Copy`
- References (`&T`) — but NOT mutable references (`&mut T`)

Types that do NOT implement `Copy` (they move instead):
- `String`
- `Vec<T>`
- `Box<T>`
- Any struct containing a non-`Copy` field (unless you explicitly derive `Copy`)

### Rule 3: Drop When Out of Scope

```rust
fn main() {
    {
        let s = String::from("hello");
        // s is valid here, we can use it
    } // s goes out of scope → Rust calls drop(s) → heap memory is freed

    // s is no longer accessible here
}
```

This is Rust's RAII (Resource Acquisition Is Initialization) pattern, borrowed from C++. When a variable goes out of scope, Rust automatically calls its `Drop` implementation (if it has one). For `String`, this frees the heap buffer. For `File`, this closes the file handle. For `MutexGuard`, this releases the lock.

No need for `defer` (Go), `finally` (Java), `with` (Python), or manual `free()` (C). Scope-based cleanup is automatic, deterministic, and zero-overhead.

### Ownership and Functions

Passing a value to a function transfers ownership, just like assignment:

```rust
fn takes_ownership(s: String) {
    println!("Got: {}", s);
} // s is dropped here — memory freed

fn makes_copy(x: i32) {
    println!("Got: {}", x);
} // x is Copy, nothing special happens

fn main() {
    let s = String::from("hello");
    takes_ownership(s);
    // println!("{}", s); // ❌ COMPILE ERROR: s was moved into the function

    let x = 42;
    makes_copy(x);
    println!("{}", x); // ✅ x was copied, original still valid
}
```

This is where newcomers first feel the "fighting the borrow checker" pain. You pass a string to a function, and suddenly you can't use it anymore. The solution? **References and borrowing.**

### Returning Ownership

Functions can also *return* ownership:

```rust
fn give_ownership() -> String {
    let s = String::from("yours now");
    s // ownership is moved to the caller
}

fn take_and_give_back(s: String) -> String {
    println!("Borrowed briefly: {}", s);
    s // give it back
}

fn main() {
    let s1 = give_ownership();          // s1 gets ownership
    let s2 = take_and_give_back(s1);    // s1 moved in, result moved to s2
    println!("{}", s2);                 // ✅
}
```

But this "take and give back" pattern is clunky. Imagine having to return a value every time just to keep using it. That's why Rust has **references**.

## References and Borrowing

A **reference** lets you access a value without taking ownership. Think of it as a pointer that doesn't own the data — you're "borrowing" it:

```rust
fn calculate_length(s: &String) -> usize {
    s.len()
} // s goes out of scope, but it doesn't own the String, so nothing is dropped

fn main() {
    let s = String::from("hello");
    let len = calculate_length(&s);  // & creates a reference
    println!("'{}' has length {}", s, len); // ✅ s is still valid!
}
```

The `&` operator creates a reference. The function parameter `s: &String` means "I'm borrowing a `String`, not owning it." When the function returns, the reference goes out of scope, but since it doesn't own the data, nothing is freed.

### The Two Borrowing Rules

Rust enforces two critical borrowing rules at compile time:

1. **You can have EITHER one mutable reference OR any number of immutable references** — but not both at the same time.
2. **References must always be valid** — no dangling pointers.

Let's break these down.

### Rule 1: Mutable XOR Immutable

**Multiple immutable references are fine:**

```rust
fn main() {
    let s = String::from("hello");
    let r1 = &s;  // immutable borrow
    let r2 = &s;  // another immutable borrow — totally fine!
    println!("{} and {}", r1, r2); // ✅
}
```

**One mutable reference is fine:**

```rust
fn main() {
    let mut s = String::from("hello");
    let r = &mut s;        // mutable borrow
    r.push_str(" world");
    println!("{}", r);     // ✅ "hello world"
}
```

**Mixing mutable and immutable is NOT allowed:**

```rust
fn main() {
    let mut s = String::from("hello");
    let r1 = &s;        // immutable borrow
    let r2 = &s;        // immutable borrow
    let r3 = &mut s;    // ❌ COMPILE ERROR: cannot borrow as mutable
                         //    because it is also borrowed as immutable
    println!("{}, {}, {}", r1, r2, r3);
}
```

Why this restriction? It prevents **data races** at compile time. A data race occurs when:
- Two or more pointers access the same data
- At least one writes
- There's no synchronization

By allowing either multiple readers OR one writer (but never both), Rust makes data races structurally impossible. This isn't a runtime check — it's a compile-time guarantee.

### Non-Lexical Lifetimes (NLL)

Modern Rust (since edition 2018) uses **Non-Lexical Lifetimes**, which means a reference's "alive" period ends at its last use, not at the end of the scope block:

```rust
fn main() {
    let mut s = String::from("hello");
    let r1 = &s;         // immutable borrow starts
    let r2 = &s;         // immutable borrow starts
    println!("{} {}", r1, r2); // last use of r1 and r2
    // r1 and r2 are no longer alive here (NLL)

    let r3 = &mut s;     // ✅ This is fine now! No conflict.
    r3.push_str(" world");
    println!("{}", r3);
}
```

Before NLL, this wouldn't compile because `r1` and `r2` would be considered alive until the end of the block. NLL made the borrow checker much more ergonomic by tracking actual usage, not just scoping.

### Rule 2: No Dangling References

In C/C++, you can create a pointer to data that's been freed (a dangling pointer). Rust prevents this at compile time:

```rust
fn dangle() -> &String {  // ❌ COMPILE ERROR
    let s = String::from("hello");
    &s  // s is dropped when this function returns —
        // the reference would point to freed memory!
}
```

The compiler error is: `this function's return type contains a borrowed value, but there is no value for it to be borrowed from`. The fix is to return the owned value instead:

```rust
fn no_dangle() -> String {
    let s = String::from("hello");
    s  // ownership is moved to the caller — no dangling reference
}
```

### Mutable References in Detail

A mutable reference (`&mut T`) gives you exclusive read-write access:

```rust
fn append_exclamation(s: &mut String) {
    s.push_str("!");
}

fn main() {
    let mut greeting = String::from("Hello");
    append_exclamation(&mut greeting);
    println!("{}", greeting); // "Hello!"
}
```

Key rules about `&mut`:
- The variable must be declared `mut`: `let mut s = ...`
- You create the mutable reference with `&mut s`
- Only ONE `&mut` reference can exist at a time
- You CANNOT have `&` and `&mut` references alive simultaneously

### Borrowing in Struct Methods

Methods use `&self` (immutable borrow), `&mut self` (mutable borrow), or `self` (take ownership):

```rust
struct TextBuffer {
    content: String,
}

impl TextBuffer {
    // Borrows self immutably — can read but not modify
    fn len(&self) -> usize {
        self.content.len()
    }

    // Borrows self mutably — can modify
    fn append(&mut self, text: &str) {
        self.content.push_str(text);
    }

    // Takes ownership — consumes the buffer (rare, used for builder patterns)
    fn into_string(self) -> String {
        self.content
    }
}

fn main() {
    let mut buf = TextBuffer {
        content: String::from("Hello"),
    };

    println!("Length: {}", buf.len());    // immutable borrow
    buf.append(" World");                  // mutable borrow
    println!("Length: {}", buf.len());    // immutable borrow again

    let s = buf.into_string();            // buf is consumed (moved)
    // println!("{}", buf.content);       // ❌ buf was moved
    println!("{}", s);                    // ✅ "Hello World"
}
```

## Slices — Borrowed Views Into Data

A **slice** is a reference to a contiguous sequence of elements in a collection. It doesn't own the data — it borrows it:

```rust
fn main() {
    let s = String::from("hello world");

    let hello = &s[0..5];   // &str slice — borrows bytes 0-4
    let world = &s[6..11];  // &str slice — borrows bytes 6-10

    println!("{} {}", hello, world); // "hello world"
}
```

The type `&str` is a **string slice** — it's a fat pointer containing:
- A pointer to the start of the data
- A length (how many bytes)

```
Stack (hello):
┌──────────┬───────┐
│ ptr      │ len   │
│ 0x7f3a.. │ 5     │
└──────────┴───────┘
        │
        ▼
Heap:  [h][e][l][l][o][ ][w][o][r][l][d]
                        ▲
                        │
Stack (world):          │
┌──────────┬───────┐    │
│ ptr      │ len   │────┘
│ ...+6    │ 5     │
└──────────┴───────┘
```

### String Literals Are Slices

When you write `let s = "hello";`, the type of `s` is `&str` — a string slice pointing to data baked into the program binary. It has a `'static` lifetime (lives for the entire program):

```rust
let s: &str = "hello";  // points into read-only program memory
```

This is why `&str` is preferred over `&String` in function parameters — it's more general:

```rust
// ✅ Accepts both String references and string literals
fn greet(name: &str) {
    println!("Hello, {}!", name);
}

fn main() {
    let owned = String::from("Alice");
    greet(&owned);     // &String automatically coerces to &str (Deref coercion)
    greet("Bob");      // &str literal works too
}
```

### Array/Vec Slices

Slices work on arrays and vectors too:

```rust
fn sum(numbers: &[i32]) -> i32 {
    numbers.iter().sum()
}

fn main() {
    let arr = [1, 2, 3, 4, 5];
    let vec = vec![10, 20, 30, 40, 50];

    println!("Array sum: {}", sum(&arr));        // whole array
    println!("Slice sum: {}", sum(&arr[1..4]));  // [2, 3, 4]
    println!("Vec sum: {}", sum(&vec));           // whole vec
    println!("Vec slice: {}", sum(&vec[2..]));    // [30, 40, 50]
}
```

The parameter `&[i32]` is a slice of `i32` — it works with arrays, vectors, or any contiguous i32 data. This is one of Rust's most elegant abstractions.

### Slices Enforce Borrowing Rules

Because slices are references, they follow borrowing rules:

```rust
fn main() {
    let mut v = vec![1, 2, 3, 4, 5];
    let slice = &v[1..3]; // immutable borrow of v
    // v.push(6);          // ❌ COMPILE ERROR: can't mutate v while slice exists
    println!("{:?}", slice);
    // After last use of slice, v can be mutated again
    v.push(6); // ✅
}
```

This prevents a subtle C++ bug: if you hold an iterator/pointer into a `vector` and then push into it, the vector might reallocate, invalidating your iterator. Rust catches this at compile time.

## Clone — Explicit Deep Copying

When you need a true deep copy of a heap-allocated value, use `.clone()`:

```rust
fn main() {
    let s1 = String::from("hello");
    let s2 = s1.clone(); // deep copy — new heap allocation
    println!("s1 = {}, s2 = {}", s1, s2); // ✅ both valid
}
```

`.clone()` is explicit because it might be expensive (heap allocation + memcpy). Rust wants you to see the cost in your code, unlike C++ where implicit copies can hide performance problems.

### When to Clone vs Borrow

| Use Case | Approach |
|----------|----------|
| Read-only access | `&T` (immutable reference) |
| Need to modify | `&mut T` (mutable reference) |
| Need independent copy for a thread | `.clone()` |
| Building data structures that need ownership | Move or `.clone()` |
| Quick prototype, fighting the borrow checker | `.clone()` (then optimize later) |

A practical tip: when you're learning Rust, it's totally fine to `.clone()` liberally to get things compiling. Once you understand ownership better, you can go back and replace clones with references for better performance.

## Ownership in Real-World Patterns

### Pattern 1: Builder Pattern with Ownership

```rust
struct Config {
    host: String,
    port: u16,
    verbose: bool,
}

struct ConfigBuilder {
    host: String,
    port: u16,
    verbose: bool,
}

impl ConfigBuilder {
    fn new() -> Self {
        ConfigBuilder {
            host: String::from("localhost"),
            port: 8080,
            verbose: false,
        }
    }

    // Each method takes ownership (self) and returns it
    fn host(mut self, host: &str) -> Self {
        self.host = host.to_string();
        self
    }

    fn port(mut self, port: u16) -> Self {
        self.port = port;
        self
    }

    fn verbose(mut self, verbose: bool) -> Self {
        self.verbose = verbose;
        self
    }

    fn build(self) -> Config {
        Config {
            host: self.host,
            port: self.port,
            verbose: self.verbose,
        }
    }
}

fn main() {
    let config = ConfigBuilder::new()
        .host("example.com")
        .port(3000)
        .verbose(true)
        .build();

    println!("{}:{} (verbose={})", config.host, config.port, config.verbose);
}
```

### Pattern 2: Splitting Borrows on Structs

```rust
struct Document {
    title: String,
    body: String,
    word_count: usize,
}

impl Document {
    // You CAN borrow different fields simultaneously
    fn title_and_body(&self) -> (&str, &str) {
        (&self.title, &self.body)
    }

    // You CAN mutably borrow one field while immutably borrowing another
    fn update_word_count(&mut self) {
        // self.body is borrowed immutably (to count words)
        // self.word_count is borrowed mutably (to update)
        self.word_count = self.body.split_whitespace().count();
    }
}
```

The borrow checker is smart enough to know that borrowing `self.title` and mutating `self.word_count` don't conflict because they're separate fields. However, this only works within a single method — across method calls, the compiler sees `&mut self` as borrowing the *entire* struct.

## 🔑 Key Points

- **Ownership = single owner, automatic cleanup.** Each value has exactly one owner. When the owner goes out of scope, the value is dropped (freed). No garbage collector needed.
- **Move semantics by default for heap types.** Assignment and function calls transfer ownership. The original variable becomes invalid. Stack-only `Copy` types are bitwise copied instead.
- **References (`&T`, `&mut T`) borrow without taking ownership.** Immutable references allow read access; mutable references allow write access. You can have many `&T` or one `&mut T`, never both.
- **The borrow checker prevents data races at compile time.** The "mutable XOR immutable" rule + "no dangling references" rule eliminates use-after-free, double-free, and data races.
- **Slices (`&[T]`, `&str`) are lightweight borrowed views.** They reference a contiguous sequence without copying. Prefer `&str` over `&String` and `&[T]` over `&Vec<T>` in function parameters.

## 💻 Code Examples

### Example 1 — Ownership Transfer Through a Pipeline

```rust
// File: pipeline.rs
fn main() {
    // === Demonstrating ownership flow through function calls ===

    let raw_data = String::from("  Hello, Rust World!  ");
    println!("Original: '{}'", raw_data);

    // Each function takes ownership and returns a new owned value
    let trimmed = trim_data(raw_data);
    // println!("{}", raw_data); // ❌ moved into trim_data

    let uppered = to_upper(trimmed);
    // println!("{}", trimmed); // ❌ moved into to_upper

    let tagged = add_tag(uppered, "greeting");

    println!("Final: {}", tagged); // "[greeting] HELLO, RUST WORLD!"
}

fn trim_data(s: String) -> String {
    s.trim().to_string() // .trim() returns &str, .to_string() creates owned String
}

fn to_upper(s: String) -> String {
    s.to_uppercase() // returns a new String (ownership transferred)
}

fn add_tag(s: String, tag: &str) -> String {
    format!("[{}] {}", tag, s) // format! creates a new owned String
}
```

**What this demonstrates**: Each function takes ownership of its `String` argument and returns a new `String`. The original variable is invalidated after each call. The `tag` parameter is `&str` because the function only needs to read it, not own it.

### Example 2 — Borrowing Rules in Practice

```rust
// File: borrowing.rs
fn main() {
    let mut accounts = vec![
        ("Alice".to_string(), 1000u64),
        ("Bob".to_string(), 500),
        ("Charlie".to_string(), 2500),
    ];

    // Multiple immutable borrows — OK
    let total = total_balance(&accounts);
    let richest = find_richest(&accounts);
    println!("Total: ${}, Richest: {}", total, richest);

    // Mutable borrow — no other borrows can coexist
    deposit(&mut accounts, "Alice", 500);
    println!("After deposit: Alice has ${}", accounts[0].1);

    // Slice borrow
    let top_two = &accounts[..2];
    println!("Top two: {:?}", top_two);
    // accounts.push(("Dave".to_string(), 100)); // ❌ can't mutate while slice exists
    drop(top_two); // Not actually needed — NLL drops it after last use above

    // Now mutation is fine again
    accounts.push(("Dave".to_string(), 100));
    println!("All accounts: {:?}", accounts);
}

fn total_balance(accounts: &[(String, u64)]) -> u64 {
    accounts.iter().map(|(_, balance)| balance).sum()
}

fn find_richest(accounts: &[(String, u64)]) -> &str {
    accounts
        .iter()
        .max_by_key(|(_, balance)| balance)
        .map(|(name, _)| name.as_str())
        .unwrap_or("nobody")
}

fn deposit(accounts: &mut Vec<(String, u64)>, name: &str, amount: u64) {
    if let Some(account) = accounts.iter_mut().find(|(n, _)| n == name) {
        account.1 += amount;
    }
}
```

**What this demonstrates**: Real-world borrowing patterns — multiple `&` borrows coexist for read-only operations, `&mut` borrow for mutation, and slice borrows that prevent resizing the underlying vector. Also shows how NLL allows the slice to "expire" after its last use.

### Example 3 — The Classic Ownership Mistakes & Fixes

```rust
// File: mistakes.rs

// === MISTAKE 1: Using a value after move ===
fn mistake_1_fixed() {
    let name = String::from("Alice");

    // WRONG: greet(name); println!("{}", name); // name was moved!
    // FIX: Use a reference
    greet(&name);
    println!("Still have: {}", name); // ✅

    // OR: Use clone if you truly need two independent copies
    let name2 = name.clone();
    greet_owned(name);     // name is moved here
    println!("{}", name2); // ✅ name2 is independent
}

// === MISTAKE 2: Returning a reference to a local ===
// WRONG:
// fn create_greeting() -> &str {
//     let s = String::from("hello");
//     &s  // ❌ s is dropped, reference would dangle
// }
// FIX: Return owned data
fn create_greeting() -> String {
    String::from("hello") // ownership transferred to caller
}

// === MISTAKE 3: Mutable + immutable borrow conflict ===
fn mistake_3_fixed() {
    let mut data = vec![1, 2, 3];

    // WRONG:
    // let first = &data[0]; // immutable borrow
    // data.push(4);         // mutable borrow — CONFLICT!
    // println!("{}", first);

    // FIX: Finish using the immutable borrow before mutating
    let first = data[0]; // Copy the i32 value (i32 is Copy!)
    data.push(4);
    println!("first was: {}, data is now: {:?}", first, data);
}

fn greet(name: &str) {
    println!("Hello, {}!", name);
}

fn greet_owned(name: String) {
    println!("Hello, {} (I own you now)!", name);
}

fn main() {
    mistake_1_fixed();
    println!("{}", create_greeting());
    mistake_3_fixed();
}
```

**What this demonstrates**: The three most common ownership/borrowing mistakes that every Rust newcomer encounters, along with the idiomatic fixes: use references instead of moving, return owned data instead of references to locals, and sequence your borrows to avoid mutable+immutable conflicts.

## ⚠️ Common Mistakes

| Mistake | Why it's wrong | Correct approach |
|---------|---------------|-----------------|
| Using a value after moving it | The variable is invalidated after a move; accessing it is a compile error | Use `&` to borrow instead, or `.clone()` if you need a copy |
| Returning `&String` from a function that creates a local `String` | The `String` is dropped when the function returns — the reference would dangle | Return `String` (owned) instead, transferring ownership to the caller |
| Having `&` and `&mut` references alive simultaneously | Violates the "mutable XOR immutable" rule — prevents data races | Finish using immutable refs before creating a mutable ref (NLL helps) |
| Using `&String` as a function parameter | It works but is unnecessarily restrictive — won't accept `&str` literals | Use `&str` instead — it accepts both `&String` (via Deref coercion) and string literals |
| Calling `.clone()` reflexively to shut up the compiler | Hides actual ownership problems and adds unnecessary heap allocations | Fix the ownership flow first; clone only when truly needed |
| Trying to mutate through an immutable reference | `&T` is read-only; you can't modify the data through it | Use `&mut T` and ensure the variable is declared `mut` |

**Starter code**:

```rust
fn main() {
    let s = String::from("ownership");
    let n = 42;

    let s2 = s;
    let n2 = n;

    // TODO: Try printing s and n. Which compiles? Why?
    // println!("s = {}", s);
    // println!("n = {}", n);
```

## 🗒️ Quick Cheat Sheet

```rust
// === Ownership ===
let s1 = String::from("hi");
let s2 = s1;           // MOVE — s1 is invalid
let s3 = s2.clone();   // CLONE — deep copy, both valid

// === References ===
let r1 = &s3;          // immutable borrow (can have many)
let r2 = &s3;          // another immutable borrow ✅
fn foo(s: &mut String)  // mutable borrow (only one at a time)

// === Slices ===
let slice: &str = &s3[0..2];       // string slice
let arr_slice: &[i32] = &v[1..4];  // array/vec slice

// === Function signatures ===
fn borrow(s: &str) {}              // read-only borrow
fn mutate(s: &mut String) {}       // mutable borrow
fn consume(s: String) -> String {} // takes + returns ownership

// === Key rules ===
// 1. Each value has ONE owner
// 2. Move by default (Copy types are exception)
// 3. Dropped when owner goes out of scope
// 4. Either N×(&T) OR 1×(&mut T) — never both
```
