# Rust

## ЁЯза Concept Overview

Pattern matching is the mechanism Rust uses to destructure and inspect data тАФ and it's woven so deeply into the language that you'll use it dozens of times in every Rust file you write. If you come from C/C++, think of `switch`/`case` on steroids: it checks not just values, but the *shape* of data. If you're from Java or Go, forget everything about `switch` тАФ Rust's `match` is an entirely different beast. It's exhaustive (the compiler forces you to handle every possible case), it destructures enums, tuples, and structs, and it integrates seamlessly with Rust's ownership system.

### `enum` тАФ The Foundation of Pattern Matching

Before diving into `match`, we need to understand `enum`, because enums and pattern matching are two sides of the same coin in Rust.

A Rust `enum` is **not** like a C/Java enum (which is just a named integer). A Rust enum is a **sum type** тАФ also called a "tagged union" or "algebraic data type" (ADT). Each variant of the enum can carry **different kinds of data**:

```rust
enum Shape {
    Circle(f64),                    // carries a radius
    Rectangle(f64, f64),            // carries width and height
    Triangle { a: f64, b: f64, c: f64 }, // carries named fields
    Point,                          // carries nothing (unit-like)
}
```

In memory, a Rust enum stores a **discriminant** (an integer tag, usually 1 byte for small enums) plus the **payload** for the largest variant. So the size of `Shape` above is the discriminant + the size of the largest variant (`Triangle` has three `f64`s = 24 bytes). You can check this with `std::mem::size_of::<Shape>()`.

Compare this to other languages:
- **C**: You'd manually write a `struct` with a `tag` field and a `union`. Error-prone and not type-checked.
- **Java**: You'd use a sealed interface hierarchy with subclasses. Verbose but safe.
- **Go**: No sum types at all. You'd use an interface with type assertions тАФ no exhaustiveness checking.
- **TypeScript**: Discriminated unions are close, but only exist at the type level.

Rust enums are checked at compile time. If you add a new variant, the compiler will show every `match` that doesn't handle it. This is massive for maintaining large codebases.

#### The Two Most Important Enums in Rust

Rust's standard library defines two enums you'll use constantly:

```rust
// Option: a value that might or might not exist
enum Option<T> {
    Some(T),
    None,
}

// Result: an operation that might succeed or fail
enum Result<T, E> {
    Ok(T),
    Err(E),
}
```

There's no `null` in Rust. If a value might be absent, you wrap it in `Option<T>`. If an operation might fail, you return `Result<T, E>`. Both are **enums**, and the compiler forces you to handle both variants before using the inner value. This eliminates null pointer exceptions and unchecked error conditions at compile time.

### `match` тАФ Exhaustive Pattern Matching

The `match` expression takes a value and compares it against a series of **patterns**. The first pattern that matches executes its arm. Unlike `switch` in C, there's no fallthrough тАФ each arm is independent.

```rust
let x = 42;

match x {
    0 => println!("zero"),
    1 => println!("one"),
    2..=10 => println!("two to ten"),    // range pattern (inclusive)
    _ => println!("something else"),      // wildcard: matches anything
}
```

#### Exhaustiveness

The compiler demands that `match` covers **every possible value**. For integers, that's impossible without a wildcard `_`. For enums, you must cover every variant:

```rust
enum Direction {
    North,
    South,
    East,
    West,
}

fn describe(d: Direction) -> &'static str {
    match d {
        Direction::North => "up",
        Direction::South => "down",
        Direction::East => "right",
        Direction::West => "left",
    }
    // No _ needed! All 4 variants are covered.
    // If you add Direction::NorthEast later, THIS function won't compile
    // until you handle it. That's exhaustiveness at work.
}
```

This is one of Rust's most powerful safety features. In Go, a `switch` on a "direction" type silently falls through to a default (or does nothing) if you add a new case. In Rust, the compiler catches it.

#### Destructuring in `match`

This is where `match` goes far beyond `switch`. You can **pull apart** the data inside enum variants, tuples, and structs:

```rust
enum Message {
    Quit,
    Echo(String),
    Move { x: i32, y: i32 },
    Color(u8, u8, u8),
}

fn process(msg: Message) {
    match msg {
        Message::Quit => {
            println!("Quitting");
        }
        Message::Echo(text) => {
            // `text` now owns the String that was inside Echo
            println!("Echo: {}", text);
        }
        Message::Move { x, y } => {
            // destructure named fields
            println!("Moving to ({}, {})", x, y);
        }
        Message::Color(r, g, b) => {
            // destructure tuple variant
            println!("Color: rgb({}, {}, {})", r, g, b);
        }
    }
}
```

When you destructure, the **variable bindings** (`text`, `x`, `y`, `r`, `g`, `b`) follow Rust's ownership rules:
- If the enum owns the data (like `String`), destructuring **moves** the value out. After the match, the original enum is consumed.
- If you only need a reference, you can use `ref` or match on a reference (`&msg`).

#### Nested Destructuring

Patterns can go arbitrarily deep:

```rust
struct Point { x: f64, y: f64 }

enum Shape {
    Circle { center: Point, radius: f64 },
    Rect { top_left: Point, bottom_right: Point },
}

fn area(s: &Shape) -> f64 {
    match s {
        Shape::Circle { center: Point { x, y }, radius } => {
            println!("Circle centered at ({}, {})", x, y);
            std::f64::consts::PI * radius * radius
        }
        Shape::Rect {
            top_left: Point { x: x1, y: y1 },
            bottom_right: Point { x: x2, y: y2 },
        } => {
            (x2 - x1).abs() * (y2 - y1).abs()
        }
    }
}
```

Here we destructure a `Shape` enum, then destructure the `Point` struct inside it, binding each field to a variable. The `x: x1` syntax means "the field `x`, bound to the variable `x1`" (since we need different names for the two points' x-coordinates).

### Match Guards

A **match guard** adds an `if` condition to a pattern. The pattern must match AND the guard must be true:

```rust
let num = 4;

match num {
    n if n < 0 => println!("negative"),
    n if n == 0 => println!("zero"),
    n if n % 2 == 0 => println!("{} is positive even", n),
    n => println!("{} is positive odd", n),
}
```

Guards are useful when patterns alone can't express the logic (e.g., you need to compare two variables, or apply a runtime condition).

**Important ownership nuance**: With match guards, the value isn't moved into the arm until the guard passes. This means a guard can reference the bound variable without consuming it:

```rust
let maybe = Some(String::from("hello"));

match &maybe {
    Some(s) if s.len() > 3 => println!("long string: {}", s),
    Some(s) => println!("short string: {}", s),
    None => println!("nothing"),
}
// `maybe` is still usable here because we matched on a reference
```

### `if let` тАФ Match a Single Pattern

When you only care about one variant and want to ignore the rest, `match` is verbose:

```rust
// Verbose
let config_max = Some(100u32);
match config_max {
    Some(max) => println!("max is {}", max),
    _ => (),  // ignore None тАФ feels like boilerplate
}

// Concise
if let Some(max) = config_max {
    println!("max is {}", max);
}
```

`if let` is syntactic sugar for a `match` with exactly one interesting arm. It's not exhaustive тАФ you're explicitly saying "I only care about this one case."

You can add an `else` clause for the non-matching case:

```rust
if let Some(max) = config_max {
    println!("max: {}", max);
} else {
    println!("no max configured");
}
```

#### `let else` (Stabilized in Rust 1.65)

`let else` is the inverse: it binds the value if the pattern matches, and **diverges** (returns, breaks, panics, etc.) if it doesn't. This is spectacular for early returns:

```rust
fn process_input(input: Option<&str>) -> String {
    let Some(text) = input else {
        return String::from("no input provided");
    };
    // `text` is now a &str, guaranteed to be Some's inner value
    format!("Processing: {}", text)
}
```

Before `let else`, you'd write:

```rust
let text = match input {
    Some(t) => t,
    None => return String::from("no input provided"),
};
```

`let else` is cleaner, especially when you have multiple early-return checks in a row ("guard clause" pattern).

### `while let` тАФ Looping on a Pattern

`while let` runs a loop as long as the pattern keeps matching. The canonical use case is popping from a stack or iterator:

```rust
let mut stack = vec![1, 2, 3, 4, 5];

while let Some(top) = stack.pop() {
    println!("Popped: {}", top);
}
// Prints 5, 4, 3, 2, 1 тАФ then stack.pop() returns None and the loop ends
```

This replaces the clunkier:

```rust
loop {
    match stack.pop() {
        Some(top) => println!("Popped: {}", top),
        None => break,
    }
}
```

### `@` Bindings тАФ Name and Test Simultaneously

Sometimes you want to test a value against a pattern AND bind it to a variable. The `@` operator does both:

```rust
enum HTTPStatus {
    Code(u16),
}

fn classify(status: HTTPStatus) {
    match status {
        HTTPStatus::Code(code @ 200..=299) => {
            println!("Success! Code: {}", code);
        }
        HTTPStatus::Code(code @ 400..=499) => {
            println!("Client error: {}", code);
        }
        HTTPStatus::Code(code @ 500..=599) => {
            println!("Server error: {}", code);
        }
        HTTPStatus::Code(code) => {
            println!("Other: {}", code);
        }
    }
}
```

Without `@`, you'd need a match guard (`code if code >= 200 && code <= 299`), which is less readable for range checks.

### The `matches!` Macro

For simple boolean checks тАФ "does this value match this pattern?" тАФ use the `matches!` macro instead of a full `match` or `if let`:

```rust
let x: Option<i32> = Some(42);

// Instead of:
// if let Some(_) = x { true } else { false }

let is_some = matches!(x, Some(_));      // true
let is_big = matches!(x, Some(n) if n > 100);  // false

// Great in iterator chains:
let values = vec![Some(1), None, Some(3), None];
let count = values.iter().filter(|v| matches!(v, Some(_))).count();
// count == 2
```

### Or-Patterns (`|`)

You can combine multiple patterns with `|`:

```rust
let x = 'c';

match x {
    'a' | 'e' | 'i' | 'o' | 'u' => println!("vowel"),
    'a'..='z' => println!("consonant"),
    _ => println!("not a lowercase letter"),
}
```

This works with enums too:

```rust
enum Coin {
    Penny, Nickel, Dime, Quarter,
}

fn is_small(c: &Coin) -> bool {
    matches!(c, Coin::Penny | Coin::Nickel | Coin::Dime)
}
```

### Patterns in Other Places

Patterns aren't limited to `match`. They appear in:

1. **`let` statements**: `let (x, y) = (1, 2);` destructs a tuple.
2. **Function parameters**: `fn print_point(&(x, y): &(i32, i32))` destructs in the signature.
3. **`for` loops**: `for (i, val) in vec.iter().enumerate()` destructs the `(usize, &T)` tuple.

```rust
// Destructuring in function parameters
fn distance(&(x1, y1): &(f64, f64), &(x2, y2): &(f64, f64)) -> f64 {
    ((x2 - x1).powi(2) + (y2 - y1).powi(2)).sqrt()
}

// Destructuring in for loops
let scores = vec![("Alice", 95), ("Bob", 87), ("Carol", 92)];
for (name, score) in &scores {
    println!("{}: {}", name, score);
}
```

### Pattern Matching and Ownership тАФ The Full Picture

This is where Rust's pattern matching intersects with ownership, and where beginners often get confused.

**Matching by value vs by reference**:

```rust
let opt: Option<String> = Some(String::from("hello"));

// This MOVES the String out of the Option:
match opt {
    Some(s) => println!("{}", s),  // `s` now owns the String
    None => {},
}
// `opt` is no longer usable тАФ the String was moved

// To avoid the move, match on a reference:
let opt2: Option<String> = Some(String::from("world"));
match &opt2 {
    Some(s) => println!("{}", s),  // `s` is &String here
    None => {},
}
// `opt2` is still usable
```

When you `match &opt2`, Rust automatically adjusts the bindings: `Some(s)` gives you `s: &String` instead of `s: String`. This is called **match ergonomics** (introduced in Rust 1.26). Before match ergonomics, you had to write `&Some(ref s)` тАФ which was noisy and confusing.

**`ref` and `ref mut` in patterns**:

If you're matching by value but want to bind a reference (without consuming the enum), use `ref`:

```rust
let msg = Message::Echo(String::from("test"));

match msg {
    Message::Echo(ref s) => {
        // `s` is &String, `msg` is NOT consumed
        println!("{}", s);
    }
    _ => {}
}
// `msg` is still usable
```

`ref mut` gives you a mutable reference:

```rust
let mut msg = Message::Echo(String::from("test"));

match msg {
    Message::Echo(ref mut s) => {
        s.push_str(" modified");
    }
    _ => {}
}
```

In modern Rust (with match ergonomics), `match &msg` is preferred over `ref` тАФ it's cleaner. But you'll see `ref` in older codebases and in cases where the automatic desugaring doesn't do what you want.

---

### Irrefutable vs Refutable Patterns

Rust distinguishes between:

- **Irrefutable patterns**: Guaranteed to match. Used in `let`, `for`, function params.
  - `let x = 5;` тАФ always matches
  - `let (a, b) = (1, 2);` тАФ always matches (compiler knows it's a 2-tuple)

- **Refutable patterns**: Might not match. Used in `if let`, `while let`, `match` arms.
  - `if let Some(x) = maybe_value` тАФ only matches if it's `Some`

The compiler enforces this distinction. You **cannot** use a refutable pattern in a `let` binding:

```rust
// ERROR: refutable pattern in local binding
let Some(x) = some_option;  // What if it's None??

// OK: use `let else`
let Some(x) = some_option else { return; };
```

## ЁЯФС Key Points

- **Rust enums are algebraic data types (sum types)**, not simple integer enumerations. Each variant can carry different data.
- **`match` is exhaustive** тАФ the compiler forces you to handle every possible case, preventing bugs when new variants are added.
- **Destructuring** lets you pull apart enums, tuples, and structs inside match arms, binding inner data to variables.
- **`if let` / `let else` / `while let`** are shorthand for single-pattern matches, reducing boilerplate without losing safety.
- **Pattern matching respects ownership** тАФ destructuring by value moves data out; matching on a reference (`&value`) gives you borrowed bindings automatically (match ergonomics).

### Example 1 тАФ Modeling a Command with `enum` + `match`

```rust
// File: main.rs

#[derive(Debug)]
enum Command {
    Quit,
    Echo(String),
    Move { x: i32, y: i32 },
    ChangeColor(u8, u8, u8),
}

fn execute(cmd: Command) {
    match cmd {
        Command::Quit => {
            println!("Exiting program.");
            std::process::exit(0);
        }
        Command::Echo(message) => {
            // `message` is moved out of the enum тАФ we own it now
            println!("ECHO: {}", message);
        }
        Command::Move { x, y } => {
            println!("Moving cursor to ({}, {})", x, y);
        }
        Command::ChangeColor(r, g, b) => {
            println!("Changing color to #{:02X}{:02X}{:02X}", r, g, b);
        }
    }
}

fn main() {
    let commands = vec![
        Command::Echo(String::from("hello, world")),
        Command::Move { x: 10, y: 20 },
        Command::ChangeColor(255, 128, 0),
        // Command::Quit,  // Uncomment to test exit
    ];

    for cmd in commands {
        execute(cmd);
    }
}
```

**What this demonstrates**: Defining a rich `enum` where each variant carries different data, then using `match` to destructure and handle each variant exhaustively. Note how `Command::Echo(message)` moves the `String` out тАФ `cmd` is consumed by the match.

### Example 2 тАФ Option Chaining with `if let`, `let else`, and `matches!`

```rust
// File: main.rs

#[derive(Debug)]
struct User {
    name: String,
    email: Option<String>,
    age: Option<u32>,
}

fn display_email(user: &User) {
    // if let: handle only the Some case
    if let Some(ref email) = user.email {
        println!("{}'s email: {}", user.name, email);
    } else {
        println!("{} has no email on file.", user.name);
    }
}

fn require_adult(user: &User) -> Result<(), String> {
    // let-else: early return if pattern doesn't match
    let Some(age) = user.age else {
        return Err(format!("{} has no age set", user.name));
    };

    if age < 18 {
        return Err(format!("{} is {} тАФ must be 18+", user.name, age));
    }

    println!("{} is {} тАФ access granted.", user.name, age);
    Ok(())
}

fn main() {
    let users = vec![
        User { name: "Alice".into(), email: Some("alice@example.com".into()), age: Some(30) },
        User { name: "Bob".into(), email: None, age: Some(16) },
        User { name: "Carol".into(), email: Some("carol@test.io".into()), age: None },
    ];

    for user in &users {
        display_email(user);

        match require_adult(user) {
            Ok(()) => {},
            Err(e) => println!("  тЪая╕П  {}", e),
        }

        // matches! macro for quick boolean checks
        let has_email = matches!(user.email, Some(_));
        let is_named_alice = matches!(user.name.as_str(), "Alice" | "alice");
        println!("  has_email={}, is_alice={}\n", has_email, is_named_alice);
    }
}
```

**What this demonstrates**: Three different pattern-matching constructs (`if let`, `let else`, `matches!`) applied to real-world data validation. The `let else` pattern is particularly powerful for guard clauses тАФ it binds the value and exits if the pattern fails.

### Example 3 тАФ Nested Destructuring with `@` Bindings

```rust
// File: main.rs

#[derive(Debug)]
struct Point {
    x: f64,
    y: f64,
}

#[derive(Debug)]
enum Shape {
    Circle { center: Point, radius: f64 },
    Rectangle { origin: Point, width: f64, height: f64 },
    Line(Point, Point),
}

fn classify(shape: &Shape) {
    match shape {
        // Nested destructuring: pull x, y out of Point inside Circle
        Shape::Circle {
            center: Point { x, y },
            radius: r @ 0.0..=1.0,
        } => {
            println!("Small circle (r={:.2}) at ({:.1}, {:.1})", r, x, y);
        }
        Shape::Circle {
            center: Point { x, y },
            radius,
        } => {
            println!("Circle (r={:.2}) at ({:.1}, {:.1})", radius, x, y);
        }
        Shape::Rectangle {
            origin: Point { x, y },
            width: w,
            height: h,
        } if w == h => {
            // Match guard: square is a special rectangle
            println!("Square (side={:.1}) at ({:.1}, {:.1})", w, x, y);
        }
        Shape::Rectangle {
            origin: Point { x, y },
            width,
            height,
        } => {
            println!("Rect {}├Ч{} at ({:.1}, {:.1})", width, height, x, y);
        }
        Shape::Line(
            Point { x: x1, y: y1 },
            Point { x: x2, y: y2 },
        ) => {
            let length = ((x2 - x1).powi(2) + (y2 - y1).powi(2)).sqrt();
            println!("Line from ({:.1},{:.1}) to ({:.1},{:.1}), len={:.2}", x1, y1, x2, y2, length);
        }
    }
}

fn main() {
    let shapes = vec![
        Shape::Circle { center: Point { x: 0.0, y: 0.0 }, radius: 0.5 },
        Shape::Circle { center: Point { x: 3.0, y: 4.0 }, radius: 10.0 },
        Shape::Rectangle { origin: Point { x: 1.0, y: 1.0 }, width: 5.0, height: 5.0 },
        Shape::Rectangle { origin: Point { x: 0.0, y: 0.0 }, width: 3.0, height: 7.0 },
        Shape::Line(Point { x: 0.0, y: 0.0 }, Point { x: 3.0, y: 4.0 }),
    ];

    for shape in &shapes {
        classify(shape);
    }
}
```

**What this demonstrates**: Deep nested destructuring (enum тЖТ struct тЖТ fields), `@` bindings to simultaneously test a range and capture the value, match guards for conditional logic (detecting squares), and field renaming (`width: w`).

### Example 4 тАФ `while let` with an Iterator and State Machine

```rust
// File: main.rs

#[derive(Debug)]
enum Token {
    Number(f64),
    Plus,
    Minus,
    Star,
    End,
}

fn tokenize(input: &str) -> Vec<Token> {
    let mut tokens = Vec::new();
    let mut chars = input.chars().peekable();

    while let Some(&ch) = chars.peek() {
        match ch {
            ' ' | '\t' => { chars.next(); }
            '+' => { tokens.push(Token::Plus); chars.next(); }
            '-' => { tokens.push(Token::Minus); chars.next(); }
            '*' => { tokens.push(Token::Star); chars.next(); }
            '0'..='9' | '.' => {
                let mut num_str = String::new();
                while let Some(&d) = chars.peek() {
                    if d.is_ascii_digit() || d == '.' {
                        num_str.push(d);
                        chars.next();
                    } else {
                        break;
                    }
                }
                if let Ok(n) = num_str.parse::<f64>() {
                    tokens.push(Token::Number(n));
                }
            }
            _ => { chars.next(); } // skip unknown
        }
    }
    tokens.push(Token::End);
    tokens
}

fn main() {
    let input = "3.14 + 2 * 10 - 5";
    let tokens = tokenize(input);

    for token in &tokens {
        match token {
            Token::Number(n) => print!("[NUM:{:.2}] ", n),
            Token::Plus =>  print!("[+] "),
            Token::Minus => print!("[-] "),
            Token::Star =>  print!("[*] "),
            Token::End =>   print!("[END]"),
        }
    }
    println!();
}
```

**What this demonstrates**: `while let` driving a character-by-character tokenizer тАФ a real-world use case. The inner `while let` accumulates digits for multi-character numbers. This shows how pattern matching naturally models state machines and parsers.

### Example 5 тАФ Ownership Semantics in Pattern Matching

```rust
// File: main.rs

#[derive(Debug)]
enum Container {
    Empty,
    SingleItem(String),
    MultiItem(Vec<String>),
}

// Matching by VALUE тАФ consumes the Container
fn consume(c: Container) {
    match c {
        Container::Empty => println!("Nothing here."),
        Container::SingleItem(item) => {
            // `item` owns the String тАФ we can mutate it
            let upper = item.to_uppercase();
            println!("Single item (consumed): {}", upper);
        }
        Container::MultiItem(items) => {
            // `items` owns the Vec
            println!("Multi items (consumed): {:?}", items);
        }
    }
    // c is MOVED тАФ can't use it here
}

// Matching by REFERENCE тАФ borrows the Container
fn inspect(c: &Container) {
    match c {
        Container::Empty => println!("Nothing here."),
        Container::SingleItem(item) => {
            // `item` is &String thanks to match ergonomics
            println!("Single item (borrowed): {}", item);
        }
        Container::MultiItem(items) => {
            // `items` is &Vec<String>
            println!("Multi items (borrowed): {} items", items.len());
        }
    }
    // c is NOT consumed тАФ caller still owns it
}

// Matching with ref mut тАФ mutable borrow
fn modify(c: &mut Container) {
    match c {
        Container::SingleItem(item) => {
            // `item` is &mut String
            item.push_str(" [MODIFIED]");
        }
        Container::MultiItem(items) => {
            items.push(String::from("extra"));
        }
        _ => {}
    }
}

fn main() {
    let mut c = Container::SingleItem(String::from("hello"));

    // Borrow: inspect without consuming
    inspect(&c);

    // Mutable borrow: modify in place
    modify(&mut c);
    inspect(&c);

    // Move: consume the container
    consume(c);
    // println!("{:?}", c);  // ERROR: c was moved into consume()
}
```

**What this demonstrates**: The three ways to match on data тАФ by value (move), by shared reference (borrow), and by mutable reference. This is where ownership and pattern matching intersect, and understanding this prevents most borrow-checker frustrations.

| Mistake | Why it's wrong | Correct approach |
|---------|---------------|-----------------|
| Using `match` when only one variant matters | Leads to useless `_ => ()` arms that add noise. | Use `if let` for single-pattern matches. |
| Forgetting that `match` on an enum **moves** the inner data by default | You can't use the original value after the match. | Match on `&value` or `ref` to borrow instead of move. |
| Writing `if let` instead of `let else` for guard clauses | `if let` introduces nested blocks; multiple guards create a "pyramid of doom." | Use `let else` for early-return patterns: `let Some(x) = val else { return; };` |
| Adding a wildcard `_` to every `match` on an enum | If a new variant is added, the `_` silently swallows it тАФ you lose exhaustiveness. | Handle every variant explicitly. Only use `_` when you intentionally want to ignore future variants. |
| Trying to use a refutable pattern in `let` | `let Some(x) = maybe;` won't compile тАФ what if it's `None`? | Use `let Some(x) = maybe else { handle_none; };` or `if let`. |
| Confusing `|` (or-pattern) with `||` (logical OR) | `match x { 1 || 2 => ... }` is a syntax error. | Use single pipe: `match x { 1 | 2 => ... }` |

## ЁЯЧТя╕П Quick Cheat Sheet

```rust
// Basic match
match value {
    Pattern => expr,
    Pattern2 | Pattern3 => expr,  // or-pattern
    n @ 1..=10 => use(n),         // @ binding
    v if v > 100 => big(v),       // match guard
    _ => fallback,                // wildcard
}

// Destructure enum
match opt { Some(x) => x, None => default }

// if let (single pattern)
if let Some(x) = opt { use(x); }

// let else (guard clause)
let Some(x) = opt else { return; };

// while let (loop)
while let Some(x) = iter.next() { use(x); }

// matches! macro (boolean)
let yes = matches!(val, Pattern);
```
