## рҹ§  Concept Overview

Ethereum is fundamentally a **transaction-based state machine**. Every block transitions the global state вҖ” a mapping of 160-bit addresses to account objects вҖ” from one deterministic snapshot to the next. Understanding exactly *what* an account is, how the two account types differ at the EVM level, and how gas/nonce mechanics constrain execution is the bedrock of every security analysis you'll ever perform. Misunderstanding any of these primitives leads directly to exploitable bugs.

### The World State (State Trie)

Ethereum's global state is stored in a **Modified Merkle Patricia Trie (MPT)**. Each leaf node maps a 20-byte address to an **account object** containing exactly four fields:

| Field | Type | Description |
|-------|------|-------------|
| `nonce` | uint64 | For EOAs: number of transactions sent. For contracts: number of contracts created via `CREATE`. |
| `balance` | uint256 | Wei balance (1 ETH = 10^18 wei). |
| `storageRoot` | bytes32 | Root hash of another MPT holding the account's persistent key-value storage (contracts only). |
| `codeHash` | bytes32 | Keccak-256 hash of the EVM bytecode. For EOAs this is `keccak256("")`. |

The state root (a single 32-byte hash) is included in every block header. This means any node can cryptographically verify any piece of state at any historical block.

**Security implication**: The state trie is *public*. There is no such thing as "private" on-chain data. Even `private` Solidity variables are readable by anyone who knows the storage slot вҖ” a critical insight we'll exploit in later phases.

### EOA vs Contract Accounts

Ethereum has exactly two account types:

#### Externally Owned Accounts (EOAs)

- Controlled by a private key (secp256k1 ECDSA key pair)
- Can **initiate** transactions (the only entity that can)
- Has no code (`codeHash = keccak256("")`)
- Has no storage trie (`storageRoot = emptyTrieRoot`)
- Nonce increments with each transaction sent

```
EOA Account Object:
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ nonce: 42                        в”Ӯ  вҶҗ Tx count
в”Ӯ balance: 5000000000000000000     в”Ӯ  вҶҗ 5 ETH in wei
в”Ӯ storageRoot: EMPTY_TRIE_HASH    в”Ӯ  вҶҗ No storage
в”Ӯ codeHash: EMPTY_CODE_HASH       в”Ӯ  вҶҗ No code
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

An EOA's address is derived from its public key: `address = keccak256(publicKey)[12:]` (rightmost 20 bytes).

#### Contract Accounts

- Controlled by their deployed bytecode (no private key)
- **Cannot** initiate transactions вҖ” can only execute when called
- Has immutable code (once deployed, `codeHash` never changes вҖ” except via `SELFDESTRUCT` or proxy patterns)
- Has persistent storage (the `storageRoot` trie)
- Nonce increments when the contract uses the `CREATE` opcode

```
Contract Account Object:
в”Ңв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”җ
в”Ӯ nonce: 1                         в”Ӯ  вҶҗ CREATE count
в”Ӯ balance: 100000000000000000000   в”Ӯ  вҶҗ 100 ETH
в”Ӯ storageRoot: 0xabc123...         в”Ӯ  вҶҗ Points to storage trie
в”Ӯ codeHash: 0xdef456...            в”Ӯ  вҶҗ Hash of bytecode
в””в”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”Җв”ҳ
```

Contract addresses are computed deterministically:
- **CREATE**: `address = keccak256(rlp([sender, nonce]))[12:]`
- **CREATE2**: `address = keccak256(0xff ++ sender ++ salt ++ keccak256(initCode))[12:]`

**Security implications**:
1. `CREATE` addresses are predictable if you know the deployer's nonce вҶ’ enables front-running deployments
2. `CREATE2` addresses are deterministic regardless of nonce вҶ’ enables vanity address mining, but also means code can be deployed, `SELFDESTRUCT`ed, and *re-deployed with different code at the same address*
3. A contract *can receive ETH even without a `receive()` or `fallback()`* вҖ” via `SELFDESTRUCT`, coinbase rewards, or pre-funded addresses. This breaks contracts that rely on `address(this).balance` for accounting.

### Transactions: The Engine of State Change

Every state transition begins with a **transaction** signed by an EOA. There are two types:

#### Message Calls (to an existing account)

```
Transaction {
    nonce:    uint64    // Must match sender's current nonce
    gasPrice: uint256   // Wei per gas unit (legacy) or maxFeePerGas (EIP-1559)
    gasLimit: uint64    // Max gas units this tx may consume
    to:       address   // Target address
    value:    uint256   // Wei to transfer
    data:     bytes     // Calldata (function selector + arguments)
    v, r, s:  uint256   // ECDSA signature components
}
```

#### Contract Creation (to = empty)

When `to` is empty, the `data` field is treated as **init code** (constructor bytecode). The EVM executes this init code, and whatever bytes it `RETURN`s become the contract's deployed runtime bytecode.

### The Nonce: Replay Protection & Ordering

The nonce serves two critical functions:

1. **Replay protection**: Each transaction includes the sender's current nonce. The network rejects transactions where `tx.nonce != account.nonce`. After inclusion, the account nonce increments. This prevents replaying the same signed transaction.

2. **Ordering**: Transactions from the same EOA are processed in strict nonce order. A pending tx with nonce 5 blocks all subsequent txs (nonce 6, 7, ...) until it's mined or replaced.

**Security implications**:
- Nonces are per-chain. Before EIP-155, signed transactions could be replayed on other chains (e.g., ETH вҶ” ETC). EIP-155 adds `chainId` to the signing hash.
- Transaction replacement (same nonce, higher gas price) is a vector for MEV вҖ” miners/searchers can replace pending transactions.

### Gas: Computation Metering

Gas is Ethereum's anti-DoS mechanism. Every EVM opcode has a fixed gas cost. Key details:

| Concept | Value | Notes |
|---------|-------|-------|
| Gas limit (per tx) | Set by sender | Max gas willing to spend |
| Block gas limit | ~30M (mainnet) | Total gas cap per block |
| Intrinsic gas | 21,000 | Base cost for any transaction |
| Calldata cost | 4 gas/zero byte, 16 gas/non-zero byte | Why calldata optimization matters |
| SSTORE (cold, 0вҶ’non-zero) | 20,000 | Most expensive common opcode |
| SLOAD (cold) | 2,100 | First read of a storage slot in tx |

#### EIP-1559 Fee Structure (Post-London)

```
effectiveGasPrice = min(maxFeePerGas, baseFee + maxPriorityFeePerGas)

Total cost = gasUsed Г— effectiveGasPrice
Burned     = gasUsed Г— baseFee          вҶҗ Sent to address(0)
Miner tip  = gasUsed Г— priorityFee     вҶҗ Sent to block proposer
Refund     = (gasLimit - gasUsed) Г— effectiveGasPrice  вҶҗ Returned to sender
```

The `baseFee` adjusts dynamically: increases when blocks are >50% full, decreases when <50% full. This creates a market-driven fee mechanism.

**Security implications**:
- **Out-of-gas reverts**: If a transaction runs out of gas mid-execution, ALL state changes revert but the gas is still consumed. This is exploitable for DoS.
- **Gas griefing**: An external call's gas can be manipulated. The `63/64` rule (EIP-150) reserves 1/64th of remaining gas for the caller after a `CALL`.
- **Gas-dependent logic**: Never use `gasleft()` for critical control flow вҖ” it's manipulable by callers.
- **Storage refunds**: `SSTORE` from non-zero to zero gives a gas refund (capped at 20% of total gas used post-EIP-3529). This was exploited via "gas tokens" before the cap.

### Network Architecture

Ethereum's P2P network consists of:

1. **Execution Layer (EL)**: Runs the EVM, processes transactions, maintains state. Clients: Geth, Nethermind, Besu, Erigon.
2. **Consensus Layer (CL)**: Manages Proof-of-Stake consensus (since The Merge, Sep 2022). Clients: Prysm, Lighthouse, Teku, Lodestar, Nimbus.

**Security implication**: Validators see pending transactions in the mempool before inclusion вҶ’ this enables MEV extraction (front-running, sandwich attacks). Flashbots and PBS (Proposer-Builder Separation) attempt to mitigate this.

### The Mempool (Transaction Pool)

When an EOA broadcasts a transaction, it enters the **mempool** вҖ” a holding area of unconfirmed transactions. Validators select transactions from the mempool to include in blocks, typically prioritizing by gas price.

The mempool is **public and observable**. Anyone running a full node can monitor pending transactions in real-time. This transparency is the root cause of:
- **Front-running**: Seeing a profitable tx and submitting your own with higher gas to get included first
- **Sandwich attacks**: Inserting txs before AND after a victim's swap to profit from price impact
- **Backrunning**: Submitting a tx immediately after a target tx to capture arbitrage

### `msg.sender` vs `tx.origin`

This distinction is critical for security:

```
EOA (tx.origin) в”Җв”Җcallв”Җв”Җв–¶ Contract A (msg.sender for B) в”Җв”Җcallв”Җв”Җв–¶ Contract B
                                                                    в”Ӯ
                                                          msg.sender = A
                                                          tx.origin  = EOA
```

- `msg.sender`: The immediate caller (can be EOA or contract)
- `tx.origin`: The original EOA that initiated the transaction (always an EOA)

**Never use `tx.origin` for authorization**. A phishing contract can trick a user into calling it, then call your contract вҖ” `tx.origin` will be the victim, but `msg.sender` will be the attacker's contract.

```solidity
// вқҢ VULNERABLE вҖ” uses tx.origin for auth
contract VulnerableWallet {
    address public owner;

    function transfer(address to, uint256 amount) external {
        require(tx.origin == owner, "Not owner"); // DANGEROUS!
        payable(to).transfer(amount);
    }
}

// вң… SECURE вҖ” uses msg.sender for auth
contract SecureWallet {
    address public owner;

    function transfer(address to, uint256 amount) external {
        require(msg.sender == owner, "Not owner"); // SAFE
        payable(to).transfer(amount);
    }
}
```

### `extcodesize` and Contract Detection

The EVM opcode `EXTCODESIZE` returns the size of code at an address. It's commonly (and incorrectly) used to determine "is this a contract?":

```solidity
function isContract(address addr) internal view returns (bool) {
    uint256 size;
    assembly { size := extcodesize(addr) }
    return size > 0;
}
```

**This check is bypassable!** During a contract's constructor execution, `EXTCODESIZE` returns 0 because the runtime code hasn't been stored yet. An attacker contract can call your function from its constructor, bypassing the "no contracts" check. We'll exploit this in Ethernaut's Gatekeeper challenges.

## рҹ”‘ Key Points

- Ethereum has exactly two account types: **EOAs** (private key, no code) and **Contracts** (code, no private key). Both share the same 4-field account structure.
- Contract addresses are deterministic: `CREATE` depends on sender+nonce, `CREATE2` depends on sender+salt+initCodeHash. Both are predictable and exploitable.
- The nonce provides replay protection and enforces transaction ordering. EIP-155 adds chain ID to prevent cross-chain replay.
- Gas is the anti-DoS meter вҖ” out-of-gas reverts all state changes but consumes gas. The 63/64 rule, storage refunds, and block gas limits all have security implications.
- The mempool is public: all pending transactions are visible, enabling front-running and MEV extraction. This is a fundamental architectural property, not a bug.
- **Never use `tx.origin` for authorization** вҖ” it enables phishing attacks across contract call chains.
- `EXTCODESIZE == 0` does NOT guarantee the caller is an EOA вҖ” constructors bypass this check.

## рҹ’» Code Deep Dive / Mechanics

### рҹ”ҙ Vulnerable Pattern: Balance-Dependent Logic

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

/// @title VulnerableGame вҖ” Balance-based game logic
/// @notice This contract assumes its balance only changes through deposit()
contract VulnerableGame {
    uint256 public targetAmount = 7 ether;
    mapping(address => uint256) public deposits;

    function deposit() external payable {
        require(msg.value == 1 ether, "Must deposit exactly 1 ETH");
        deposits[msg.sender] += msg.value;

        // BUG: Uses address(this).balance instead of internal accounting
        if (address(this).balance == targetAmount) {
            // Winner! Last depositor wins the pot
            payable(msg.sender).transfer(address(this).balance);
        }
    }
}
```

**Attack Vector**: An attacker can use `selfdestruct` to force-send ETH to this contract, pushing the balance past `targetAmount` without going through `deposit()`. The equality check `== targetAmount` will never be satisfied, permanently locking all funds.

```solidity
// Attacker forces ETH into the game contract
contract ForceFeeder {
    constructor(address target) payable {
        selfdestruct(payable(target));
    }
}
```

### рҹҹў Secure Implementation

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

contract SecureGame {
    uint256 public targetAmount = 7 ether;
    uint256 public totalDeposited; // вҶҗ Internal accounting, not balance
    mapping(address => uint256) public deposits;

    function deposit() external payable {
        require(msg.value == 1 ether, "Must deposit exactly 1 ETH");
        deposits[msg.sender] += msg.value;
        totalDeposited += msg.value; // вҶҗ Track internally

        // FIXED: Uses internal counter, immune to forced ETH
        if (totalDeposited == targetAmount) {
            uint256 payout = totalDeposited;
            totalDeposited = 0;
            payable(msg.sender).transfer(payout);
        }
    }
}
```

**Why this is secure**: Internal accounting (`totalDeposited`) is solely controlled by the contract's logic. Force-sent ETH via `selfdestruct` or coinbase rewards doesn't affect the internal counter.

### Foundry Test: Force-Send ETH PoC

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.20;

import "forge-std/Test.sol";

contract ForceFeeder {
    constructor(address target) payable {
        selfdestruct(payable(target));
    }
}

contract VulnerableGame {
    uint256 public targetAmount = 7 ether;

    function deposit() external payable {
        require(msg.value == 1 ether, "Must deposit exactly 1 ETH");
        if (address(this).balance == targetAmount) {
            payable(msg.sender).transfer(address(this).balance);
        }
    }
}

contract ForceETHTest is Test {
    VulnerableGame game;

    function setUp() public {
        game = new VulnerableGame();
    }

    function testForceETHBreaksGame() public {
        // Force-send 1 wei to the game contract
        new ForceFeeder{value: 1 wei}(address(game));

        // Now the balance is 1 wei, not a multiple of 1 ether
        assertEq(address(game).balance, 1 wei);

        // Even after 7 legitimate deposits, balance = 7 ether + 1 wei
        // The == 7 ether check will NEVER pass
        for (uint i = 0; i < 7; i++) {
            address depositor = address(uint160(i + 100));
            vm.deal(depositor, 1 ether);
            vm.prank(depositor);
            game.deposit{value: 1 ether}();
        }

        // Balance is 7 ether + 1 wei вҖ” funds permanently locked!
        assertEq(address(game).balance, 7 ether + 1 wei);
    }
}
```
